Rust for C/C++ Programmers

Rust 内存管理

你将学到： Rust 的 ownership 系统 — 该语言中最重要的概念。在本章之后，你将理解 move semantics、borrowing 规则和 Drop trait。如果你理解了本章，Rust 的其余部分自然就会跟上。如果你还在挣扎，请重读它 — 大多数 C/C++ 开发者在第二遍时才会豁然开朗。

  • C/C++ 中的内存管理是 bug 的来源：
    • 在 C 中：内存使用 malloc() 分配，使用 free() 释放。没有针对悬空指针、use-after-free 或 double-free 的检查
    • 在 C++ 中：RAII（资源获取即初始化）和智能指针有所帮助，但 std::move(ptr) 在 move 后仍然可以编译 — use-after-move 是 UB
  • Rust 使 RAII 万无一失：
    • Move 是破坏性的 — 编译器拒绝让你触碰被 move 的变量
    • 不需要 Rule of Five（不需要 copy ctor、move ctor、copy assign、move assign、destructor）
    • Rust 完全控制内存分配，但在编译时强制执行安全性
    • 这是通过所有权、borrowing、可变性和生命周期等机制的组合来实现的
    • Rust 运行时分配可以发生在栈和堆上

对于 C++ 开发者 — 智能指针映射：

C++Rust安全性改进
std::unique_ptr<T>Box<T>move 后使用不可能
std::shared_ptr<T>Rc<T>（单线程）默认没有引用循环
std::shared_ptr<T>（线程安全）Arc<T>显式线程安全
std::weak_ptr<T>Weak<T>必须检查有效性
原始指针*const T / *mut T仅在 unsafe 块中

对于 C 开发者：Box<T> 取代 malloc/free 对。Rc<T> 取代手动引用计数。原始指针存在但被限制在 unsafe 块中。

Rust ownership、borrowing 和生命周期

  • 回想一下，Rust 只允许对变量进行单个可变引用和任意数量的只读引用
    • 变量的初始声明建立 ownership
    • 后续引用从原始所有者 borrow。规则是 borrow 的作用域永远不能超过 owning 作用域。换句话说，borrow 的 lifetime 不能超过 owning lifetime
fn main() {
    let a = 42; // 所有者
    let b = &a; // 第一次 borrow
    {
        let aa = 42;
        let c = &a; // 第二次 borrow；a 仍在作用域内
        // 正确：c 在这里超出作用域
        // aa 在这里超出作用域
    }
    // let d = &aa; // 无法编译，除非 aa 被移动到外部作用域
    // b 在 a 之前隐式超出作用域
    // a 最后超出作用域
}
  • Rust 可以使用几种不同的机制将参数传递给方法
    • 按值（复制）：通常是可以简单复制的类型（例如 u8、u32、i8、i32）
    • 按引用：这相当于传递指向实际值的指针。这也称为 borrowing，引用可以是不可变的（&）或可变的（&mut）
    • 按 move：这将值的"所有权"转移给函数。调用者不能再引用原始值
fn foo(x: &u32) {
    println!("{x}");
}
fn bar(x: u32) {
    println!("{x}");
}
fn main() {
    let a = 42;
    foo(&a);    // 按引用
    bar(a);     // 按值（复制）
}
  • Rust 禁止方法中的悬空引用
    • 方法返回的引用必须仍在作用域内
    • 当 Rust 引用超出作用域时会自动 drop
fn no_dangling() -> &u32 {
    // a 的生命周期从这里开始
    let a = 42;
    // 无法编译。a 的生命周期在这里结束
    &a
}

fn ok_reference(a: &u32) -> &u32 {
    // 正确，因为 a 的生命周期始终超过 ok_reference()
    a
}
fn main() {
    let a = 42;     // a 的生命周期从这里开始
    let b = ok_reference(&a);
    // b 的生命周期在这里结束
    // a 的生命周期在这里结束
}

Rust move semantics

  • 默认情况下，Rust 赋值转移所有权
fn main() {
    let s = String::from("Rust");    // 从堆分配一个字符串
    let s1 = s; // 将所有权转移给 s1。s 在此时无效
    println!("{s1}");
    // 这将无法编译
    //println!("{s}");
    // s1 在这里超出作用域，内存被释放
    // s 在这里超出作用域，但什么也没发生，因为它不拥有任何东西
}

Rust move semantics 和 borrowing

fn foo(s : String) {
    println!("{s}");
    // s 指向的堆内存将在这里被释放
}
fn bar(s : &String) {
    println!("{s}");
    // 什么也没发生 -- s 被借用
}
fn main() {
    let s = String::from("Rust string move example");    // 从堆分配一个字符串
    foo(s); // 转移所有权；s 现在无效
    // println!("{s}");  // 无法编译
    let t = String::from("Rust string borrow example");
    bar(&t);    // t 继续拥有所有权
    println!("{t}");
}

Rust move semantics 和 ownership

  • 可以通过 move 转移所有权
    • move 完成后引用未决引用是非法的
    • 如果不需要 move，请考虑借用
struct Point {
    x: u32,
    y: u32,
}
fn consume_point(p: Point) {
    println!("{} {}", p.x, p.y);
}
fn borrow_point(p: &Point) {
    println!("{} {}", p.x, p.y);
}
fn main() {
    let p = Point {x: 10, y: 20};
    // 尝试翻转这两行
    borrow_point(&p);
    consume_point(p);
}

Rust Clone

  • clone() 方法可用于复制原始内存。原始引用继续有效（缺点是分配量是 2 倍）
fn main() {
    let s = String::from("Rust");    // 从堆分配一个字符串
    let s1 = s.clone(); // 复制字符串；在堆上创建新分配
    println!("{s1}");
    println!("{s}");
    // s1 在这里超出作用域，内存被释放
    // s 在这里超出作用域，内存被释放
}

Rust Copy trait

  • Rust 使用 Copy trait 为内置类型实现复制语义
    • 示例包括 u8、u32、i8、i32 等。复制语义使用"按值传递"
    • 用户定义的数据类型可以选择使用 derive 宏自动实现 Copy trait 来选择 copy 语义
    • 编译器将在新赋值后为副本分配空间
// 尝试注释掉这行以查看下面 let p1 = p; 的变化
#[derive(Copy, Clone, Debug)]   // 我们稍后将详细讨论这个
struct Point{x: u32, y:u32}
fn main() {
    let p = Point {x: 42, y: 40};
    let p1 = p;     // 现在这将执行复制而不是 move
    println!("p: {p:?}");
    println!("p1: {p:?}");
    let p2 = p1.clone();    // 语义上与复制相同
}

Rust Drop trait

  • Rust 在作用域结束时自动调用 drop() 方法
    • drop 是名为 Drop 的通用 trait 的一部分。编译器为所有类型提供默认的 NOP 实现，但类型可以覆盖它。例如，String 类型覆盖它以释放堆分配的内存
    • 对于 C 开发者：这取代了手动 free() 调用的需要 — 资源在超出作用域时自动释放（RAII）
  • 关键安全性： 你不能直接调用 .drop()（编译器禁止它）。相反，使用 drop(obj) 将值移入函数，运行其析构函数，并防止进一步使用 — 消除 double-free bug

对于 C++ 开发者： Drop 直接映射到 C++ 析构函数（~ClassName()）：

C++ 析构函数Rust Drop
语法~MyClass() { ... }impl Drop for MyType { fn drop(&mut self) { ... } }
何时调用作用域结束（RAII）作用域结束（相同）
move 时调用源留在"有效但未指定状态" — 析构函数仍在被 move 的源上运行源消失 — 被 move 的值上没有析构函数调用
手动调用obj.~MyClass()（危险，很少使用）drop(obj)（安全 — 获取所有权，调用 drop，防止进一步使用）
顺序声明顺序的逆序声明顺序的逆序（相同）
Rule of Five必须管理 copy ctor、move ctor、copy assign、move assign、destructor只有 Drop — 编译器处理 move 语义，Clone 是可选的
需要虚析构函数？是，如果通过基指针删除不 — 没有继承，所以没有切片问题
struct Point {x: u32, y:u32}

// 等价于：~Point() { printf("Goodbye point x:%u, y:%u\n", x, y); }
impl Drop for Point {
    fn drop(&mut self) {
        println!("Goodbye point x:{}, y:{}", self.x, self.y);
    }
}
fn main() {
    let p = Point{x: 42, y: 42};
    {
        let p1 = Point{x:43, y: 43};
        println!("Exiting inner block");
        // p1.drop() 在这里调用 — 类似于 C++ 作用域结束析构函数
    }
    println!("Exiting main");
    // p.drop() 在这里调用
}

练习：Move、Copy 和 Drop

🟡 中级 — 自由实验；编译器会指导你

  • 创建你自己的 Point 实验，在 #[derive(Debug)] 中使用和不使用 Copy，确保你理解差异。目的是牢固理解 move vs copy 的工作原理，所以一定要提问
  • 为 Point 实现自定义 Drop，在 drop 中将 x 和 y 设置为 0。这种模式对于释放锁和其他资源很有用
struct Point{x: u32, y: u32}
fn main() {
    // 创建 Point，将其赋值给不同的变量，创建新作用域，
    // 将 point 传递给函数等。
}
答案（点击展开） 
#[derive(Debug)]
struct Point { x: u32, y: u32 }

impl Drop for Point {
    fn drop(&mut self) {
        println!("Dropping Point({}, {})", self.x, self.y);
        self.x = 0;
        self.y = 0;
        // 注意：在 drop 中设置为 0 演示了这种模式，
        // 但在 drop 完成后你无法观察到这些值
    }
}

fn consume(p: Point) {
    println!("Consuming: {:?}", p);
    // p 在这里被 drop
}

fn main() {
    let p1 = Point { x: 10, y: 20 };
    let p2 = p1;  // Move — p1 不再有效
    // println!("{:?}", p1);  // 无法编译：p1 被 move 了

    {
        let p3 = Point { x: 30, y: 40 };
        println!("p3 in inner scope: {:?}", p3);
        // p3 在这里被 drop（作用域结束）
    }

    consume(p2);  // p2 被 move 到 consume 并在那里被 drop
    // println!("{:?}", p2);  // 无法编译：p2 被 move 了

    // 现在尝试：在 Point 上添加 #[derive(Copy, Clone)]（并移除 Drop impl）
    // 观察 p1 在 let p2 = p1; 后如何保持有效
}
// 输出：
// p3 in inner scope: Point { x: 30, y: 40 }
// Dropping Point(30, 40)
// Consuming: Point { x: 10, y: 20 }
// Dropping Point(10, 20)
