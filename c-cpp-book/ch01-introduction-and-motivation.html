Rust 的理由

想直接跳到代码？ 跳转到 给我看些代码

无论你是来自 C 还是 C++，核心痛点都是相同的：内存安全 bug 可以干净地编译，但在运行时崩溃、损坏或泄漏。

  • 超过 70% 的 CVE 是由内存安全问题引起的——缓冲区溢出、悬空指针、释放后使用
  • C++ 的 shared_ptr、unique_ptr、RAII 和移动语义是朝着正确方向迈出的步骤，但它们只是创可贴，而非治愈——它们让移动后使用、引用循环、迭代器失效和异常安全漏洞大开
  • Rust 提供你依赖 C/C++ 的性能，但具有编译时保证的安全性

📖 深入探讨： 参见 为什么 C/C++ 开发者需要 Rust 获取具体的漏洞示例、Rust 消除的完整问题列表，以及为什么 C++ 智能指针不够

Rust 如何解决这些问题？

缓冲区溢出和边界违规

  • 所有 Rust 数组、切片和字符串都有显式的边界关联。编译器插入检查以确保任何边界违规导致运行时崩溃（Rust 术语中的 panic）——永远不会是未定义行为

悬空指针和引用

  • Rust 引入生命周期和借用检查来在编译时消除悬空引用
  • 没有悬空指针，没有释放后使用——编译器根本不会让你这样做

移动后使用

  • Rust 的所有权系统使移动具有破坏性——一旦你移动了一个值，编译器拒绝让你使用原始值。没有僵尸对象，没有"有效但未指定状态"

资源管理

  • Rust 的 Drop trait 是正确实现的 RAII——编译器在资源离开作用域时自动释放它们，并防止移动后使用，这是 C++ RAII 无法做到的
  • 不需要规则五（无需定义复制构造函数、移动构造函数、复制赋值、移动赋值、析构函数）

错误处理

  • Rust 没有异常。所有错误都是值（Result<T, E>），使错误处理显式并在类型签名中可见

迭代器失效

  • Rust 的借用检查器禁止在迭代集合时修改它。你根本无法编写困扰 C++ 代码库的 bug：
#![allow(unused)]
fn main() {
// Rust 等效的迭代中删除：retain()
pending_faults.retain(|f| f.id != fault_to_remove.id);

// 或者：收集到新的 Vec（函数式风格）
let remaining: Vec<_> = pending_faults
    .into_iter()
    .filter(|f| f.id != fault_to_remove.id)
    .collect();
}

数据竞争

  • 类型系统通过 Send 和 Sync trait 在编译时防止数据竞争

Rust 的其他独特卖点和功能

  • 线程之间没有数据竞争（编译时 Send/Sync 检查）
  • 没有移动后使用（不同于 C++ std::move 留下僵尸对象）
  • 没有未初始化的变量
    • 所有变量在使用前必须初始化
  • 没有简单的内存泄漏
    • Drop trait = 正确的 RAII，不需要规则五
    • 编译器在内存离开作用域时自动释放
  • 没有忘记解锁互斥锁
    • 锁守卫是访问数据的唯一方式（Mutex<T> 包装数据，而不是访问）
  • 没有异常处理的复杂性
    • 错误是值（Result<T, E>），在函数签名中可见，用 ? 传播
  • 出色支持类型推断、枚举、模式匹配、零成本抽象
  • 内置支持依赖管理、构建、测试、格式化、代码检查
    • cargo 取代 make/CMake + lint + 测试框架
