Rust for C/C++ Programmers

Rust crate 和模块

你将学到： Rust 如何将代码组织成模块和 crate — 默认私有的可见性、pub 修饰符、workspace 和 crates.io 生态系统。取代 C/C++ 头文件、#include 和 CMake 依赖管理。

  • 模块是 crate 内代码的基本组织单元
    • 每个源文件（.rs）都是它自己的模块，可以使用 mod 关键字创建嵌套模块。
    • （子）模块中的所有类型默认是私有的，除非显式标记为 pub（公共），否则在同一 crate 内不可见。pub 的作用域可以进一步限制为 pub(crate) 等
    • 即使类型是公共的，除非使用 use 关键字导入，否则它不会自动在另一个模块的作用域内可见。子模块可以使用 use super:: 引用父作用域中的类型
    • 源文件（.rs）不会自动包含在 crate 中，除非它们在 main.rs（可执行文件）或 lib.rs 中显式列出

练习：模块和函数

  • 我们将看看如何修改我们的 hello world 来调用另一个函数
    • 如前所述，函数使用 fn 关键字定义。-> 关键字声明函数返回一个值（默认是 void），类型为 u32（无符号 32 位整数）
    • 函数按模块作用域，即两个模块中具有完全相同名称的两个函数不会有名称冲突
      • 模块作用域扩展到所有类型（例如，mod a { struct foo; } 中的 struct foo 是与 mod b { struct foo; }（b::foo）不同的类型（a::foo））

Workspace 和 crate（包）

  • 任何重要的 Rust 项目都应该使用 workspace 来组织组件 crate
    • workspace 只是用于构建目标二进制文件的本地 crate 集合。workspace 根目录的 Cargo.toml 应该有一个指向组成包（crate）的指针
[workspace]
resolver = "2"
members = ["package1", "package2"]

|-- package2/
|   |-- Cargo.toml  # 包 2 配置
|   `-- src/
|       `-- main.rs # 包 2 源代码

使用 crates.io 的社区 crate

  • Rust 拥有充满活力的社区 crate 生态系统（参见 https://crates.io/）
    • Rust 的理念是保持标准库紧凑，将功能外包给社区 crate
    • 使用社区 crate 没有硬性规定，但经验法则应该是确保 crate 具有合理的成熟度（由版本号指示），并且正在积极维护。如果对 crate 有疑问，请联系内部来源
  • 发布在 crates.io 上的每个 crate 都有主要和次要版本
    • crate 应该遵守此处定义的主要和次要 SemVer 指南：https://doc.rust-lang.org/cargo/reference/semver.html
    • TL;DR 版本是同一次要版本不应有破坏性更改。例如，v0.11 必须与 v0.15 兼容（但 v0.20 可能有破坏性更改）

crate 依赖和 SemVer

  • crate 可以定义对 crate 特定版本、特定次要或主要版本的依赖，或不关心。以下示例显示声明对 rand crate 依赖的 Cargo.toml 条目
  • 至少 0.10.0，但任何 < 0.11.0 都可以
[dependencies]
rand = { version = "0.10.0"}
  • 仅 0.10.0，没有其他
[dependencies]
rand = { version = "=0.10.0"}
  • 不关心；cargo 将选择最新版本
[dependencies]
rand = { version = "*"}
  • 参考：https://doc.rust-lang.org/cargo/reference/specifying-dependencies.html

use rand::RngExt;

fn main() {
    let mut rng = rand::rng();
    let n: u32 = rng.random_range(1..=100);
    println!("Random number (1-100): {n}");

Cargo.toml 和 Cargo.lock

  • 如前所述，Cargo.lock 是从 Cargo.toml 自动生成的
    • Cargo.lock 背后的主要思想是确保可重复构建。例如，如果 Cargo.toml 指定了 0.10.0 版本，cargo 可以选择任何 < 0.11.0 的版本
    • Cargo.lock 包含构建期间使用的 rand crate 的特定版本。
    • 建议在 git 仓库中包含 Cargo.lock 以确保可重复构建

Cargo test 功能

  • Rust 单元测试位于同一源文件中（按惯例），通常分组到单独的模块中
    • 测试代码永远不会包含在实际二进制文件中。这通过 cfg（配置）功能实现。配置对于创建平台特定代码（Linux vs Windows）很有用
    • 测试可以使用 cargo test 执行。参考：https://doc.rust-lang.org/reference/conditional-compilation.html
#![allow(unused)]
fn main() {
pub fn add(left: u64, right: u64) -> u64 {
    left + right
}
// 仅在测试期间包含
#[cfg(test)]
mod tests {
    use super::*; // 这使得父作用域中的所有类型可见
    #[test]
    fn it_works() {
        let result = add(2, 2); // 或者，super::add(2, 2);
        assert_eq!(result, 4);
    }
}
}

其他 Cargo 功能

  • cargo 有几个其他有用的功能，包括：
    • cargo clippy 是 lint Rust 代码的好方法。通常，应该修复警告（或者如果真的有必要则偶尔抑制）
    • cargo format 执行 rustfmt 工具来格式化源代码。使用该工具确保签入代码的标准格式，并结束关于风格的争论
    • cargo doc 可用于从 /// 风格的注释生成文档。crates.io 上所有 crate 的文档都是使用此方法生成的

构建配置文件：控制优化

在 C 中，你传递 -O0、-O2、-Os、-flto 给 gcc/clang。在 Rust 中，你在 Cargo.toml 中配置构建配置文件：

[profile.release]

构建脚本（build.rs）：链接 C 库

在 C 中，你使用 Makefile 或 CMake 链接库和运行代码生成。Rust 在 crate 根目录使用 build.rs 文件：

// build.rs — 在编译 crate 之前运行

fn main() {
    // 链接系统 C 库（类似 gcc 中的 -lbmc_ipmi）
    println!("cargo::rustc-link-lib=bmc_ipmi");

    // 在哪里找到库（类似 -L/usr/lib/bmc）
    println!("cargo::rustc-link-search=/usr/lib/bmc");

    // 如果 C 头文件更改则重新运行
    println!("cargo::rerun-if-changed=wrapper.h");
}

你甚至可以直接从 Rust crate 编译 C 源文件：

# Cargo.toml
[build-dependencies]
cc = "1"  # C 编译器集成

// build.rs
fn main() {
    cc::Build::new()
        .file("src/c_helpers/ipmi_raw.c")
        .include("/usr/include/bmc")
        .compile("ipmi_raw");   // 生成 libipmi_raw.a，自动链接
    println!("cargo::rerun-if-changed=src/c_helpers/ipmi_raw.c");
}

测试模式和策略

C 固件团队通常使用 CUnit、CMocka 或自定义框架编写测试，需要大量样板代码。Rust 的内置测试框架功能更强大。本节介绍生产代码所需的模式。

#[should_panic] — 测试预期失败

#![allow(unused)]
fn main() {
// 测试某些条件导致 panic（类似 C 的 assert 失败）
#[test]
#[should_panic(expected = "index out of bounds")]
fn test_bounds_check() {
    let v = vec![1, 2, 3];
    let _ = v[10];  // 应该 panic
}

#[test]
#[should_panic(expected = "temperature exceeds safe limit")]
fn test_thermal_shutdown() {
    fn check_temperature(celsius: f64) {
        if celsius > 105.0 {
            panic!("temperature exceeds safe limit: {celsius}°C");
        }
    }
    check_temperature(110.0);
}
}

    // 此测试仅在具有 GPU 的机器上运行

    // 使用以下命令运行：cargo test -- --include-ignored  （运行所有测试）

使用构建器函数的测试夹具

C 使用 setUp()/tearDown() 函数。Rust 使用辅助函数和 Drop：

#![allow(unused)]
fn main() {
struct TestFixture {
    temp_dir: std::path::PathBuf,
    config: Config,
}

impl TestFixture {
    fn new() -> Self {
        let temp_dir = std::env::temp_dir().join(format!("test_{}", std::process::id()));
        std::fs::create_dir_all(&temp_dir).unwrap();
        let config = Config {
            log_dir: temp_dir.clone(),
            max_retries: 3,
            ..Default::default()
        };
        Self { temp_dir, config }
    }
}

impl Drop for TestFixture {
    fn drop(&mut self) {
        // 自动清理 — 类似 C 的 tearDown() 但不会被忘记
        let _ = std::fs::remove_dir_all(&self.temp_dir);
    }
}

#[test]
fn test_with_fixture() {
    let fixture = TestFixture::new();
    // 使用 fixture.config, fixture.temp_dir...
    assert!(fixture.temp_dir.exists());
    // fixture 在这里自动 drop → 清理运行
}
}

使用 trait 模拟硬件接口

在 C 中，模拟硬件需要预处理器技巧或函数指针交换。在 Rust 中，trait 使这变得自然：

#![allow(unused)]
fn main() {
// 用于 IPMI 通信的生产 trait
trait IpmiTransport {
    fn send_command(&self, cmd: u8, data: &[u8]) -> Result<Vec<u8>, String>;
}

// 真实实现（用于生产）
struct RealIpmi { /* BMC 连接详情 */ }
impl IpmiTransport for RealIpmi {
    fn send_command(&self, cmd: u8, data: &[u8]) -> Result<Vec<u8>, String> {
        // 实际与 BMC 硬件通信
        todo!("Real IPMI call")
    }
}

// 模拟实现（用于测试）
struct MockIpmi {
    responses: std::collections::HashMap<u8, Vec<u8>>,
}
impl IpmiTransport for MockIpmi {
    fn send_command(&self, cmd: u8, _data: &[u8]) -> Result<Vec<u8>, String> {
        self.responses.get(&cmd)
            .cloned()
            .ok_or_else(|| format!("No mock response for cmd 0x{cmd:02x}"))
    }
}

// 适用于真实和模拟的通用函数
fn read_sensor_temperature(transport: &dyn IpmiTransport) -> Result<f64, String> {
    let response = transport.send_command(0x2D, &[])?;
    if response.len() < 2 {
        return Err("Response too short".into());
    }
    Ok(response[0] as f64 + (response[1] as f64 / 256.0))
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_temperature_reading() {
        let mut mock = MockIpmi { responses: std::collections::HashMap::new() };
        mock.responses.insert(0x2D, vec![72, 128]); // 72.5°C

        let temp = read_sensor_temperature(&mock).unwrap();
        assert!((temp - 72.5).abs() < 0.01);
    }

    #[test]
    fn test_short_response() {
        let mock = MockIpmi { responses: std::collections::HashMap::new() };
        // 没有配置响应 → 错误
        assert!(read_sensor_temperature(&mock).is_err());
    }
}
}

使用 proptest 进行基于属性的测试

不是测试特定值，而是测试必须始终成立的属性：

#![allow(unused)]
fn main() {
// Cargo.toml: [dev-dependencies] proptest = "1"
use proptest::prelude::*;

fn parse_sensor_id(s: &str) -> Option<u32> {
    s.strip_prefix("sensor_")?.parse().ok()
}

fn format_sensor_id(id: u32) -> String {
    format!("sensor_{id}")
}

proptest! {
    #[test]
    fn roundtrip_sensor_id(id in 0u32..10000) {
        // 属性：格式化然后解析应该返回原始值
        let formatted = format_sensor_id(id);
        let parsed = parse_sensor_id(&formatted);
        prop_assert_eq!(parsed, Some(id));
    }

    #[test]
    fn parse_rejects_garbage(s in "[^s].*") {
        // 属性：不以 's' 开头的字符串永远不应该解析
        let result = parse_sensor_id(&s);
        prop_assert!(result.is_none());
    }
}
}
