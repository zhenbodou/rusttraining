Rust for C/C++ Programmers

避免过度使用 clone()

你将学到： 为什么 .clone() 在 Rust 中是代码异味，如何重构所有权以消除不必要的复制，以及表明所有权设计问题的特定模式。

  • 来自 C++，.clone() 感觉像是安全的默认值——"只是复制它"。但过度克隆隐藏所有权问题并损害性能。
  • 经验法则：如果你克隆是为了满足借用检查器，你可能需要重构所有权。

何时 clone() 是错误的

#![allow(unused)]
fn main() {
// 坏：克隆 String 只是为了传递给仅读取它的函数
fn log_message(msg: String) {  // 不必要地获取所有权
    println!("[LOG] {}", msg);
}
let message = String::from("GPU 测试通过");
log_message(message.clone());  // 浪费：分配全新的 String
log_message(message);           // 原始值被消耗——克隆是多余的
}
#![allow(unused)]
fn main() {
// 好：接受借用——零分配
fn log_message(msg: &str) {    // 借用，不拥有
    println!("[LOG] {}", msg);
}
let message = String::from("GPU 测试通过");
log_message(&message);          // 无克隆，无分配
log_message(&message);          // 可以再次调用——message 未被消耗
}

真实示例：返回 &str 而不是克隆

#![allow(unused)]
fn main() {
// 示例：healthcheck.rs——返回借用的视图，零分配
pub fn serial_or_unknown(&self) -> &str {
    self.serial.as_deref().unwrap_or(UNKNOWN_VALUE)
}

pub fn model_or_unknown(&self) -> &str {
    self.model.as_deref().unwrap_or(UNKNOWN_VALUE)
}
}

C++ 等效将返回 const std::string& 或 std::string_view——但在 C++ 中两者都不检查生命周期。在 Rust 中，借用检查器保证返回的 &str 不能比 self 存活更久。

真实示例：静态字符串切片——根本无堆

#![allow(unused)]
fn main() {
// 示例：healthcheck.rs——编译时字符串表
const HBM_SCREEN_RECIPES: &[&str] = &[
    "hbm_ds_ntd", "hbm_ds_ntd_gfx", "hbm_dt_ntd", "hbm_dt_ntd_gfx",
    "hbm_burnin_8h", "hbm_burnin_24h",
];
}

在 C++ 中这通常是 std::vector<std::string>（首次使用时堆分配）。Rust 的 &'static [&'static str] 生活在只读内存中——零运行时成本。

何时 clone() 是适当的

情况为什么 clone 可以示例
Arc::clone() 用于线程增加引用计数（~1 ns），不复制数据let flag = stop_flag.clone();
将数据移动到生成的线程线程需要自己的副本let ctx = ctx.clone(); thread::spawn(move || { ... })
从 &self 字段提取无法从借用中移动当返回拥有的 String 时 self.name.clone()
Option 中包装的小型 Copy 类型.copied() 比 .clone() 更清晰opt.get(0).copied() 用于 Option<&u32> → Option<u32>

检查清单：我应该克隆吗？

  1. 我可以接受 &str / &T 而不是 String / T 吗？ → 借用，不克隆
  2. 我可以重构以避免需要两个所有者吗？ → 通过引用传递或使用作用域
  3. 这是 Arc::clone() 吗？ → 没关系，它是 O(1)
  4. 我将数据移动到线程/闭包吗？ → 克隆是必要的
  5. 我在热点循环中克隆吗？ → 分析并考虑借用或 Cow<T>

Cow<'a, T>：写时克隆——能借用时借用，必须时克隆

Cow（Clone on Write）是一个 enum，持有要么借用引用要么拥有值。它相当于"可能时避免分配，但如果需要修改则分配"。C++ 没有直接等效——最接近的是有时返回 const std::string& 有时返回 std::string 的函数。

为什么 Cow 存在

#![allow(unused)]
fn main() {
// 无 Cow——你必须选择：始终借用或始终克隆
fn normalize(s: &str) -> String {          // 始终分配！
    if s.contains(' ') {
        s.replace(' ', "_")               // 新 String（需要分配）
    } else {
        s.to_string()                     // 不必要的分配！
    }
}

// 使用 Cow——不变时借用，修改时分配
use std::borrow::Cow;

fn normalize(s: &str) -> Cow<'_, str> {
    if s.contains(' ') {
        Cow::Owned(s.replace(' ', "_"))    // 分配（必须修改）
    } else {
        Cow::Borrowed(s)                   // 零分配（直通）
    }
}
}

Cow 如何工作

use std::borrow::Cow;

// Cow<'a, str> 本质上是：
// enum Cow<'a, str> {
//     Borrowed(&'a str),     // 零成本引用
//     Owned(String),          // 堆分配的拥有值
// }

fn greet(name: &str) -> Cow<'_, str> {
    if name.is_empty() {
        Cow::Borrowed("stranger")         // 静态字符串——无分配
    } else if name.starts_with(' ') {
        Cow::Owned(name.trim().to_string()) // 修改——需要分配
    } else {
        Cow::Borrowed(name)               // 直通——无分配
    }
}

fn main() {
    let g1 = greet("Alice");     // Cow::Borrowed("Alice")
    let g2 = greet("");          // Cow::Borrowed("stranger")
    let g3 = greet(" Bob ");     // Cow::Owned("Bob")

    // Cow<str> 实现 Deref<Target = str>，所以你可以将它用作 &str：
    println!("Hello, {g1}!");    // 有效——Cow 自动解引用为 &str
    println!("Hello, {g2}!");
    println!("Hello, {g3}!");
}

真实用例：配置值规范化

use std::borrow::Cow;

/// 规范化 SKU 名称：修剪空白，小写。
/// 如果已规范化则返回 Cow::Borrowed（零分配）。
fn normalize_sku(sku: &str) -> Cow<'_, str> {
    let trimmed = sku.trim();
    if trimmed == sku && sku.chars().all(|c| c.is_lowercase() || !c.is_alphabetic()) {
        Cow::Borrowed(sku)   // 已规范化——无分配
    } else {
        Cow::Owned(trimmed.to_lowercase())  // 需要修改——分配
    }
}

fn main() {
    let s1 = normalize_sku("server-x1");   // Borrowed——零分配
    let s2 = normalize_sku("  Server-X1 "); // Owned——必须分配
    println!("{s1}, {s2}"); // "server-x1, server-x1"
}

何时使用 Cow

情况使用 Cow？
函数大多数时候返回输入不变✅ 是——避免不必要的克隆
解析/规范化字符串（修剪、小写、替换）✅ 是——通常输入已经有效
始终修改——每个代码路径分配❌ 否——只返回 String
简单直通（从不修改）❌ 否——只返回 &str
长期存储在结构中的数据❌ 否——使用 String（拥有）

C++ 比较：Cow<str> 像返回 std::variant<std::string_view, std::string> 的函数 ——除了自动解引用且无样板访问值。

Weak<T>：打破引用循环——Rust 的 weak_ptr

Weak<T> 是 C++ std::weak_ptr<T> 的 Rust 等效。它持有对 Rc<T> 或 Arc<T> 值的非拥有引用。当 Weak 引用仍然存在时值可以被释放——如果值消失则调用 upgrade() 返回 None。

为什么 Weak 存在

如果两个值相互指向，Rc<T> 和 Arc<T> 创建引用循环——都不会达到 refcount 0，所以都不会被删除（内存泄漏）。 Weak 打破循环：

use std::rc::{Rc, Weak};
use std::cell::RefCell;

#[derive(Debug)]
struct Node {
    value: String,
    parent: RefCell<Weak<Node>>,      // Weak——不阻止父节点删除
    children: RefCell<Vec<Rc<Node>>>,  // Strong——父节点拥有子节点
}

impl Node {
    fn new(value: &str) -> Rc<Node> {
        Rc::new(Node {
            value: value.to_string(),
            parent: RefCell::new(Weak::new()),
            children: RefCell::new(Vec::new()),
        })
    }

    fn add_child(parent: &Rc<Node>, child: &Rc<Node>) {
        // 子节点获得父节点的弱引用（无循环）
        *child.parent.borrow_mut() = Rc::downgrade(parent);
        // 父节点获得子节点的强引用
        parent.children.borrow_mut().push(Rc::clone(child));
    }
}

fn main() {
    let root = Node::new("root");
    let child = Node::new("child");
    Node::add_child(&root, &child);

    // 通过 upgrade() 从子节点访问父节点
    if let Some(parent) = child.parent.borrow().upgrade() {
        println!("子节点的父节点：{}", parent.value); // "root"
    }

    println!("根强计数：{}", Rc::strong_count(&root));  // 1
    println!("根弱计数：{}", Rc::weak_count(&root));      // 1
}

C++ 比较

// C++——weak_ptr 打破 shared_ptr 循环
struct Node {
    std::string value;
    std::weak_ptr<Node> parent;                  // Weak——无所有权
    std::vector<std::shared_ptr<Node>> children;  // Strong——拥有子节点

    static auto create(const std::string& v) {
        return std::make_shared<Node>(Node{v, {}, {}});
    }
};

auto root = Node::create("root");
auto child = Node::create("child");
child->parent = root;          // weak_ptr 赋值
root->children.push_back(child);

if (auto p = child->parent.lock()) {   // lock() → shared_ptr 或 null
    std::cout << "父节点：" << p->value << std::endl;
}

何时使用 Weak

情况模式
父↔子树关系父节点持有 Rc<Child>，子节点持有 Weak<Parent>
观察者模式/事件监听器事件源持有 Weak<Observer>，观察者持有 Rc<Source>

在新代码中优先使用 arena 模式（案例研究 2）而不是 Rc/Weak 用于树结构。 Vec<T> + 索引更简单、更快，且零引用计数开销。当你需要具有动态生命周期的共享所有权时使用 Rc/Weak。

Copy vs Clone，PartialEq vs Eq——何时派生什么

  • Copy ≈ C++ 可平凡复制（无自定义复制构造函数/析构函数）。 像 int、enum 和简单 POD 结构这样的类型——编译器自动生成按位 memcpy。在 Rust 中，Copy 是相同的想法：赋值 let b = a; 执行隐式按位复制且两个变量都保持有效。
  • Clone ≈ C++ 复制构造函数/operator= 深复制。 当 C++ 类有自定义复制构造函数（例如，深复制 std::vector 成员）时，Rust 中的等效是实现 Clone。你必须显式调用 .clone()——Rust 从不隐藏 = 后面的昂贵复制。
  • 关键区别： 在 C++ 中，平凡复制和深复制都通过相同的 = 语法隐式发生。Rust 强制你选择：Copy 类型静默复制（便宜），非 Copy 类型默认移动，你必须选择显式复制使用 .clone()。
  • 类似地，C++ operator== 不区分 a == a 始终成立的类型（如整数）和它不成立的类型（如带 NaN 的 float）。Rust 将此编码到 PartialEq vs Eq。
