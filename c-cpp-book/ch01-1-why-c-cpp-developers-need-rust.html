Rust 消除的内容——完整列表

在深入示例之前，这里是执行摘要。安全 Rust 在结构上防止 此列表中的每个问题——不是通过纪律、工具或代码审查，而是通过类型系统和编译器：

内存泄漏（无需 GC）✅✅Drop trait = 正确的 RAII；自动清理，无需规则五
悬空指针✅✅生命周期系统在编译时证明引用比其引用物存活更久

整数溢出/下溢 UB✅✅Debug 构建在溢出时 panic；release 构建回绕（无论哪种方式都是定义的行为）

底线： 这些不是通过编码标准强制执行的理想目标。它们是编译时保证。如果你的代码编译通过，这些 bug 就不可能存在。

C 和 C++ 共有的问题

想跳过示例？ 跳转到 Rust 如何解决所有这些问题 或直接到 给我看些代码

两种语言共享一组核心的内存安全问题，这是超过 70% CVE（常见漏洞和暴露）的根本原因：

缓冲区溢出

C 数组、指针和字符串没有内在边界。超出它们非常容易：

#include <stdlib.h>
#include <string.h>

void buffer_dangers() {
    char buffer[10];
    strcpy(buffer, "This string is way too long!");  // 缓冲区溢出

    int arr[5] = {1, 2, 3, 4, 5};
    int *ptr = arr;           // 丢失大小信息
    ptr[10] = 42;             // 无边界检查——未定义行为
}

在 C++ 中，std::vector::operator[] 仍然不执行边界检查。只有 .at() 会——但谁会捕获异常呢？

悬空指针和释放后使用

int *bar() {
    int i = 42;
    return &i;    // 返回栈变量地址——悬空！
}

void use_after_free() {
    char *p = (char *)malloc(20);
    free(p);
    *p = '\0';   // 释放后使用——未定义行为
}

未初始化的变量和未定义行为

C 和 C++ 都允许未初始化的变量。结果值是不确定的，读取它们是未定义行为：

int x;               // 未初始化
if (x > 0) { ... }  // UB——x 可能是任何值

整数溢出在 C 中对无符号类型是定义的，但对有符号类型是未定义的。在 C++ 中，有符号溢出也是未定义行为。两种编译器都可以并利用这一点进行"优化"，以令人惊讶的方式破坏程序。

NULL 指针解引用

int *ptr = NULL;
*ptr = 42;           // SEGV——但编译器不会阻止你

在 C++ 中，std::optional<T> 有帮助，但很冗长，经常被 .value() 绕过，这会抛出异常。

C++ 在此基础上增加了更多问题

C 读者：如果你不使用 C++，可以 跳过 ahead 到 Rust 如何解决这些问题。

C++ 引入了智能指针、RAII、移动语义和异常来解决 C 的问题。这些只是创可贴，而非治愈——它们将失败模式从"运行时崩溃"转变为"运行时更微妙的 bug"：

unique_ptr 和 shared_ptr——创可贴，而非解决方案

C++ 智能指针相比原始 malloc/free 是重大改进，但它们没有解决根本问题：

C++ 缓解措施修复什么不修复什么
std::unique_ptr通过 RAII 防止泄漏移动后使用仍然编译；留下僵尸 nullptr
std::shared_ptr共享所有权引用循环静默泄漏；weak_ptr 纪律是手动的
std::optional替换一些空指针使用.value() 在空时抛出——隐藏控制流
std::string_view避免复制如果源字符串被释放则悬空——无生命周期检查
移动语义高效传输移动后的对象处于**"有效但未指定状态"**——UB 等待发生
RAII自动清理需要规则五才能正确；一个错误破坏一切
// unique_ptr：移动后使用干净地编译
std::unique_ptr<int> ptr = std::make_unique<int>(42);
std::unique_ptr<int> ptr2 = std::move(ptr);
std::cout << *ptr;  // 编译！运行时未定义行为。
                     // 在 Rust 中，这是编译错误："移动后使用值"

// shared_ptr：引用循环静默泄漏
struct Node {
    std::shared_ptr<Node> next;
    std::shared_ptr<Node> parent;  // 循环！析构函数永远不会被调用。
};
auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;
b->parent = a;  // 内存泄漏——引用计数永远不会达到 0
                 // 在 Rust 中，Rc<T> + Weak<T> 使循环显式且可打破

移动后使用——沉默的杀手

C++ std::move 不是移动——它是转换。原始对象保持在"有效但未指定状态"。编译器让你继续使用它：

auto vec = std::make_unique<std::vector<int>>({1, 2, 3});
auto vec2 = std::move(vec);
vec->size();  // 编译！但解引用 nullptr——运行时崩溃

在 Rust 中，移动是破坏性的。原始绑定消失了：

#![allow(unused)]
fn main() {
let vec = vec![1, 2, 3];
let vec2 = vec;           // 移动——vec 被消耗
// vec.len();             // 编译错误：移动后使用值
}

迭代器失效——来自生产 C++ 的真实 bug

这些不是人为的示例——它们代表在大型 C++ 代码库中发现的真实 bug 模式：

// BUG 1：erase  without reassigning iterator（未定义行为）
while (it != pending_faults.end()) {
    if (*it != nullptr && (*it)->GetId() == fault->GetId()) {
        pending_faults.erase(it);   // ← 迭代器失效！
        removed_count++;            //   下次循环使用悬空迭代器
    } else {
        ++it;
    }
}
// 修复：it = pending_faults.erase(it);

// BUG 2：基于索引的 erase 跳过元素
for (auto i = 0; i < entries.size(); i++) {
    if (config_status == ConfigDisable::Status::Disabled) {
        entries.erase(entries.begin() + i);  // ← 移动元素
    }                                         //   i++ 跳过移动的那个
}

// BUG 3：一个 erase 路径正确，另一个不是
while (it != incomplete_ids.end()) {
    if (current_action == nullptr) {
        incomplete_ids.erase(it);  // ← BUG：迭代器未重新赋值
        continue;
    }
    it = incomplete_ids.erase(it); // ← 正确路径
}

这些编译时没有任何警告。 在 Rust 中，借用检查器使所有三个成为编译错误——你不能在迭代集合时修改它，就这样。

异常安全和 dynamic_cast/new 模式

现代 C++ 代码库仍然严重依赖没有编译时安全的模式：

// 典型的 C++ 工厂模式——每个分支都是潜在的 bug
DriverBase* driver = nullptr;
if (dynamic_cast<ModelA*>(device)) {
    driver = new DriverForModelA(framework);
} else if (dynamic_cast<ModelB*>(device)) {
    driver = new DriverForModelB(framework);
}
// 如果 driver 仍然是 nullptr 怎么办？如果 new 抛出怎么办？谁拥有 driver？

在典型的 10 万行 C++ 代码库中，你可能会发现数百个 dynamic_cast 调用（每个都是潜在的运行时失败）、数百个原始 new 调用（每个都是潜在的泄漏）和数百个 virtual/override 方法（到处是 vtable 开销）。

悬空引用和 lambda 捕获

int& get_reference() {
    int x = 42;
    return x;  // 悬空引用——编译，运行时 UB
}

auto make_closure() {
    int local = 42;
    return [&local]() { return local; };  // 悬空捕获！
}

Rust 如何解决所有这些问题

上面列出的每个问题——来自 C 和 C++——都被 Rust 的编译时保证防止：

移动后使用移动是破坏性的——编译器拒绝让你触碰原始值
内存泄漏Drop trait = 无规则五的 RAII；自动、正确的清理
引用循环所有权是树形的；Rc + Weak 使循环显式

数据竞争Send/Sync trait 使数据竞争成为编译错误
未初始化的变量所有变量必须初始化；编译器强制执行
整数 UBDebug 在溢出时 panic；release 回绕（都是定义的行为）
异常没有异常；Result<T, E> 在类型签名中可见，用 ? 传播
