Rust for Python Programmers

理解所有权

你将学到： 为什么 Rust 有所有权（无 GC！），move 语义 vs Python 的引用计数， 借用（& 和 &mut），生命周期基础，以及智能指针（Box、Rc、Arc）。

难度： 🟡 中级

这是 Python 开发者最难的概念。在 Python 中，你从不考虑谁"拥有" 数据 —— 垃圾回收器处理它。在 Rust 中，每个值恰好有一个 所有者，编译器在编译时跟踪这个。

Python：到处共享引用

# Python —— 一切都是引用，gc 清理
a = [1, 2, 3]
b = a              # b 和 a 指向同一个列表
b.append(4)
print(a)            # [1, 2, 3, 4] —— 惊喜！a 也变了

# 谁拥有这个列表？a 和 b 都引用它。
# 垃圾回收器在没有引用时释放它。
# 你从不考虑这个。

Rust：单一所有权

#![allow(unused)]
fn main() {
// Rust —— 每个值恰好有一个所有者
let a = vec![1, 2, 3];
let b = a;           # 所有权从 a MOVE 到 b
println!("{:?}", a); # ❌ 编译错误：move 后使用值

a 不再存在。b 是唯一所有者。
println!("{:?}", b); # ✅ [1, 2, 3]

当 b 超出作用域时，Vec 被释放。确定性的。无 GC。
}

三个所有权规则

#![allow(unused)]
fn main() {
1. 每个值恰好有一个所有者变量。
2. 当所有者超出作用域时，值被 drop（释放）。
3. 所有权可以转移（move）但不能复制（除非 Clone）。
}

Move 语义 —— Python 最大的冲击

# Python —— 赋值复制引用，不是数据
def process(data):
    data.append(42)
    # 原始列表被修改！

my_list = [1, 2, 3]
process(my_list)
print(my_list)       # [1, 2, 3, 42] —— 被 process 修改！

#![allow(unused)]
fn main() {
// Rust —— 传递给函数 MOVE 所有权（对于非 Copy 类型）
fn process(mut data: Vec<i32>) -> Vec<i32> {
    data.push(42);
    data  # 必须返回它以归还所有权！
}

let my_vec = vec![1, 2, 3];
let my_vec = process(my_vec);  # 所有权移入并移出
println!("{:?}", my_vec);      # [1, 2, 3, 42]

或更好 —— 借用而不是 move：
fn process_borrowed(data: &mut Vec<i32>) {
    data.push(42);
}

let mut my_vec = vec![1, 2, 3];
process_borrowed(&mut my_vec);  # 临时借出
println!("{:?}", my_vec);       # [1, 2, 3, 42] —— 仍然是我们的
}

所有权可视化

Python：                              Rust：

  a ──────┐                           a ──→ [1, 2, 3]
           ├──→ [1, 2, 3]
  b ──────┘                           之后：let b = a;

  （a 和 b 共享一个对象）          a（无效，已 move）
  （refcount = 2）                  b ──→ [1, 2, 3]
                                    （只有 b 拥有数据）

  del a → refcount = 1              drop(b) → 数据释放
  del b → refcount = 0 → 释放        （确定性，无 GC）

Move 语义 vs 引用计数

Copy vs Move

#![allow(unused)]
fn main() {
// 简单类型（整数、浮点数、bool、char）被复制，不是 move
let x = 42;
let y = x;    # x 被复制到 y（两者都有效）
println!("{x} {y}");  # ✅ 42 42

堆分配类型（String、Vec、HashMap）被 move
let s1 = String::from("hello");
let s2 = s1;  # s1 被 move 到 s2
println!("{s1}");  # ❌ 错误：move 后使用值

要显式复制堆数据，使用 .clone()
let s1 = String::from("hello");
let s2 = s1.clone();  # 深复制
println!("{s1} {s2}");  # ✅ hello hello（两者都有效）
}

Python 开发者的心智模型

Python：                    Rust：
─────────                  ─────
int, float, bool           Copy 类型（i32、f64、bool、char）
→ 赋值时复制              → 赋值时复制（行为相似）
                           （注意：Python 缓存小整数；Rust 复制始终可预测）

list, dict, str            Move 类型（Vec、HashMap、String）
→ 共享引用                → 所有权转移（行为不同！）
→ gc 清理                 → 所有者 drop 数据
→ 用 list(x) 克隆          → 用 x.clone() 克隆
   或 copy.deepcopy(x)

Python 的共享模型导致 bug 时

# Python —— 意外别名
def remove_duplicates(items):
    seen = set()
    result = []
    for item in items:
        if item not in seen:
            seen.add(item)
            result.append(item)
    return result

original = [1, 2, 2, 3, 3, 3]
alias = original          # 别名，不是复制
unique = remove_duplicates(alias)
# original 仍然是 [1, 2, 2, 3, 3, 3] —— 但只是因为我们没有修改
# 如果 remove_duplicates 修改输入，original 也会受影响

#![allow(unused)]
fn main() {
use std::collections::HashSet;

// Rust —— 所有权防止意外别名
fn remove_duplicates(items: &[i32]) -> Vec<i32> {
    let mut seen = HashSet::new();
    items.iter()
        .filter(|&&item| seen.insert(item))
        .copied()
        .collect()
}

let original = vec![1, 2, 2, 3, 3, 3];
let unique = remove_duplicates(&original); # 借用 —— 不能修改
original 保证不变 —— 编译器通过 & 阻止修改
}

借用和生命周期

借用 = 借书

#![allow(unused)]

将所有权想象成一本实体书：

Python：  每个人都有复印件（共享引用 + GC）

生命周期回答："这个引用存活多久？"
通常编译器推断它们。你很少显式编写它们。

简单情况 —— 编译器处理：
fn first_word(s: &str) -> &str {
    s.split_whitespace().next().unwrap_or("")
}
编译器知道：返回的 &str 与输入 &str 存活时间相同

当你需要显式生命周期时（罕见）：
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}
'a 说："返回值与两个输入存活时间相同"
}

对于 Python 开发者：最初不要担心生命周期。编译器会在你需要时告诉你，95% 的时间它会自动推断它们。 将生命周期注解视为你在编译器无法自行找出关系时给它的提示。

智能指针

对于单一所有权太受限的情况，Rust 提供智能指针。 这些更接近 Python 的引用模型 —— 但显式且选择加入。

#![allow(unused)]
fn main() {
// Box<T> —— 堆分配，单一所有者（如 Python 的正常分配）
let boxed = Box::new(42);  # 堆分配 i32

// Rc<T> —— 引用计数（如 Python 的 refcount！）
use std::rc::Rc;
let shared = Rc::new(vec![1, 2, 3]);
let clone1 = Rc::clone(&shared);  # 增加 refcount
let clone2 = Rc::clone(&shared);  # 增加 refcount
所有三个指向同一个 Vec。当所有都被 drop 时，Vec 被释放。
类似于 Python 的引用计数，但 Rc 不处理循环 ——
使用 Weak<T> 打破循环（Python 的 GC 自动处理循环）

// Arc<T> —— 原子引用计数（用于多线程代码的 Rc）
use std::sync::Arc;
let thread_safe = Arc::new(vec![1, 2, 3]);
跨线程共享时使用 Arc（Rc 是单线程的）

// RefCell<T> —— 运行时借用检查（如 Python 的"什么都可以"模型）
use std::cell::RefCell;
let cell = RefCell::new(42);
*cell.borrow_mut() = 99;  # 运行时可变借用（如果双重借用则 panic）
}

何时使用每个

智能指针Python 类比用例
Box<T>正常分配大数据、递归类型、trait 对象
Rc<T>Python 默认 refcount共享所有权，单线程
Arc<T>线程安全 refcount共享所有权，多线程
RefCell<T>Python 的"就修改它"内部可变性（逃生舱）
Rc<RefCell<T>>Python 正常对象模型共享 + 可变（图结构）

关键见解：Rc<RefCell<T>> 给你 Python 般的语义（共享、可变数据） 但你必须显式选择加入。Rust 的默认（拥有、move）更快并避免 引用计数的开销。对于带循环的图状结构，使用 Weak<T> 打破引用循环 —— 与 Python 不同，Rust 的 Rc 没有循环收集器。

📌 另见：第 13 章 — 并发 涵盖用于多线程共享状态的 Arc<Mutex<T>>。

练习

🏋️ 练习：发现 Borrow Checker 错误（点击展开）

挑战：以下代码有 3 个 borrow checker 错误。识别每个错误并修复它们，不使用 .clone()：

fn main() {
    let mut names = vec!["Alice".to_string(), "Bob".to_string()];
    let first = &names[0];
    names.push("Charlie".to_string());
    println!("First: {first}");

    let greeting = make_greeting(names[0]);
    println!("{greeting}");
}

fn make_greeting(name: String) -> String {
    format!("Hello, {name}!")
}
🔑 解答 
fn main() {
    let mut names = vec!["Alice".to_string(), "Bob".to_string()];
    let first = &names[0];
    println!("First: {first}"); # 在可变之前使用借用
    names.push("Charlie".to_string()); # 现在安全 —— 无存活的不可变借用

    let greeting = make_greeting(&names[0]); # 传递引用，不是拥有
    println!("{greeting}");
}

fn make_greeting(name: &str) -> String { # 接受 &str，不是 String
    format!("Hello, {name}!")
}

修复的错误：

  1. 不可变借用 + 可变：first 借用 names，然后 push 修改它。修复：在 push 之前使用 first。
  2. 从 Vec move：names[0] 尝试从 Vec move String（不允许）。修复：用 &names[0] 借用。
  3. 函数获取所有权：make_greeting(String) 消耗值。修复：改为接受 &str。
