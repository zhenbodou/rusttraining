Rust for Python Programmers

无 GIL：真正的并行

你将学到： 为什么 GIL 限制 Python 并发，Rust 的 Send/Sync trait 用于编译时线程安全， Arc<Mutex<T>> vs Python threading.Lock，channel vs queue.Queue，以及 async/await 差异。

难度： 🔴 高级

GIL（全局解释器锁）是 Python 对 CPU 密集型工作的最大限制。 Rust 没有 GIL —— 线程真正并行运行，类型系统在编译时防止数据竞争。

关键见解：Python 线程对 CPU 工作顺序运行（GIL 序列化它们）。Rust 线程真正并行运行 —— 4 个线程 = 约 4 倍加速。

📌 前提：在 tackling 本章之前，确保你熟悉 第 7 章 — 所有权和借用。Arc、Mutex 和 move 闭包都建立在所有权概念之上。

Python 的 GIL 问题

# Python —— 线程对 CPU 密集型工作无帮助
import threading
import time

counter = 0

def increment(n):
    global counter
    for _ in range(n):
        counter += 1  # 不是线程安全！但 GIL"保护"简单操作

threads = [threading.Thread(target=increment, args=(1_000_000,)) for _ in range(4)]
start = time.perf_counter()
for t in threads:
    t.start()
for t in threads:
    t.join()
elapsed = time.perf_counter() - start

print(f"Counter: {counter}")    # 可能不是 4,000,000！
print(f"Time: {elapsed:.2f}s")  # 与单线程差不多（GIL）

# 对于真正并行，Python 需要多进程：
from multiprocessing import Pool
with Pool(4) as pool:
    results = pool.map(cpu_work, data)  # 独立进程，pickle 开销

Rust —— 真正并行，编译时安全

use std::sync::atomic::{AtomicI64, Ordering};
use std::sync::Arc;
use std::thread;

fn main() {
    let counter = Arc::new(AtomicI64::new(0));

    let handles: Vec<_> = (0..4).map(|_| {
        let counter = Arc::clone(&counter);
        thread::spawn(move || {
            for _ in 0..1_000_000 {
                counter.fetch_add(1, Ordering::Relaxed);
            }
        })
    }).collect();

    for h in handles {
        h.join().unwrap();
    }

    println!("Counter: {}", counter.load(Ordering::Relaxed)); # 始终 4,000,000
    在所有核心上运行 —— 真正并行，无 GIL
}

线程安全：类型系统保证

Python —— 运行时错误

# Python —— 数据竞争在运行时捕获（或根本不）
import threading

shared_list = []

def append_items(items):
    for item in items:
        shared_list.append(item)  # 由于 GIL，append"线程安全"
        # 但复杂操作不安全：
        # if item not in shared_list:
        #     shared_list.append(item)  # 竞争条件！

# 使用 Lock 保证安全：
lock = threading.Lock()
def safe_append(items):
    for item in items:
        with lock:
            if item not in shared_list:
                shared_list.append(item)
# 忘记锁？无编译器警告。bug 在生产中发现。

Rust —— 编译时错误

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    尝试在无保护下跨线程共享 Vec：
    let shared = vec![];
    thread::spawn(move || shared.push(1));
    ❌ 编译错误：Vec 无保护不是 Send/Sync

    带 Mutex（Rust 的 threading.Lock 等价物）：
    let shared = Arc::new(Mutex::new(Vec::new()));

    let handles: Vec<_> = (0..4).map(|i| {
        let shared = Arc::clone(&shared);
        thread::spawn(move || {
            let mut data = shared.lock().unwrap(); # 访问需要锁
            data.push(i);
            `data` 超出作用域时锁自动释放
            无"忘记解锁" —— RAII 保证
        })
    }).collect();

    for h in handles {
        h.join().unwrap();
    }

    println!("{:?}", shared.lock().unwrap()); # [0, 1, 2, 3]（顺序可能不同）
}

Send 和 Sync Trait

#![allow(unused)]
fn main() {
// Rust 使用两个标记 trait 强制执行线程安全：

Send —— "这个类型可以转移到另一个线程"
大多数类型是 Send。Rc<T> 不是（线程使用 Arc<T>）。

Sync —— "这个类型可以从多个线程引用"
大多数类型是 Sync。Cell<T>/RefCell<T> 不是（使用 Mutex<T>）。

编译器自动检查这些：
thread::spawn(move || { ... })
  ↑ 闭包的捕获必须是 Send
  ↑ 共享引用必须是 Sync
  ↑ 如果不是 → 编译错误

Python 没有等价物。线程安全 bug 在运行时发现。
Rust 在编译时捕获它们。这是"无畏并发"。
}

Mutex 中毒

如果线程在持有 Mutex 时 panic，锁变为中毒。Python 没有等价物 —— 如果线程在持有 threading.Lock() 时崩溃，锁保持卡住。

#![allow(unused)]
fn main() {
use std::sync::{Arc, Mutex};
use std::thread;

let data = Arc::new(Mutex::new(vec![1, 2, 3]));
let data2 = Arc::clone(&data);

let _ = thread::spawn(move || {
    let mut guard = data2.lock().unwrap();
    guard.push(4);
    panic!("oops!");  # 锁现在中毒
}).join();

后续锁尝试返回 Err(PoisonError)
match data.lock() {
    Ok(guard) => println!("Data: {guard:?}"),
    Err(poisoned) => {
        println!("Lock was poisoned! Recovering...");
        let guard = poisoned.into_inner();
        println!("Recovered: {guard:?}");  # [1, 2, 3, 4]
    }
}
}

原子排序（简要说明）

原子操作上的 Ordering 参数控制内存可见性保证：

Ordering何时使用
Relaxed顺序不重要的简单计数器
Acquire/Release生产者 - 消费者：写入者使用 Release，读取者使用 Acquire
SeqCst如有疑问 —— 最严格排序，最直观

Python 的 threading 模块在 GIL 后隐藏这些细节。在 Rust 中，你显式选择 —— 使用 SeqCst 直到性能分析显示你需要更弱的。

async/await 比较

Python 和 Rust 都有 async/await 语法，但它们在底层工作非常不同。

Python async/await

# Python —— asyncio 用于并发 I/O
import asyncio
import aiohttp

async def fetch_url(session, url):
    async with session.get(url) as resp:
        return await resp.text()

async def main():
    urls = ["https://example.com", "https://httpbin.org/get"]

    async with aiohttp.ClientSession() as session:
        tasks = [fetch_url(session, url) for url in urls]
        results = await asyncio.gather(*tasks)

    for url, result in zip(urls, results):
        print(f"{url}: {len(result)} bytes")

asyncio.run(main())

# Python async 是单线程的（仍然 GIL）！
# 它只对 I/O 密集型工作有帮助（等待网络/磁盘）。
# async 中的 CPU 密集型工作仍然阻塞事件循环。

Rust async/await

// Rust —— tokio 用于并发 I/O（和 CPU 并行！）
use reqwest;
use tokio;

async fn fetch_url(url: &str) -> Result<String, reqwest::Error> {
    reqwest::get(url).await?.text().await
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let urls = vec!["https://example.com", "https://httpbin.org/get"];

    let tasks: Vec<_> = urls.iter()
        .map(|url| tokio::spawn(fetch_url(url)))  # 无 GIL 限制
        .collect()                                 # 可以使用所有 CPU 核心

    let results = futures::future::join_all(tasks).await;

    for (url, result) in urls.iter().zip(results) {
        match result {
            Ok(Ok(body)) => println!("{url}: {} bytes", body.len()),
            Ok(Err(e)) => println!("{url}: error {e}"),
            Err(e) => println!("{url}: task failed {e}"),
        }
    }

    Ok(())
}

def process_item(item):

with Pool(8) as pool:

💼 案例研究：并行图像处理管道

一个数据科学团队每晚处理 50,000 张卫星图像。他们的 Python 管道使用 multiprocessing.Pool：

# Python —— CPU 密集型图像工作的 multiprocessing
import multiprocessing
from PIL import Image
import numpy as np

def process_image(path: str) -> dict:
    img = np.array(Image.open(path))
    # CPU 密集型：直方图均衡化、边缘检测、分类
    histogram = np.histogram(img, bins=256)[0]
    edges = detect_edges(img)       # 约 200ms/图像
    label = classify(edges)          # 约 100ms/图像
    return {"path": path, "label": label, "edge_count": len(edges)}

# 问题：每个子进程复制完整 Python 解释器
# 内存：50MB/工作者 × 16 工作者 = 800MB 开销
# 启动：2-3 秒 fork 和 pickle 参数
with multiprocessing.Pool(16) as pool:
    results = pool.map(process_image, image_paths)  # 50k 图像约 4.5 小时

痛点：fork 的 800MB 内存开销，参数/结果的 pickle 序列化，GIL 阻止使用线程，错误处理不透明（工作者中的异常难以调试）。

use rayon::prelude::*;
use image::GenericImageView;

struct ImageResult {
    path: String,
    label: String,
    edge_count: usize,
}

fn process_image(path: &str) -> Result<ImageResult, image::ImageError> {
    let img = image::open(path)?;
    let histogram = compute_histogram(&img);       # 约 50ms（无 numpy 开销）
    let edges = detect_edges(&img);                # 约 40ms（SIMD 优化）
    let label = classify(&edges);                  # 约 20ms
    Ok(ImageResult {
        path: path.to_string(),
        label,
        edge_count: edges.len(),
    })
}

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let paths: Vec<String> = load_image_paths()?;

    Rayon 自动使用所有 CPU 核心 —— 无 fork，无 pickle，无 GIL
    let results: Vec<ImageResult> = paths
        .par_iter()                                # 并行迭代器
        .filter_map(|p| process_image(p).ok())     # 优雅跳过错误
        .collect();                                # 并行收集

    println!("Processed {} images", results.len());
    Ok(())
}
50k 图像约 35 分钟（vs Python 4.5 小时）
内存：约 50MB 总计（共享线程，无 fork）

结果：

指标Python (multiprocessing)Rust (rayon)
时间（50k 图像）约 4.5 小时约 35 分钟
内存开销800MB（16 工作者）约 50MB（共享）
错误处理不透明 pickle 错误每步 Result<T, E>
启动成本2-3 秒（fork + pickle）无（线程）

关键教训：对于 CPU 密集型并行工作，Rust 的线程 + rayon 替换 Python 的 multiprocessing，无序列化开销、共享内存和编译时安全。

练习

🏋️ 练习：线程安全计数器（点击展开）

挑战：在 Python 中，你可能使用 threading.Lock 保护共享计数器。将此转换为 Rust：生成 10 个线程，每个递增共享计数器 1000 次。打印最终值（应为 10000）。使用 Arc<Mutex<u64>>。

🔑 解答 
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0u64));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            for _ in 0..1000 {
                let mut num = counter.lock().unwrap();
                *num += 1;
            }
        }));
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Final count: {}", *counter.lock().unwrap());
}

关键要点：Arc<Mutex<T>> 是 Python 的 lock = threading.Lock() + 共享变量的 Rust 等价物 —— 但 Rust 不会编译 如果你忘记 Arc 或 Mutex。Python 愉快地运行有竞争的程序并静默给你错误答案。
