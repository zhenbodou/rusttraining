公共 API 设计清单

  1. 接受引用，返回自有 —— fn process(input: &str) -> String
  2. 对参数使用 impl Trait —— fn read(r: impl Read) 而非 fn read<R: Read>(r: R) 以获得更清晰的签名
  3. 返回 Result，不是 panic! —— 让调用者决定如何处理错误
  4. 实现标准 trait —— Debug、Display、Clone、Default、From/Into
  5. 使无效状态不可表示 —— 使用 type state 和 newtype
  6. 对复杂配置使用构建器模式 —— 如果字段必需则带 type-state
  7. 密封你不想让用户实现的 trait —— pub trait Sealed: private::Sealed {}
  8. 标记类型和函数为 #[must_use] —— 防止静默丢弃重要的 Result、guard 或值。应用于任何忽略返回值几乎肯定是 bug 的类型： 
    #![allow(unused)]
fn main() {
#[must_use = "dropping the guard immediately releases the lock"]
pub struct LockGuard<'a, T> { /* ... */ }

#[must_use]
pub fn validate(input: &str) -> Result<ValidInput, ValidationError> { /* ... */ }
}
#![allow(unused)]
fn main() {
// 密封 trait 模式——用户可以使用但不能实现：
mod private {
    pub trait Sealed {}
}

pub trait DatabaseDriver: private::Sealed {
    fn connect(&self, url: &str) -> Connection;
}

// 只有这个 crate 中的类型可以实现 Sealed → 只有我们可以实现 DatabaseDriver
pub struct PostgresDriver;
impl private::Sealed for PostgresDriver {}
impl DatabaseDriver for PostgresDriver {
    fn connect(&self, url: &str) -> Connection { /* ... */ }
}
}

#[non_exhaustive] —— 标记公共 enum 和结构体，以便添加变体或字段不是破坏性更改。下游 crate 必须在 match 语句中使用通配符臂（_ =>），并且不能用结构体字面量语法构造类型：

#![allow(unused)]
fn main() {
#[non_exhaustive]
pub enum DiagError {
    Timeout,
    HardwareFault,
    // 在未来版本中添加新变体不是 semver 破坏。
}
}

人性化参数模式——impl Into、AsRef、Cow

Rust 最有影响力的 API 模式之一是在函数参数中接受最通用类型， 所以调用者不需要在每个调用点重复 .to_string()、&*s 或 .as_ref()。 这是 Rust 特定版本的"接受什么要宽松"。

impl Into<T> —— 接受任何可转换的

#![allow(unused)]
fn main() {
// ❌ 摩擦：调用者必须手动转换
fn connect(host: String, port: u16) -> Connection {
    // ...
}
connect("localhost".to_string(), 5432);  // 烦人的 .to_string()
connect(hostname.clone(), 5432);          // 如果我们已经有 String 则不必要的 clone

// ✅ 人性化：接受任何转换为 String 的东西
fn connect(host: impl Into<String>, port: u16) -> Connection {
    let host = host.into();  // 转换一次，在函数内
    // ...
}
connect("localhost", 5432);     // &str —— 零摩擦
connect(hostname, 5432);        // String —— 移动，无 clone
connect(arc_str, 5432);         // 如果 From 实现则 Arc<str>
}

这有效因为 Rust 的 From/Into trait 对提供 blanket 转换。 当你接受 impl Into<T> 时，你在说："给我任何知道如何成为 T 的东西。"

AsRef<T> —— 借用为引用

AsRef<T> 是 Into<T> 的借用对应物。当你只需要读取数据，不需要拥有时使用它：

#![allow(unused)]
fn main() {
use std::path::Path;

// ❌ 强制调用者转换为 &Path
fn file_exists(path: &Path) -> bool {
    path.exists()
}
file_exists(Path::new("/tmp/test.txt"));  // 笨拙

// ✅ 接受任何可以表现为 &Path 的东西
fn file_exists(path: impl AsRef<Path>) -> bool {
    path.as_ref().exists()
}
file_exists("/tmp/test.txt");                    // &str ✅
file_exists(String::from("/tmp/test.txt"));      // String ✅
file_exists(Path::new("/tmp/test.txt"));         // &Path ✅
file_exists(PathBuf::from("/tmp/test.txt"));     // PathBuf ✅

// 对字符串类似参数的相同模式：
fn log_message(msg: impl AsRef<str>) {
    println!("[LOG] {}", msg.as_ref());
}
log_message("hello");                    // &str ✅
log_message(String::from("hello"));      // String ✅
}

Cow<T> —— 写时克隆

Cow<'a, T>（Clone on Write）延迟分配直到需要变异。 它持有借用 &T 或自有 T::Owned。当大多数调用不需要修改数据时这完美：

#![allow(unused)]
fn main() {
use std::borrow::Cow;

/// 标准化诊断消息——仅在需要更改时分配。
fn normalize_message(msg: &str) -> Cow<'_, str> {
    if msg.contains('\t') || msg.contains('\r') {
        // 必须分配——我们需要修改内容
        Cow::Owned(msg.replace('\t', "    ").replace('\r', ""))
    } else {
        // 无分配——只借用原始
        Cow::Borrowed(msg)
    }
}

// 大多数消息无分配通过：
let clean = normalize_message("All tests passed");          // Borrowed —— 免费
let fixed = normalize_message("Error:\tfailed\r\n");        // Owned —— 分配

// Cow<str> 实现 Deref<Target=str>，所以它像 &str 一样工作：
println!("{}", clean);
println!("{}", fixed.to_uppercase());
}

快速参考：使用哪个

你需要函数内数据的所有权吗？
├── 是 → impl Into<T>
│         "给我任何可以成为 T 的东西"
└── 否  → 你只需要读取它吗？
     ├── 是 → impl AsRef<T> 或 &T
     │         "给我任何我可以借用为 &T 的东西"
     └── 可能（有时可能需要修改？）
          └── Cow<'_, T>
              "可能时借用，必须时才克隆"
模式所有权分配何时使用
&str借用从不简单字符串参数
impl AsRef<str>借用从不接受 String、&str 等——只读
impl Into<String>自有转换时接受 &str、String —— 将存储/拥有
Cow<'_, str>任一仅当修改时通常不修改的处理
&[u8] / impl AsRef<[u8]>借用从不面向字节的 API

Borrow<T> vs AsRef<T>：两者提供 &T，但 Borrow<T> 额外保证 Eq、Ord 和 Hash 在原始和借用形式之间一致。这就是为什么 HashMap<String, V>::get() 接受 &Q where String: Borrow<Q> —— 不是 AsRef。当借用形式用作查找键时使用 Borrow；对一般"给我引用"参数使用 AsRef。

之后（类型安全、自文档化）：

    Lenient,

    todo!()

什么改进了：

方面之前之后
格式验证运行时字符串比较编译时 enum
路径类型原始 &str（可以是任何）&Path（文件系统特定）

解析非验证——TryFrom 和验证类型

"Parse, don't validate" 是一个原则，说：不要检查数据然后传递周围原始未检查形式——而是解析为只有在数据有效时才能存在的类型。 Rust 的 TryFrom trait 是这个的标准工具。

问题：无强制的验证

#![allow(unused)]
fn main() {
// ❌ 验证然后使用：无防止在检查后使用无效值
fn process_port(port: u16) {
    if port == 0 || port > 65535 {
        panic!("Invalid port");           // 我们检查了，但是...
    }
    start_server(port);                    // 如果有人直接调用 start_server(0) 怎么办？
}

// ❌ 字符串类型：email 只是 String —— 任何垃圾通过
fn send_email(to: String, body: String) {
    // `to` 实际上是有效 email 吗？我们不知道。
    // 有人可以传递 "not-an-email" 我们只在 SMTP 服务器发现。
}
}

解决方案：用 TryFrom 解析为验证的 Newtype

use std::convert::TryFrom;
use std::fmt;

/// 验证的 TCP 端口号（1–65535）。
/// 如果你有 `Port`，它保证有效。
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Port(u16);

impl TryFrom<u16> for Port {
    type Error = PortError;

    fn try_from(value: u16) -> Result<Self, Self::Error> {
        if value == 0 {
            Err(PortError::Zero)
        } else {
            Ok(Port(value))
        }
    }
}

impl Port {
    pub fn get(&self) -> u16 { self.0 }
}

#[derive(Debug)]
pub enum PortError {
    Zero,
    InvalidFormat,
}

impl fmt::Display for PortError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            PortError::Zero => write!(f, "port must be non-zero"),
            PortError::InvalidFormat => write!(f, "invalid port format"),
        }
    }
}

impl std::error::Error for PortError {}

// 现在类型系统强制有效性：
fn start_server(port: Port) {
    // 无需验证——Port 只能通过 TryFrom 构造，
    // 已经验证它有效。
    println!("Listening on port {}", port.get());
}

// 用法：
fn main() -> Result<(), Box<dyn std::error::Error>> {
    let port = Port::try_from(8080)?;   // ✅ 在边界验证一次
    start_server(port);                  // 下游任何地方无需重新验证

    let bad = Port::try_from(0);         // ❌ Err(PortError::Zero)
    Ok(())
}

真实示例：验证的 IPMI 地址

#![allow(unused)]
fn main() {
/// 验证的 IPMI 从地址（0x20–0xFE，仅偶数）。
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct IpmiAddr(u8);

#[derive(Debug)]
pub enum IpmiAddrError {
    Odd(u8),
    OutOfRange(u8),
}

impl fmt::Display for IpmiAddrError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            IpmiAddrError::Odd(v) => write!(f, "IPMI address 0x{v:02X} must be even"),
            IpmiAddrError::OutOfRange(v) => {
                write!(f, "IPMI address 0x{v:02X} out of range (0x20..=0xFE)")
            }
        }
    }
}

impl TryFrom<u8> for IpmiAddr {
    type Error = IpmiAddrError;

    fn try_from(value: u8) -> Result<Self, Self::Error> {
        if value % 2 != 0 {
            Err(IpmiAddrError::Odd(value))
        } else if value < 0x20 || value > 0xFE {
            Err(IpmiAddrError::OutOfRange(value))
        } else {
            Ok(IpmiAddr(value))
        }
    }
}

impl IpmiAddr {
    pub fn get(&self) -> u8 { self.0 }
}

// 下游代码从不需要重新检查：
fn send_ipmi_command(addr: IpmiAddr, cmd: u8, data: &[u8]) -> Result<Vec<u8>, IpmiError> {
    // addr.get() 保证是有效、偶数 IPMI 地址
    raw_ipmi_send(addr.get(), cmd, data)
}
}

用 FromStr 解析字符串

对通常从文本解析的类型（CLI 参数、配置文件），实现 FromStr：

#![allow(unused)]
fn main() {
use std::str::FromStr;

impl FromStr for Port {
    type Err = PortError;

    fn from_str(s: &str) -> Result<Self, Self::Err> {
        let n: u16 = s.parse().map_err(|_| PortError::InvalidFormat)?;
        Port::try_from(n)
    }
}

// 现在与 .parse() 一起工作：
let port: Port = "8080".parse()?;   // 一步验证

复杂验证的 TryFrom 链

#![allow(unused)]
fn main() {
/// 只有在所有字段有效时才能存在的验证配置。
pub struct ValidConfig {
    pub host: Hostname,
    pub port: Port,
    pub timeout_ms: Timeout,
}

impl TryFrom<RawConfig> for ValidConfig {
    type Error = ConfigError;

    fn try_from(raw: RawConfig) -> Result<Self, Self::Error> {
        Ok(ValidConfig {
            host: Hostname::try_from(raw.host)
                .map_err(ConfigError::InvalidHost)?,
            port: Port::try_from(raw.port)
                .map_err(ConfigError::InvalidPort)?,
            timeout_ms: Timeout::try_from(raw.timeout_ms)
                .map_err(ConfigError::InvalidTimeout)?,
        })
    }
}

// 在边界解析一次，在内部到处使用验证类型：
fn load_config(path: &str) -> Result<ValidConfig, ConfigError> {
    let raw: RawConfig = serde_json::from_str(&std::fs::read_to_string(path)?)?;
    ValidConfig::try_from(raw)  // 所有验证在这里发生
}
}

总结：验证 vs 解析

方法数据检查？编译器强制有效性？需要重新验证？
运行时检查（if/assert）✅❌每个函数边界
验证 newtype + TryFrom✅✅从不——类型是证明

规则：在边界解析，在内部到处使用验证类型。 原始字符串、整数和字节切片进入你的系统，通过 TryFrom/FromStr 解析为验证类型，从那时起类型系统保证它们有效。

Feature 标志和条件编译

# Cargo.toml
[features]
default = ["json"]          # 默认启用
json = ["dep:serde_json"]   # 启用 JSON 支持
xml = ["dep:quick-xml"]     # 启用 XML 支持
full = ["json", "xml"]      # Meta-feature：启用所有

[dependencies]
serde = "1"
serde_json = { version = "1", optional = true }
quick-xml = { version = "0.31", optional = true }

#![allow(unused)]
fn main() {
// 基于 feature 的条件编译：
#[cfg(feature = "json")]
pub fn to_json<T: serde::Serialize>(value: &T) -> String {
    serde_json::to_string(value).unwrap()
}

#[cfg(feature = "xml")]
pub fn to_xml<T: serde::Serialize>(value: &T) -> String {
    quick_xml::se::to_string(value).unwrap()
}

// 如果未启用必需 feature 则编译错误：
#[cfg(not(any(feature = "json", feature = "xml")))]
compile_error!("At least one format feature (json, xml) must be enabled");
}

最佳实践：

  • 保持 default feature 最小——用户可以选择加入
  • 对可选依赖使用 dep: 语法（Rust 1.60+）以避免创建隐式 feature
  • 在 README 和 crate 级文档中记录 feature

Workspace 组织

对于大项目，使用 Cargo workspace 共享依赖和构建产物：

# 根 Cargo.toml
[workspace]
members = [
    "core",         # 共享类型和 trait
    "parser",       # 解析库
    "server",       # 二进制——主应用
    "client",       # 客户端库
    "cli",          # CLI 二进制
]

# 共享依赖版本：
[workspace.dependencies]
serde = { version = "1", features = ["derive"] }
tokio = { version = "1", features = ["full"] }
tracing = "0.1"

# 在每个成员的 Cargo.toml 中：
# [dependencies]
# serde = { workspace = true }

好处：

  • 单个 Cargo.lock —— 所有 crate 使用相同依赖版本
  • cargo test --workspace 运行所有测试
  • 共享构建缓存——编译一个 crate 有利于所有
  • 组件之间干净的依赖边界

.cargo/config.toml：项目级配置

.cargo/config.toml 文件（在 workspace 根或 $HOME/.cargo/ 中）自定义 Cargo 行为而无需修改 Cargo.toml：

# .cargo/config.toml

# 此 workspace 的默认目标
[build]
target = "x86_64-unknown-linux-gnu"

# 自定义运行器——例如通过 QEMU 运行交叉编译二进制
[target.aarch64-unknown-linux-gnu]
runner = "qemu-aarch64-static"
linker = "aarch64-linux-gnu-gcc"

# Cargo 别名——自定义快捷命令
[alias]
xt = "test --workspace --release"        # cargo xt = release 中运行所有测试
ci = "clippy --workspace -- -D warnings" # cargo ci = 带警告错误 lint
cov = "llvm-cov --workspace"             # cargo cov = 覆盖率（需要 cargo-llvm-cov）

# 构建脚本的环境变量
[env]
IPMI_LIB_PATH = "/usr/lib/bmc"

常见配置模式：

设置目的示例
[build] target默认编译目标x86_64-unknown-linux-musl 用于静态构建
[target.X] runner如何运行二进制"qemu-aarch64-static" 用于交叉编译
[target.X] linker使用哪个链接器"aarch64-linux-gnu-gcc"
[alias]自定义 cargo 子命令xt = "test --workspace"
[env]构建时环境变量库路径、feature 切换
[net] offline防止网络访问true 用于气隙构建

编译时环境变量：env!() 和 option_env!()

Rust 可以在编译时将环境变量嵌入二进制——对版本字符串、构建元数据和配置有用：

#![allow(unused)]
fn main() {
// env!() —— 如果变量缺失则在编译时 panic
const VERSION: &str = env!("CARGO_PKG_VERSION"); // Cargo.toml 中的 "0.1.0"
const PKG_NAME: &str = env!("CARGO_PKG_NAME");   // Cargo.toml 中的 Crate 名称

// option_env!() —— 返回 Option<&str>，如果缺失不 panic
const BUILD_SHA: Option<&str> = option_env!("GIT_SHA");
const BUILD_TIME: Option<&str> = option_env!("BUILD_TIMESTAMP");

fn print_version() {
    println!("{PKG_NAME} v{VERSION}");
    if let Some(sha) = BUILD_SHA {
        println!("  commit: {sha}");
    }
    if let Some(time) = BUILD_TIME {
        println!("  built:  {time}");
    }
}
}

cfg_attr：条件属性

cfg_attr 仅当条件为真时应用属性。这比 #[cfg()] 更有针对性，后者包含/排除整个项：

#![allow(unused)]
fn main() {
// 仅当启用 "serde" feature 时 derive Serialize：
#[cfg_attr(feature = "serde", derive(serde::Serialize, serde::Deserialize))]
#[derive(Debug, Clone)]
pub struct DiagResult {
    pub fc: u32,
    pub passed: bool,
    pub message: String,
}
// 无 "serde" feature：完全不需要 serde 依赖
// 带 "serde" feature：DiagResult 可序列化

// 测试的条件属性：
#[cfg_attr(test, derive(PartialEq))]  // 仅在测试构建中 derive PartialEq
pub struct LargeStruct { /* ... */ }

// 平台特定函数属性：
#[cfg_attr(target_os = "linux", link_name = "ioctl")]
#[cfg_attr(target_os = "freebsd", link_name = "__ioctl")]
extern "C" fn platform_ioctl(fd: i32, request: u64) -> i32;
}
模式它做什么
#[cfg(feature = "x")]包含/排除整个项
#[cfg_attr(feature = "x", derive(Foo))]仅当 feature "x" 开启时添加 derive(Foo)
#[cfg_attr(test, allow(unused))]仅在测试构建中抑制警告
#[cfg_attr(doc, doc = "...")]仅在 cargo doc 中可见的文档

练习：API 重构 ★★（约 30 分钟）

重构以下函数以使用人性化参数和验证类型：

#![allow(unused)]
fn main() {
// 之前：
pub fn connect(host: &str, port: u16, timeout_secs: u32) -> Result<Connection, String> {
    if port == 0 { return Err("port must be non-zero".into()); }
    if timeout_secs == 0 { return Err("timeout must be positive".into()); }
    // ...
}
}

要求：

  • 对 host 使用 impl Into<String> 或 impl AsRef<str>
  • 为 port 创建 Port newtype（带 TryFrom<u16>）
  • 为 timeout 创建 Timeout newtype
  • 返回结构化错误类型
🔑 解决方案 
#![allow(unused)]
fn main() {
use std::convert::TryFrom;
use std::fmt;

#[derive(Debug)]
pub struct Port(u16);

impl TryFrom<u16> for Port {
    type Error = PortError;
    fn try_from(value: u16) -> Result<Self, Self::Error> {
        if value == 0 { Err(PortError::Zero) }
        else { Ok(Port(value)) }
    }
}

#[derive(Debug)]
pub enum PortError { Zero }
impl fmt::Display for PortError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "port must be non-zero")
    }
}
impl std::error::Error for PortError {}

#[derive(Debug)]
pub struct Timeout(std::time::Duration);

impl TryFrom<u32> for Timeout {
    type Error = TimeoutError;
    fn try_from(value: u32) -> Result<Self, Self::Error> {
        if value == 0 { Err(TimeoutError::Zero) }
        else { Ok(Timeout(std::time::Duration::from_secs(value as u64))) }
    }
}

#[derive(Debug)]
pub enum TimeoutError { Zero }
impl fmt::Display for TimeoutError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "timeout must be positive")
    }
}
impl std::error::Error for TimeoutError {}

#[derive(Debug)]
pub enum ConnectError {
    InvalidPort(PortError),
    InvalidTimeout(TimeoutError),
    InvalidHost(String),
}

impl From<PortError> for ConnectError {
    fn from(e: PortError) -> Self { ConnectError::InvalidPort(e) }
}
impl From<TimeoutError> for ConnectError {
    fn from(e: TimeoutError) -> Self { ConnectError::InvalidTimeout(e) }
}

pub fn connect(
    host: impl Into<String>,

    let _host = host.into();
    let _port = port;
    let _timeout = timeout;
    // ... 实际连接逻辑
    Ok(Connection { })
}
