12. 宏 —— 编写代码的代码 🟡

你将学到：

  • 带模式匹配和重复的声明式宏（macro_rules!）

  • 过程宏：derive、属性和函数式

声明式宏（macro_rules!）

宏在语法上匹配模式并在编译时展开为代码：

#![allow(unused)]
fn main() {
// 创建 HashMap 的简单宏
macro_rules! hashmap {
    // 匹配：用逗号分隔的 key => value 对
    ( $( $key:expr => $value:expr ),* $(,)? ) => {
        {
            let mut map = std::collections::HashMap::new();
            $( map.insert($key, $value); )*
            map
        }
    };
}

let scores = hashmap! {
    "Alice" => 95,
    "Bob" => 87,
    "Carol" => 92,
};
// 展开为：
// let mut map = HashMap::new();
// map.insert("Alice", 95);
// map.insert("Bob", 87);
// map.insert("Carol", 92);
// map
}

宏片段类型：

片段匹配示例
$x:expr任何表达式42、a + b、foo()
$x:ty类型i32、Vec<String>
$x:ident标识符my_var、Config
$x:pat模式Some(x)、_
$x:stmt语句let x = 5;
$x:tt单个令牌树任何（最灵活）
$x:literal字面值42、"hello"、true

重复：$( ... ),* 表示"零个或多个，逗号分隔"

#![allow(unused)]
fn main() {
// 自动生成测试函数
macro_rules! test_cases {
    ( $( $name:ident: $input:expr => $expected:expr ),* $(,)? ) => {
        $(
            #[test]
            fn $name() {
                assert_eq!(process($input), $expected);
            }
        )*
    };
}

test_cases! {
    test_empty: "" => "",
    test_hello: "hello" => "HELLO",
    test_trim: "  spaces  " => "SPACES",
}
// 生成三个单独的 #[test] 函数
}

何时（不）使用宏

使用宏当：

  • 减少 trait/泛型无法处理的样板（可变参数、DRY 测试生成）
  • 创建 DSL（html!、sql!、vec!）
  • 条件代码生成（cfg!、compile_error!）

不使用宏当：

  • 函数或泛型可以工作（宏更难调试、自动补全无帮助）
  • 你需要宏内的类型检查（宏在令牌上操作，不是类型）
  • 模式只使用一两次（不值得抽象成本）
#![allow(unused)]
fn main() {
// ❌ 不必要的宏——函数可以：
macro_rules! double {
    ($x:expr) => { $x * 2 };
}

// ✅ 只用函数：
fn double(x: i32) -> i32 { x * 2 }

// ✅ 好宏使用——可变参数，不能是函数：
macro_rules! println {
    ($($arg:tt)*) => { /* format string + args */ };
}
}

过程宏概述

过程宏是转换令牌流的 Rust 函数。它们需要带 proc-macro = true 的单独 crate：

#![allow(unused)]
fn main() {
// 三种过程宏：

// 1. Derive 宏——#[derive(MyTrait)]
// 从结构体定义生成 trait 实现
#[derive(Debug, Clone, Serialize, Deserialize)]
struct Config {
    name: String,
    port: u16,
}

// 2. 属性宏——#[my_attribute]
// 转换注释项
#[route(GET, "/api/users")]
async fn list_users() -> Json<Vec<User>> { /* ... */ }

// 3. 函数式宏——my_macro!(...)
// 自定义语法
let query = sql!(SELECT * FROM users WHERE id = ?);
}

实践中的 Derive 宏

最常见的 proc 宏类型。这是 #[derive(Debug)] 如何工作的概念：

#![allow(unused)]
fn main() {
// 输入（你的结构体）：
#[derive(Debug)]
struct Point {
    x: f64,
    y: f64,
}

// derive 宏生成：
impl std::fmt::Debug for Point {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        f.debug_struct("Point")
            .field("x", &self.x)
            .field("y", &self.y)
            .finish()
    }
}
}

DeriveCrate生成什么

PartialEq、Eqstd相等比较

实用建议：大量使用 derive 宏——它们消除容易出错的样板。 编写你自己的 proc 宏是高级主题；在构建自定义之前使用现有的 （serde、thiserror、clap）。

宏卫生和 $crate

卫生意味着宏内创建的标识符不与调用者作用域中的标识符冲突。 Rust 的 macro_rules! 是部分卫生的：

macro_rules! make_var {
    () => {
        let x = 42; // 这个 'x' 在宏的作用域内
    };
}

fn main() {
    let x = 10;
    make_var!();   // 创建不同的 'x'（卫生）
    println!("{x}"); // 打印 10，不是 42 —— 宏的 x 不泄漏
}

$crate：在库中编写宏时，使用 $crate 引用你自己的 crate —— 它正确解析，无论用户如何导入你的 crate：

#![allow(unused)]
fn main() {
// 在 my_diagnostics crate 中：

pub fn log_result(msg: &str) {
    println!("[diag] {msg}");
}

#[macro_export]
macro_rules! diag_log {
    ($($arg:tt)*) => {
        // ✅ $crate 总是解析为 my_diagnostics，即使用户
        // 在他们的 Cargo.toml 中重命名 crate
        $crate::log_result(&format!($($arg)*))
    };
}

// ❌ 无 $crate：
// my_diagnostics::log_result(...)  ← 如果用户写则中断：
//   [dependencies]
//   diag = { package = "my_diagnostics", version = "1" }
}

规则：在 #[macro_export] 宏中总是使用 $crate::。从不用 你的 crate 名称直接。

递归宏和 tt Munching

递归宏一次处理一个令牌——称为 tt munching（令牌树 munching）的技术：

// 计算传递给宏的表达式数量
macro_rules! count {
    // 基本情况：无令牌剩余
    () => { 0usize };
    // 递归情况：消耗一个表达式，计算其余
    ($head:expr $(, $tail:expr)* $(,)?) => {
        1usize + count!($($tail),*)
    };
}

fn main() {
    let n = count!("a", "b", "c", "d");
    assert_eq!(n, 4);

    // 也在编译时工作：
    const N: usize = count!(1, 2, 3);
    assert_eq!(N, 3);
}
#![allow(unused)]
fn main() {
// 从表达式列表构建异构元组：
macro_rules! tuple_from {
    // 基本：单个元素
    ($single:expr $(,)?) => { ($single,) };
    // 递归：第一个元素 + 其余
    ($head:expr, $($tail:expr),+ $(,)?) => {
        ($head, tuple_from!($($tail),+))
    };
}

let t = tuple_from!(1, "hello", 3.14, true);
// 展开为：(1, ("hello", (3.14, (true,))))
}

片段说明符细微差别：

片段陷阱
$x:expr贪婪解析——1 + 2 是一个表达式，不是三个令牌
$x:ty贪婪解析——Vec<String> 是一个类型；不能后跟 + 或 <
$x:tt正好匹配一个令牌树——最灵活、最少检查
$x:ident只有普通标识符——不是路径如 std::io
$x:pat在 Rust 2021 中，匹配 `A

何时使用 tt：当你需要将令牌转发给另一个宏而无需解析器约束它们时。 $($args:tt)* 是"接受一切"模式（println!、format!、vec! 使用）。

用 syn 和 quote 编写 Derive 宏

Derive 宏生活在单独的 crate（proc-macro = true）中，使用 syn（解析 Rust）和 quote（生成 Rust）转换令牌流：

# my_derive/Cargo.toml
[lib]
proc-macro = true

[dependencies]
syn = { version = "2", features = ["full"] }
quote = "1"
proc-macro2 = "1"

#![allow(unused)]
fn main() {
// my_derive/src/lib.rs
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput};

/// 生成 `describe()` 方法的 derive 宏
/// 返回结构体名称和字段名称。
#[proc_macro_derive(Describe)]
pub fn derive_describe(input: TokenStream) -> TokenStream {
    let input = parse_macro_input!(input as DeriveInput);
    let name = &input.ident;
    let name_str = name.to_string();

    // 提取字段名称（仅用于带命名字段的结构体）
    let fields = match &input.data {
        syn::Data::Struct(data) => {
            data.fields.iter()
                .filter_map(|f| f.ident.as_ref())
                .map(|id| id.to_string())
                .collect::<Vec<_>>()
        }
        _ => vec![],
    };

    let field_list = fields.join(", ");

    let expanded = quote! {
        impl #name {
            pub fn describe() -> String {
                format!("{} {{ {} }}", #name_str, #field_list)
            }
        }
    };

    TokenStream::from(expanded)
}
}
// 在应用 crate 中：
use my_derive::Describe;

#[derive(Describe)]
struct SensorReading {
    sensor_id: u16,
    value: f64,
    timestamp: u64,
}

fn main() {
    println!("{}", SensorReading::describe());
    // "SensorReading { sensor_id, value, timestamp }"
}

工作流：TokenStream（原始令牌）→ syn::parse（AST）→ 检查/转换 → quote!（生成令牌）→ TokenStream（返回编译器）。

Crate角色关键类型
proc-macro编译器接口TokenStream
syn将 Rust 源解析为 ASTDeriveInput、ItemFn、Type
quote从模板生成 Rust 令牌quote!{}、#variable 插值
proc-macro2syn/quote 和 proc-macro 之间的桥TokenStream、Span

实用提示：在编写自己的之前，从研究简单 derive 宏的源代码开始， 如 thiserror 或 derive_more。cargo expand 命令（通过 cargo-expand） 显示任何宏展开为什么——对调试无价。

要点——宏

  • macro_rules! 用于简单代码生成；proc 宏（syn + quote）用于复杂 derive
  • 可能时优先泛型/traits 而非宏——宏更难调试和维护
  • $crate 确保卫生；tt munching 启用递归模式匹配

另见： 第 2 章 — Traits 了解何时 traits/泛型击败宏。第 13 章 — 测试 了解测试宏生成的代码。

练习：声明式宏——map! ★（约 15 分钟）

编写 map! 宏从键值对创建 HashMap：

let m = map! {
    "host" => "localhost",
    "port" => "8080",
};
assert_eq!(m.get("host"), Some(&"localhost"));

要求：支持尾随逗号和空调用 map!{}。

🔑 解决方案 
macro_rules! map {
    () => { std::collections::HashMap::new() };
    ( $( $key:expr => $val:expr ),+ $(,)? ) => {{
        let mut m = std::collections::HashMap::new();
        $( m.insert($key, $val); )+
        m
    }};
}

fn main() {
    let config = map! {
        "host" => "localhost",
        "port" => "8080",
        "timeout" => "30",
    };
    assert_eq!(config.len(), 3);
    assert_eq!(config["host"], "localhost");

    let empty: std::collections::HashMap<String, String> = map!();
    assert!(empty.is_empty());

    let scores = map! { 1 => 100, 2 => 200 };
    assert_eq!(scores[&1], 100);
}
