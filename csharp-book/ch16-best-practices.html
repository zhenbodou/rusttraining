Rust for C# Programmers

C# 开发者的最佳实践

你将学到： 五个关键思维转变（GC→所有权、异常→Results、继承→组合）， 惯用项目组织、错误处理策略、测试模式，以及 C# 开发者在 Rust 中犯的最常见错误。

难度： 🟡 中级

1. 思维转变

  • 从 GC 到所有权：思考谁拥有数据以及何时释放
  • 从异常到 Results：使错误处理显式且可见
  • 从继承到组合：使用 traits 组合行为
  • 从 Null 到 Option：在类型系统中使值缺失显式

3. 错误处理策略

#![allow(unused)]
fn main() {
// 为你的应用程序创建通用 Result 类型
pub type AppResult<T> = Result<T, AppError>;

#[derive(Error, Debug)]
pub enum AppError {
    #[error("Database error: {0}")]
    Database(#[from] sqlx::Error),

    #[error("HTTP error: {0}")]
    Http(#[from] reqwest::Error),

    #[error("Validation error: {message}")]
    Validation { message: String },

    #[error("Business logic error: {message}")]
    Business { message: String },
}

// 在整个应用程序中使用
pub async fn create_user(data: CreateUserRequest) -> AppResult<User> {
    validate_user_data(&data)?;  // 返回 AppError::Validation
    let user = repository.create_user(data).await?;  // 返回 AppError::Database
    Ok(user)
}
}

4. 测试模式

#![allow(unused)]
fn main() {
// 像 C# 单元测试一样构建测试
#[cfg(test)]
mod tests {
    use super::*;
    use rstest::*;  // 用于参数化测试，类似 C# [Theory]

    #[test]
    fn test_basic_functionality() {
        // Arrange
        let input = "test data";

        // Act
        let result = process_data(input);

        // Assert
        assert_eq!(result, "expected output");
    }

    #[rstest]
    #[case(1, 2, 3)]
    #[case(5, 5, 10)]
    #[case(0, 0, 0)]
    fn test_addition(#[case] a: i32, #[case] b: i32, #[case] expected: i32) {
        assert_eq!(add(a, b), expected);
    }

    #[tokio::test]  // 用于异步测试
    async fn test_async_functionality() {
        let result = async_function().await;
        assert!(result.is_ok());
    }
}
}

5. 要避免的常见错误

#![allow(unused)]
fn main() {
// [错误] 不要尝试实现继承
// 而不是：
// struct Manager : Employee  // 这在 Rust 中不存在

// [好] 使用带 traits 的组合
trait Employee {
    fn get_salary(&self) -> u32;
}

trait Manager: Employee {
    fn get_team_size(&self) -> usize;
}

// [错误] 不要到处使用 unwrap()（类似忽略异常）
let value = might_fail().unwrap();  // 可能 panic！

// [好] 正确处理错误
let value = match might_fail() {
    Ok(v) => v,
    Err(e) => {
        log::error!("Operation failed: {}", e);
        return Err(e.into());
    }
};

// [错误] 不要克隆一切（像不必要地复制对象）
let data = expensive_data.clone();  // 昂贵！

// [好] 可能时使用借用
let data = &expensive_data;  // 只是引用

// [错误] 不要到处使用 RefCell（像使一切可变）
struct Data {
    value: RefCell<i32>,  // 内部可变性——谨慎使用
}

// [好] 更喜欢 owned 或 borrowed 数据
struct Data {
    value: i32,  // 简单清晰
}
}

本指南为 C# 开发者提供了全面了解他们现有知识如何转化为 Rust，突出相似性和方法上的根本差异。关键是理解 Rust 的约束（如所有权）旨在防止 C# 中可能的整类错误，以初始复杂性为代价。

6. 避免过度 clone() 🟡

C# 开发者本能地克隆数据，因为 GC 处理成本。在 Rust 中，每个 .clone() 是显式分配。大多数可以用借用消除。

#![allow(unused)]
fn main() {
// [错误] C# 习惯：克隆字符串传递
fn greet(name: String) {
    println!("Hello, {name}");
}

let user_name = String::from("Alice");
greet(user_name.clone());  // 不必要的分配
greet(user_name.clone());  // 再次

// [好] 改用借用——零分配
fn greet(name: &str) {
    println!("Hello, {name}");
}

let user_name = String::from("Alice");
greet(&user_name);  // 借用
greet(&user_name);  // 再次借用——无成本
}

何时 clone 合适：

  • 将数据移动到线程或 'static 闭包（Arc::clone 便宜——它增加计数器）
  • 缓存：你真正需要独立副本
  • 原型设计：先让它工作，然后稍后移除 clone

决策检查清单：

  1. 你能传递 &T 或 &str 代替吗？→ 这样做
  2. 被调用者需要所有权吗？→ 通过移动传递，不是 clone
  3. 它跨线程共享吗？→ 使用 Arc<T>（clone 只是引用计数增加）
  4. 以上都不是？→ clone() 是合理的

7. 避免在生产代码中使用 unwrap() 🟡

忽略异常的 C# 开发者在 Rust 中到处写 .unwrap()。两者同样危险。

#![allow(unused)]
fn main() {
// [错误] "我稍后修复这个"陷阱
let config = std::fs::read_to_string("config.toml").unwrap();
let port: u16 = config_value.parse().unwrap();
let conn = db_pool.get().await.unwrap();

// [好] 用 ? 传播应用程序代码
let config = std::fs::read_to_string("config.toml")?;
let port: u16 = config_value.parse()?;
let conn = db_pool.get().await?;

// [好] 仅在失败确实是 bug 时使用 expect()
let home = std::env::var("HOME")
    .expect("HOME environment variable must be set");  // 记录不变量
}

8. 与借用检查器斗争（以及如何停止） 🟡

每个 C# 开发者都遇到借用检查器拒绝看似有效代码的阶段。修复通常是结构变化，不是变通方法。

#![allow(unused)]
fn main() {
// [错误] 尝试迭代时变异（C# foreach + 修改模式）
let mut items = vec![1, 2, 3, 4, 5];
for item in &items {
    if *item > 3 {
        items.push(*item * 2);  // 错误：无法可变借用 items
    }
}

// [好] 先收集，然后变异
let extras: Vec<i32> = items.iter()
    .filter(|&&x| x > 3)
    .map(|&x| x * 2)
    .collect();
items.extend(extras);
}
#![allow(unused)]
fn main() {
// [错误] 返回对局部的引用（C# 通过 GC 自由返回引用）
fn get_greeting() -> &str {
    let s = String::from("hello");
    &s  // 错误：s 在函数结束时被丢弃
}

// [好] 返回 owned 数据
fn get_greeting() -> String {
    String::from("hello")  // 调用者拥有它
}
}

解决借用检查器冲突的常见模式：

C# 习惯Rust 解决方案
在结构体中存储引用使用 owned 数据，或添加生命周期参数
自由变异共享状态使用 Arc<Mutex<T>> 或重构以避免共享
返回对局部的引用返回 owned 值
迭代时修改集合收集更改，然后应用
多个可变引用将结构体拆分为独立部分

9. 折叠赋值金字塔 🟢

C# 开发者编写 if (x != null) { if (x.Value > 0) { ... } } 链。Rust 的 match、if let 和 ? 扁平化这些。

#![allow(unused)]
fn main() {
// [错误] 来自 C# 的嵌套空检查风格
fn process(input: Option<String>) -> Option<usize> {
    match input {
        Some(s) => {
            if !s.is_empty() {
                match s.parse::<usize>() {
                    Ok(n) => {
                        if n > 0 {
                            Some(n * 2)
                        } else {
                            None
                        }
                    }
                    Err(_) => None,
                }
            } else {
                None
            }
        }
        None => None,
    }
}

// [好] 用组合器扁平化
fn process(input: Option<String>) -> Option<usize> {
    input
        .filter(|s| !s.is_empty())
        .and_then(|s| s.parse::<usize>().ok())
        .filter(|&n| n > 0)
        .map(|n| n * 2)
}
}
