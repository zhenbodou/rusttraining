Rust for C# Programmers

异步编程：C# Task vs Rust Future

你将学到： Rust 的惰性 Future vs C# 的急切 Task，执行器模型（tokio）， 通过 Drop + select! vs CancellationToken 的取消，以及并发请求的现实世界模式。

难度： 🔴 高级

C# 开发者非常熟悉 async/await。Rust 使用相同的关键字但有根本不同的执行模型。

执行器模型

// C# —— 运行时提供内置线程池和任务调度器
// async/await "开箱即用"
public async Task<string> FetchDataAsync(string url)
{
    using var client = new HttpClient();
    return await client.GetStringAsync(url);  // 由 .NET 线程池调度
}
// .NET 管理线程池、任务调度和同步上下文

// Rust —— 无内置异步运行时。你选择执行器。
// 最流行的是 tokio。
async fn fetch_data(url: &str) -> Result<String, reqwest::Error> {
    let body = reqwest::get(url).await?.text().await?;
    Ok(body)
}

// 你必须有运行时来执行异步代码：
#[tokio::main]  // 这个宏设置 tokio 运行时
async fn main() {
    let data = fetch_data("https://example.com").await.unwrap();
    println!("{}", &data[..100]);
}

#![allow(unused)]
fn main() {
// 重要：Future 在 Rust 中是惰性的！
async fn compute() -> i32 { println!("Computing!"); 42 }

let future = compute();  // 无打印！Future 尚未轮询。
let result = future.await; // 现在"Computing!"被打印
}
// C# Task 立即开始！
var task = ComputeAsync();  // "Computing!"立即打印
var result = await task;    // 只等待完成

取消：CancellationToken vs Drop / select!

// C# —— 带 CancellationToken 的协作取消
public async Task ProcessAsync(CancellationToken ct)
{
    while (!ct.IsCancellationRequested)
    {
        await Task.Delay(1000, ct);  // 如果取消则抛出
        DoWork();
    }
}

var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));
await ProcessAsync(cts.Token);

#![allow(unused)]
fn main() {
// Rust —— 通过 drop future 取消，或使用 tokio::select!
use tokio::time::{sleep, Duration};

async fn process() {
    loop {
        sleep(Duration::from_secs(1)).await;
        do_work();
    }
}

// 带 select! 的超时模式
async fn run_with_timeout() {
    tokio::select! {
        _ = process() => { println!("Completed"); }
        _ = sleep(Duration::from_secs(5)) => { println!("Timed out!"); }
    }
    // 当 select! 选择超时分支时，process() future 被 DROP
    // —— 自动清理，无需 CancellationToken
}
}

现实世界模式：带超时的并发请求

// C# —— 带超时的并发 HTTP 请求
public async Task<string[]> FetchAllAsync(string[] urls, CancellationToken ct)
{
    var tasks = urls.Select(url => httpClient.GetStringAsync(url, ct));
    return await Task.WhenAll(tasks);
}

#![allow(unused)]
fn main() {
// Rust —— 带 tokio::join! 或 futures::join_all 的并发请求
use futures::future::join_all;

async fn fetch_all(urls: &[&str]) -> Vec<Result<String, reqwest::Error>> {
    let futures = urls.iter().map(|url| reqwest::get(*url));
    let responses = join_all(futures).await;

    let mut results = Vec::new();
    for resp in responses {
        results.push(resp?.text().await);
    }
    results
}

// 带超时：
async fn fetch_all_with_timeout(urls: &[&str]) -> Result<Vec<String>, &'static str> {
    tokio::time::timeout(
        Duration::from_secs(10),
        async {
            let futures: Vec<_> = urls.iter()
                .map(|url| async { reqwest::get(*url).await?.text().await })
                .collect();
            let results = join_all(futures).await;
            results.into_iter().collect::<Result<Vec<_>, _>>()
        }
    )
    .await
    .map_err(|_| "Request timed out")?
    .map_err(|_| "Request failed")
}
}

挑战：编写一个异步函数，从两个 URL 并发获取，返回先响应的那个，并取消另一个。（这在 C# 中是 Task.WhenAny。）

🔑 解决方案 
use tokio::time::{sleep, Duration};

// 模拟异步获取
async fn fetch(url: &str, delay_ms: u64) -> String {
    sleep(Duration::from_millis(delay_ms)).await;
    format!("Response from {url}")
}

async fn fetch_first(url1: &str, url2: &str) -> String {
    tokio::select! {
        result = fetch(url1, 200) => {
            println!("URL 1 won");
            result
        }
        result = fetch(url2, 500) => {
            println!("URL 2 won");
            result
        }
    }
    // 失败分支的 future 自动被 drop（取消）
}

#[tokio::main]
async fn main() {
    let result = fetch_first("https://fast.api", "https://slow.api").await;
    println!("{result}");
}

关键要点：tokio::select! 是 Rust 中等价于 Task.WhenAny——它竞逐多个 futures，当第一个完成时完成，并 drop（取消）其余的。

使用 tokio::spawn 生成独立任务

在 C# 中，Task.Run 启动独立于调用者运行的工作。Rust 的等价物是 tokio::spawn：

#![allow(unused)]
fn main() {
use tokio::task;

async fn background_work() {
    // 独立运行——即使调用者的 future 被 drop
    let handle = task::spawn(async {
        tokio::time::sleep(Duration::from_secs(2)).await;
        42
    });

    // 当生成的任务运行时做其他工作...
    println!("Doing other work");

    // 当你需要时等待结果
    let result = handle.await.unwrap(); // 42
}
}
// C# 等价物
var task = Task.Run(async () => {
    await Task.Delay(2000);
    return 42;
});
// 做其他工作...
var result = await task;

关键区别：常规 async {} 块是惰性的——在等待之前不做任何事。tokio::spawn 立即在运行时启动它，像 C# 的 Task.Run。

Pin：为什么 Rust Async 有 C# 没有的概念

C# 开发者从不遇到 Pin——CLR 的垃圾回收器自由移动对象并自动更新所有引用。Rust 没有 GC。当编译器将 async fn 转换为状态机时，该结构体可能包含指向其自己字段的内部指针。移动结构体会使这些指针无效。

Pin<T> 是一个包装器，说："这个值在内存中不会移动。"

#![allow(unused)]
fn main() {
// 你将在这些上下文中看到 Pin：
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
    //           ^^^^^^^^^^^^^^ 固定——内部引用保持有效
}

// 从 trait 返回装箱的 future：
fn make_future() -> Pin<Box<dyn Future<Output = i32> + Send>> {
    Box::pin(async { 42 })
}
}

实际上，你几乎从不自己写 Pin。 async fn 和 .await 语法处理它。你只会在以下情况遇到它：

  • 编译器错误消息（遵循建议）
  • tokio::select!（使用 pin!() 宏）
  • 返回 dyn Future 的 trait 方法（使用 Box::pin(async { ... })）
