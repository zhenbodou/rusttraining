Rust for C# Programmers

智能指针：当单一所有权不够时

你将学到： Box<T>、Rc<T>、Arc<T>、Cell<T>、RefCell<T> 和 Cow<'a, T>—— 何时使用每个，它们如何与 C# 的 GC 管理引用比较，Drop 作为 Rust 的 IDisposable， Deref 强制转换，以及选择正确智能指针的决策树。

难度： 🔴 高级

在 C# 中，每个对象本质上由 GC 引用计数。在 Rust 中，单一所有权是默认——但有时你需要共享所有权、堆分配或内部可变性。这就是智能指针的用武之地。

Box<T> —— 简单堆分配

#![allow(unused)]
fn main() {
// 栈分配（Rust 中的默认）
let x = 42;           // 在栈上

// 使用 Box 堆分配
let y = Box::new(42); // 在堆上，类似 C# `new int(42)`（装箱）
println!("{}", y);     // 自动解引用：打印 42

// 常见用途：递归类型（编译时无法知道大小）
#[derive(Debug)]
enum List {
    Cons(i32, Box<List>),  // Box 给出已知的指针大小
    Nil,
}

let list = List::Cons(1, Box::new(List::Cons(2, Box::new(List::Nil))));
}
// C# —— 一切已经在堆上（引用类型）
// Box<T> 仅在 Rust 中需要，因为栈是默认
var list = new LinkedListNode<int>(1);  // 总是堆分配

Rc<T> —— 共享所有权（单线程）

#![allow(unused)]
fn main() {
use std::rc::Rc;

// 数据的多个所有者——类似多个 C# 引用
let shared = Rc::new(vec![1, 2, 3]);
let clone1 = Rc::clone(&shared); // 引用计数：2
let clone2 = Rc::clone(&shared); // 引用计数：3

println!("Count: {}", Rc::strong_count(&shared)); // 3
// 数据在最后一个 Rc 超出作用域时被丢弃

// 常见用途：共享配置、图节点、树结构
}

Arc<T> —— 共享所有权（线程安全）

#![allow(unused)]
fn main() {
use std::sync::Arc;
use std::thread;

// Arc = 原子引用计数——安全地跨线程共享
let data = Arc::new(vec![1, 2, 3]);

let handles: Vec<_> = (0..3).map(|i| {
    let data = Arc::clone(&data);
    thread::spawn(move || {
        println!("Thread {i}: {:?}", data);
    })
}).collect();

for h in handles { h.join().unwrap(); }
}
// C# —— 所有引用默认线程安全（GC 处理它）
var data = new List<int> { 1, 2, 3 };
// 可以自由跨线程共享（但变异仍然不安全！）

Cell<T> 和 RefCell<T> —— 内部可变性

#![allow(unused)]
fn main() {
use std::cell::RefCell;

// 有时你需要通过共享引用变异数据。
// RefCell 将借用检查从编译时移动到运行时。
struct Logger {
    entries: RefCell<Vec<String>>,
}

impl Logger {
    fn new() -> Self {
        Logger { entries: RefCell::new(Vec::new()) }
    }

    fn log(&self, msg: &str) { // &self，不是 &mut self！
        self.entries.borrow_mut().push(msg.to_string());
    }

    fn dump(&self) {
        for entry in self.entries.borrow().iter() {
            println!("{entry}");
        }
    }
}
// ⚠️ RefCell 在违反借用规则时在运行时 panic
// 谨慎使用——可能时更喜欢编译时检查
}

Cow<'a, str> —— 写入时克隆

#![allow(unused)]
fn main() {
use std::borrow::Cow;

// 有时你有一个可能*需要*变成 String 的 &str
fn normalize(input: &str) -> Cow<'_, str> {
    if input.contains('\t') {
        // 仅在我们需要修改时分配
        Cow::Owned(input.replace('\t', "    "))
    } else {
        // 借用原始——零分配
        Cow::Borrowed(input)
    }
}

let clean = normalize("hello");           // Cow::Borrowed —— 无分配
let dirty = normalize("hello\tworld");    // Cow::Owned —— 分配
// 两者都可以作为 &str 通过 Deref 使用
println!("{clean} / {dirty}");
}

Drop：Rust 的 IDisposable

在 C# 中，IDisposable + using 处理资源清理。Rust 的等价物是 Drop trait——但它是自动的，不是可选的：

// C# —— 必须记得使用 'using' 或调用 Dispose()
using var file = File.OpenRead("data.bin");
// 在 using 块结束时调用 Dispose()

// 忘记 'using' 是资源泄漏！
var file2 = File.OpenRead("data.bin");
// GC *最终* 会终结，但时机不可预测

// Rust —— Drop 在值超出作用域时自动运行
{
    let file = File::open("data.bin")?;
    // 使用 file...
}   // file.drop() 在这里调用，确定性——不需要 'using'

// 自定义 Drop（类似实现 IDisposable）
struct TempFile {
    path: std::path::PathBuf,
}

impl Drop for TempFile {
    fn drop(&mut self) {
        // 当 TempFile 超出作用域时保证运行
        let _ = std::fs::remove_file(&self.path);
        println!("Cleaned up {:?}", self.path);
    }
}

fn main() {
    let tmp = TempFile { path: "scratch.tmp".into() };
    // ... 使用 tmp ...
}   // scratch.tmp 在这里自动删除

与 C# 的关键区别： 在 Rust 中，每个类型都可以有确定性清理。你永远不会忘记 using 因为没什么可忘记的——Drop 在所有者超出作用域时运行。这个模式称为 RAII（资源获取即初始化）。

规则：如果你的类型持有资源（文件句柄、网络连接、锁守卫、临时文件），实现 Drop。所有权系统保证它恰好运行一次。

Deref 强制转换：自动智能指针解包

Rust 自动"解包"智能指针当你调用方法或将它们传递给函数时。这称为 Deref 强制转换：

#![allow(unused)]
fn main() {
let boxed: Box<String> = Box::new(String::from("hello"));

// Deref 强制转换链：Box<String> → String → str
println!("Length: {}", boxed.len());   // 调用 str::len() —— 自动解引用！

fn greet(name: &str) {
    println!("Hello, {name}");
}

let s = String::from("Alice");
greet(&s);       // &String → &str 通过 Deref 强制转换
greet(&boxed);   // &Box<String> → &String → &str —— 两层！
}
// C# 无等价物——你需要显式强制转换或 .ToString()
// 最接近：隐式转换运算符，但那些需要显式定义

为什么这重要： 你可以传递 &String 到期望 &str 的地方，&Vec<T> 到期望 &[T] 的地方，&Box<T> 到期望 &T 的地方——都无需显式转换。这就是为什么 Rust API 通常接受 &str 和 &[T] 而不是 &String 和 &Vec<T>。

Rc vs Arc：何时使用哪个

Rc<T>Arc<T>
线程安全❌ 仅单线程✅ 线程安全（原子操作）
开销更低（非原子引用计数）更高（原子引用计数）
编译器强制执行不会在 thread::spawn 中编译到处工作
与...组合RefCell<T> 用于变异Mutex<T> 或 RwLock<T> 用于变异

经验法则： 从 Rc 开始。编译器会告诉你是否需要 Arc。

🏋️ 练习：选择正确的智能指针（点击展开）

挑战：对于每个场景，选择正确的智能指针并解释为什么。

  1. 递归树数据结构
  2. 多个组件读取的共享配置对象（单线程）
  3. 跨 HTTP 处理线程共享的请求计数器
  4. 可能返回借用或拥有字符串的缓存
  5. 需要通过共享引用变异的日志缓冲区
🔑 解决方案
  1. Box<T> —— 递归类型需要间接以便编译时已知大小
  2. Rc<T> —— 共享只读访问，单线程，不需要 Arc 开销
  3. Arc<Mutex<u64>> —— 跨线程共享（Arc）带变异（Mutex）
  4. Cow<'a, str> —— 有时返回 &str（缓存命中），有时 String（缓存未命中）
  5. RefCell<Vec<String>> —— &self 后面的内部可变性（单线程）

经验法则：从 owned 类型开始。当你需要间接时使用 Box，当你需要共享时使用 Rc/Arc，当你需要内部可变性时使用 RefCell/Mutex，当你想要常见情况的零拷贝时使用 Cow。
