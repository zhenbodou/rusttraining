生命周期：告诉编译器引用活多久

你将学到： 为什么生命周期存在（无 GC 意味着编译器需要证明），生命周期注解语法， 省略规则，结构体生命周期，'static 生命周期，以及带修复的常见借用检查器错误。

难度： 🔴 高级

C# 开发者从不考虑引用生命周期——垃圾回收器处理可达性。在 Rust 中，编译器需要证明每个引用在使用期间都是有效的。生命周期就是那个证明。

为什么生命周期存在

#![allow(unused)]
fn main() {
// 这不会编译——编译器无法证明返回的引用有效
fn longest(a: &str, b: &str) -> &str {
    if a.len() > b.len() { a } else { b }
}
// 错误：缺少生命周期注解——编译器不知道
// 返回值是从 `a` 还是 `b` 借用
}

生命周期注解

// 生命周期 'a 说："返回值至少与两个输入活得一样长"
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}

fn main() {
    let result;
    let string1 = String::from("long string");
    {
        let string2 = String::from("xyz");
        result = longest(&string1, &string2);
        println!("Longest: {result}"); // ✅ 两个引用在这里仍然有效
    }
    // println!("{result}"); // ❌ 错误：string2 活得不够长
}

C# 比较

// C# —— GC 保持对象存活，只要任何引用存在
string Longest(string a, string b) => a.Length > b.Length ? a : b;

// 无生命周期问题——GC 跟踪可达性自动
// 但：GC 暂停，不可预测的内存使用，无编译时证明

生命周期省略规则

大多数时候你不需要写生命周期注解。编译器自动应用三条规则：

规则描述示例
规则 1每个引用参数获得自己的生命周期fn foo(x: &str, y: &str) → fn foo<'a, 'b>(x: &'a str, y: &'b str)
规则 2如果只有一个输入生命周期，它被分配给所有输出生命周期fn first(s: &str) -> &str → fn first<'a>(s: &'a str) -> &'a str
规则 3如果一个输入是 &self 或 &mut self，该生命周期被分配给所有输出fn name(&self) -> &str → 工作因为有 &self
#![allow(unused)]
fn main() {
// 这些是等价的——编译器自动添加生命周期：
fn first_word(s: &str) -> &str { /* ... */ }           // 省略
fn first_word<'a>(s: &'a str) -> &'a str { /* ... */ } // 显式

// 但这需要显式注解——两个输入，输出从哪个借用？
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str { /* ... */ }
}

结构体生命周期

// 借用数据（而不是拥有它）的结构体
struct Excerpt<'a> {
    text: &'a str,  // 从某个必须比这个结构体活得长的 String 借用
}

impl<'a> Excerpt<'a> {
    fn new(text: &'a str) -> Self {
        Excerpt { text }
    }

    fn first_sentence(&self) -> &str {
        self.text.split('.').next().unwrap_or(self.text)
    }
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let excerpt = Excerpt::new(&novel); // excerpt 从 novel 借用
    println!("First sentence: {}", excerpt.first_sentence());
    // novel 必须在 excerpt 存在时保持存活
}
// C# 等价物——无生命周期问题，但也无编译时保证
class Excerpt
{
    public string Text { get; }
    public Excerpt(string text) => Text = text;
    public string FirstSentence() => Text.Split('.')[0];
}
// 如果字符串在其他地方被变异会怎样？运行时意外。

'static 生命周期

#![allow(unused)]
fn main() {
// 'static 意味着"活在整个程序持续期间"
let s: &'static str = "I'm a string literal"; // 存储在二进制中，总是有效

// 你看到 'static 的常见地方：
// 1. 字符串字面量
// 2. 全局常量
// 3. Thread::spawn 需要 'static（线程可能活得比调用者长）
std::thread::spawn(move || {
    // 发送到线程的闭包必须拥有它们的数据或使用 'static 引用
    println!("{s}"); // 好：&'static str
});

// 'static 不意味着"不朽"——它意味着"如果需要可以永远活"
let owned = String::from("hello");
// owned 不是 'static，但它可以被移动到线程中（所有权转移）
}

常见借用检查器错误和修复

错误原因修复
missing lifetime specifier多个输入引用，输出模糊添加 <'a> 注解将输出绑定到正确的输入
does not live long enough引用活得比它指向的数据长扩展数据的作用域，或返回_owned_数据代替
cannot borrow as mutable不可变借用仍然活动在变异之前使用不可变引用，或重构
cannot move out of borrowed content尝试获取借用数据的所有权使用 .clone()，或重构以避免移动
lifetime may not live long enough结构体借用活得比源长确保源数据的作用域包含结构体的使用

多个生命周期参数

有时引用来自不同源，有不同的生命周期：

// 两个独立的生命周期：返回值只从 'a 借用，不从 'b
fn first_with_context<'a, 'b>(data: &'a str, _context: &'b str) -> &'a str {
    // 返回值从 'data' 借用——'context' 可以有更短的生命周期
    data.split(',').next().unwrap_or(data)
}

fn main() {
    let data = String::from("alice,bob,charlie");
    let result;
    {
        let context = String::from("user lookup"); // 更短的生命周期
        result = first_with_context(&data, &context);
    } // context 丢弃——但结果从 data 借用，不从 context ✅
    println!("{result}");
}
// C# —— 无生命周期跟踪意味着你不能表达"从 A 借用但不从 B"
string FirstWithContext(string data, string context) => data.Split(',')[0];
// 对 GC 语言好，但 Rust 可以无需 GC 证明安全性

现实世界生命周期模式

模式 1：返回引用的迭代器

#![allow(unused)]
fn main() {
// 从输入产生借用切片的解析器
struct CsvRow<'a> {
    fields: Vec<&'a str>,
}

fn parse_csv_line(line: &str) -> CsvRow<'_> {
    // '_ 告诉编译器"从输入推断生命周期"
    CsvRow {
        fields: line.split(',').collect(),
    }
}
}

模式 2："有疑问时返回_owned_"

#![allow(unused)]
fn main() {
// 当生命周期变得复杂时，返回_owned_数据是实用的修复
fn format_greeting(first: &str, last: &str) -> String {
    // 返回_owned_ String——不需要生命周期注解
    format!("Hello, {first} {last}!")
}

// 仅当以下情况时借用：
// 1. 性能重要（避免分配）
// 2. 输入和输出生命周期之间的关系清晰
}

模式 3：泛型上的生命周期边界

#![allow(unused)]
fn main() {
// "T 必须活得至少与 'a 一样长"
fn store_reference<'a, T: 'a>(cache: &mut Vec<&'a T>, item: &'a T) {
    cache.push(item);
}

// 在 trait 对象中常见：Box<dyn Display + 'a>
fn make_printer<'a>(text: &'a str) -> Box<dyn std::fmt::Display + 'a> {
    Box::new(text)
}
}

何时使用 'static

场景使用 'static？替代方案
字符串字面量✅ 是——它们总是 'static—
thread::spawn 闭包通常——线程活得比调用者长对借用数据使用 thread::scope
全局配置✅ lazy_static! 或 OnceLock通过参数传递引用
长期存储的 trait 对象通常——Box<dyn Trait + 'static>用 'a 参数化容器
临时借用❌ 从不——过度约束使用实际生命周期
🏋️ 练习：生命周期注解（点击展开）

挑战：添加正确的生命周期注解使这个编译：

#![allow(unused)]
fn main() {
struct Config {
    db_url: String,
    api_key: String,
}

// TODO: 添加生命周期注解
fn get_connection_info(config: &Config) -> (&str, &str) {
    (&config.db_url, &config.api_key)
}

// TODO: 这个结构体从 Config 借用——添加生命周期参数
struct ConnectionInfo {
    db_url: &str,
    api_key: &str,
}
}
🔑 解决方案 
#![allow(unused)]
fn main() {
struct Config {
    db_url: String,
    api_key: String,
}

// 规则 3 不适用（无 &self），规则 2 适用（一个输入 → 输出）
// 所以编译器自动处理这个——不需要注解！
fn get_connection_info(config: &Config) -> (&str, &str) {
    (&config.db_url, &config.api_key)
}

// 结构体生命周期注解需要：
struct ConnectionInfo<'a> {
    db_url: &'a str,
    api_key: &'a str,
}

fn make_info<'a>(config: &'a Config) -> ConnectionInfo<'a> {
    ConnectionInfo {
        db_url: &config.db_url,
        api_key: &config.api_key,
    }
}
}

关键要点：生命周期省略通常 saves 你不必在函数上写注解，但借用数据的结构体总是需要显式 <'a>。
