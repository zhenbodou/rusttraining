Rust for C# Programmers

Rust 中的测试 vs C#

你将学到： 内置 #[test] vs xUnit，带 rstest 的参数化测试（类似 [Theory]）， 带 proptest 的属性测试，带 mockall 的 mocking，以及异步测试模式。

难度： 🟡 中级

单元测试

// C# —— xUnit
using Xunit;

public class CalculatorTests
{
    [Fact]
    public void Add_ReturnsSum()
    {
        var calc = new Calculator();
        Assert.Equal(5, calc.Add(2, 3));
    }

    [Theory]
    [InlineData(1, 2, 3)]
    [InlineData(0, 0, 0)]
    [InlineData(-1, 1, 0)]
    public void Add_Theory(int a, int b, int expected)
    {
        Assert.Equal(expected, new Calculator().Add(a, b));
    }
}

#![allow(unused)]
fn main() {
// Rust —— 内置测试，无需外部框架
pub fn add(a: i32, b: i32) -> i32 { a + b }

#[cfg(test)]  // 仅在 `cargo test` 期间编译
mod tests {
    use super::*;  // 从父模块导入

    #[test]
    fn add_returns_sum() {
        assert_eq!(add(2, 3), 5);
    }

    #[test]
    fn add_negative_numbers() {
        assert_eq!(add(-1, 1), 0);
    }

    #[test]
    #[should_panic(expected = "overflow")]
    fn add_overflow_panics() {
        let _ = add(i32::MAX, 1); // 在 debug 模式中 panic
    }
}
}

参数化测试（类似 [Theory]）

#![allow(unused)]
fn main() {
// 使用 `rstest` crate 进行参数化测试
use rstest::rstest;

#[rstest]
#[case(1, 2, 3)]
#[case(0, 0, 0)]
#[case(-1, 1, 0)]
fn test_add(#[case] a: i32, #[case] b: i32, #[case] expected: i32) {
    assert_eq!(add(a, b), expected);
}

// Fixtures —— 类似测试设置方法
#[rstest]
fn test_with_fixture(#[values(1, 2, 3)] x: i32) {
    assert!(x > 0);
}
}

断言比较

Assert.True(condition)assert!(condition)

Assert.Null(obj)assert!(option.is_none())无 null——使用 Option

异步测试

// C# —— 带 xUnit 的异步测试
[Fact]
public async Task GetUser_ReturnsUser()
{
    var service = new UserService();
    let user = await service.GetUserAsync(1);
    Assert.Equal("Alice", user.Name);
}

#![allow(unused)]
fn main() {
// Rust —— 带 tokio 的异步测试
#[tokio::test]
async fn get_user_returns_user() {
    let service = UserService::new();
    let user = service.get_user(1).await.unwrap();
    assert_eq!(user.name, "Alice");
}
}

带 mockall 的 Mocking

#![allow(unused)]
fn main() {
use mockall::automock;

#[automock]                         // 生成 MockUserRepo 结构体
trait UserRepo {
    fn find_by_id(&self, id: u32) -> Option<User>;
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn service_returns_user_from_repo() {
        let mut mock = MockUserRepo::new();
        mock.expect_find_by_id()
            .with(mockall::predicate::eq(1))
            .returning(|_| Some(User { name: "Alice".into() }));

        let service = UserService::new(mock);
        let user = service.get_user(1).unwrap();
        assert_eq!(user.name, "Alice");
    }
}
}
// C# —— Moq 等价物
var mock = new Mock<IUserRepo>();
mock.Setup(r => r.FindById(1)).Returns(new User { Name = "Alice" });
var service = new UserService(mock.Object);
Assert.Equal("Alice", service.GetUser(1).Name);
🏋️ 练习：编写全面的测试（点击展开）

挑战：给定这个函数，编写涵盖以下内容的测试：快乐路径、空输入、数字字符串和 Unicode。

#![allow(unused)]
fn main() {
pub fn title_case(input: &str) -> String {
    input.split_whitespace()
        .map(|word| {
            let mut chars = word.chars();
            match chars.next() {
                Some(c) => format!("{}{}", c.to_uppercase(), chars.as_str().to_lowercase()),
                None => String::new(),
            }
        })
        .collect::<Vec<_>>()
        .join(" ")
}
}
🔑 解决方案 
#![allow(unused)]
fn main() {
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn happy_path() {
        assert_eq!(title_case("hello world"), "Hello World");
    }

    #[test]
    fn empty_input() {
        assert_eq!(title_case(""), "");
    }

    #[test]
    fn single_word() {
        assert_eq!(title_case("rust"), "Rust");
    }

    #[test]
    fn already_title_case() {
        assert_eq!(title_case("Hello World"), "Hello World");
    }

    #[test]
    fn all_caps() {
        assert_eq!(title_case("HELLO WORLD"), "Hello World");
    }

    #[test]
    fn extra_whitespace() {
        // split_whitespace 处理多个空格
        assert_eq!(title_case("  hello   world  "), "Hello World");
    }

    #[test]
    fn unicode() {
        assert_eq!(title_case("café résumé"), "Café Résumé");
    }

    #[test]
    fn numeric_words() {
        assert_eq!(title_case("hello 42 world"), "Hello 42 World");
    }
}
}

关键要点：Rust 的内置测试框架处理大多数单元测试需求。对参数化测试使用 rstest，对 mocking 使用 mockall——无需像 xUnit 这样的大型测试框架。

带 proptest 的属性测试

熟悉 FsCheck 的 C# 开发者将认识基于属性的测试：你不是编写单独的测试用例，而是描述必须对所有可能输入成立的属性，框架生成数千个随机输入尝试破坏它们。

为什么属性测试重要

// C# —— 手写单元测试检查特定情况
[Fact]
public void Reverse_Twice_Returns_Original()
{
    var list = new List<int> { 1, 2, 3 };
    list.Reverse();
    list.Reverse();
    Assert.Equal(new[] { 1, 2, 3 }, list);
}
// 但空列表呢？单个元素？10,000 个元素？负数呢？
// 你需要数十个手写用例。

#![allow(unused)]
fn main() {
// Rust —— proptest 自动生成数千个输入
use proptest::prelude::*;

fn reverse<T: Clone>(v: &[T]) -> Vec<T> {
    v.iter().rev().cloned().collect()
}

proptest! {
    #[test]
    fn reverse_twice_is_identity(ref v in prop::collection::vec(any::<i32>(), 0..1000)) {
        let reversed_twice = reverse(&reverse(v));
        prop_assert_eq!(v, &reversed_twice);
    }
    // proptest 用数百个随机 Vec<i32> 值运行这个：
    // [], [0], [i32::MIN, i32::MAX], [42; 999], 随机序列...
    // 如果失败，它收缩到最小的失败输入！
}
}

C# 开发者的常见模式

#![allow(unused)]
fn main() {
use proptest::prelude::*;

// 1. 往返属性：serialize → deserialize = 恒等
// （类似测试 JsonSerializer.Serialize → Deserialize）
proptest! {
    #[test]
    fn json_roundtrip(name in "[a-zA-Z]{1,50}", age in 0u32..150) {
        let user = User { name: name.clone(), age };
        let json = serde_json::to_string(&user).unwrap();
        let parsed: User = serde_json::from_str(&json).unwrap();
        prop_assert_eq!(user, parsed);
    }
}

// 2. 不变量属性：输出总是满足条件
proptest! {
    #[test]
    fn sort_output_is_sorted(ref v in prop::collection::vec(any::<i32>(), 0..500)) {
        let mut sorted = v.clone();
        sorted.sort();
        // 每个相邻对必须有序
        for window in sorted.windows(2) {
            prop_assert!(window[0] <= window[1]);
        }
    }
}

// 3. 预言机属性：比较两个实现
proptest! {
    #[test]
    fn fast_path_matches_slow_path(input in "[0-9a-f]{1,100}") {
        let result_fast = parse_hex_fast(&input);
        let result_slow = parse_hex_slow(&input);
        prop_assert_eq!(result_fast, result_slow);
    }
}

// 4. 自定义策略：生成域特定测试数据
fn valid_email() -> impl Strategy<Value = String> {
    ("[a-z]{1,20}", "[a-z]{1,10}", prop::sample::select(vec!["com", "org", "io"]))
        .prop_map(|(user, domain, tld)| format!("{}@{}.{}", user, domain, tld))
}

proptest! {
    #[test]
    fn email_parsing_accepts_valid_emails(email in valid_email()) {
        let result = Email::new(&email);
        prop_assert!(result.is_ok(), "Failed to parse: {}", email);
    }
}
}

proptest vs FsCheck 比较

特性C# FsCheckRust proptest
随机输入生成Arb.Generate<T>()any::<T>()
自定义生成器Arb.Register<T>()impl Strategy<Value = T>
失败时收缩自动自动
字符串模式手动"[regex]" 策略
集合生成Gen.ListOfprop::collection::vec(strategy, range)
组合生成器Gen.Select.prop_map()、.prop_flat_map()
配置（用例数）Config.MaxTest#![proptest_config(ProptestConfig::with_cases(10000))] 在 proptest! 块内

何时使用属性测试 vs 单元测试

使用单元测试当使用proptest当
测试特定边界情况验证所有输入的不变量
测试错误消息/代码往返属性（parse ↔ format）
集成/mock 测试比较两个实现
行为取决于精确值"对所有 X，属性 P 成立"

集成测试：tests/ 目录

单元测试与 #[cfg(test)] 一起生活在 src/ 中。集成测试生活在单独的 tests/ 目录中，测试 crate 的公共 API——就像 C# 集成测试将项目作为外部程序集引用一样。

my_crate/
├── src/
│   ├── lib.rs          // 公共 API
│   └── internal.rs     // 私有实现
├── tests/
│   ├── smoke.rs        // 每个文件是单独的测试二进制
│   ├── api_tests.rs
│   └── common/
│       └── mod.rs      // 共享测试助手
└── Cargo.toml

编写集成测试

tests/ 中的每个文件编译为依赖于你的库的单独 crate：

#![allow(unused)]
fn main() {
// tests/smoke.rs —— 只能从 my_crate 访问 pub 项
use my_crate::{process_order, Order, OrderResult};

#[test]
fn process_valid_order_returns_confirmation() {
    let order = Order::new("SKU-001", 3);
    let result = process_order(order);
    assert!(matches!(result, OrderResult::Confirmed { .. }));
}
}

共享测试助手

将共享设置代码放在 tests/common/mod.rs 中（不是 tests/common.rs，它将被视为自己的测试文件）：

#![allow(unused)]
fn main() {
// tests/common/mod.rs
use my_crate::Config;

pub fn test_config() -> Config {
    Config::builder()
        .database_url("sqlite::memory:")
        .build()
        .expect("test config must be valid")
}
}
#![allow(unused)]
fn main() {
// tests/api_tests.rs
mod common;

use my_crate::App;

#[test]
fn app_starts_with_test_config() {
    let config = common::test_config();
    let app = App::new(config);
    assert!(app.is_healthy());
}
}

运行特定测试类型

cargo test                  # 运行所有测试（单元 + 集成）
cargo test --lib            # 仅单元测试（类似 dotnet test --filter Category=Unit）
cargo test --test smoke     # 仅运行 tests/smoke.rs
cargo test --test api_tests # 仅运行 tests/api_tests.rs

与 C# 的关键区别： 集成测试文件只能访问 crate 的 pub API。私有函数不可见——这强制你通过公共接口测试，这通常是更好的测试设计。
