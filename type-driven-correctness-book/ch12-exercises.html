练习 1：NVMe 管理命令（类型化命令）

为 NVMe 管理命令设计类型化命令接口：

  • Identify → IdentifyResponse（型号、序列号、固件版本）
  • GetLogPage → SmartLog（温度、可用备用、读取数据单元）
  • GetFeature → 特定功能的响应

要求：

  1. 命令类型决定响应类型
  2. 无运行时分发 —— 仅静态分发
  3. 添加 NamespaceId 新类型，防止将命名空间 ID 与其他 u32 混淆
解决方案 
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct NamespaceId(pub u32);

pub trait NvmeAdminCmd {
    type Response;
    fn opcode(&self) -> u8;
    fn nsid(&self) -> Option<NamespaceId>;
    fn parse_response(&self, raw: &[u8]) -> io::Result<Self::Response>;
}

pub struct Identify { pub nsid: NamespaceId }
impl NvmeAdminCmd for Identify {
    type Response = IdentifyResponse;
    fn opcode(&self) -> u8 { 0x06 }
    fn nsid(&self) -> Option<NamespaceId> { Some(self.nsid) }
    fn parse_response(&self, raw: &[u8]) -> io::Result<IdentifyResponse> { /* ... */ }
}

pub struct GetLogPage { pub log_id: u8 }
impl NvmeAdminCmd for GetLogPage {
    type Response = SmartLog;
    fn opcode(&self) -> u8 { 0x02 }
    fn parse_response(&self, raw: &[u8]) -> io::Result<SmartLog> { /* ... */ }
}

练习 2：固件更新状态机（类型状态）

建模 BMC 固件更新生命周期，带有状态：Idle → Uploading → Verifying → Applying → Rebooting → Complete。

要求：

  1. 每个状态是不同型号
  2. 上传只能从 Idle 开始
  3. 验证需要上传完成
  4. 应用只能在成功验证后发生 —— 使用 VerifiedImage 证明令牌

练习 3：传感器读数管道（量纲分析）

构建完整传感器管道：

  1. 定义新类型：RawAdc、Celsius、Fahrenheit、Volts、Millivolts、Watts
  2. 实现 From<Celsius> for Fahrenheit 等
  3. 创建 impl Mul<Volts, Output=Watts> for Amperes（P = V × I）
  4. 构建泛型 Threshold<T> 检查器
  5. 编写管道：ADC → 校准 → 阈值检查 → 结果

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]

impl From<Celsius> for Fahrenheit { fn from(c: Celsius) -> Self { Fahrenheit(c.0 * 9.0 / 5.0 + 32.0) } }

impl Mul<Amperes> for Volts {

    fn mul(self, rhs: Amperes) -> Watts { Watts(self.0 * rhs.0) }

练习 4：PCIe 能力遍历（幻象类型 + 验证边界）

建模 PCIe 能力链表：

  1. RawCapability —— 来自配置空间的未验证字节
  2. ValidCapability —— 解析和验证（通过 TryFrom）
  3. 每个能力类型（MSI、MSI-X、PCIe Express、电源管理）有自己的幻象类型寄存器布局
解决方案 
pub struct Msi;
pub struct MsiX;
pub struct PciExpress;
pub struct PowerMgmt;

pub struct RawCapability { pub id: u8, pub next_ptr: u8, pub data: Vec<u8> }

pub struct ValidCapability<Kind> {
    id: u8, next_ptr: u8, data: Vec<u8>, _kind: PhantomData<Kind>,
}

impl TryFrom<RawCapability> for ValidCapability<PowerMgmt> {
    type Error = &'static str;
    fn try_from(raw: RawCapability) -> Result<Self, Self::Error> {
        if raw.id != 0x01 { return Err("not PM"); }
        Ok(ValidCapability { id: raw.id, next_ptr: raw.next_ptr, data: raw.data, _kind: PhantomData })
    }
}

impl ValidCapability<PowerMgmt> {
    pub fn pm_control(&self) -> u16 { u16::from_le_bytes([self.data[0], self.data[1]]) }
}

练习 5：多协议健康检查（能力混入）

创建健康检查框架：

  1. 定义成分 traits：HasIpmi、HasRedfish、HasNvmeCli、HasGpio
  2. 创建混入 traits：
    • ThermalHealthMixin（需要 HasIpmi + HasGpio）
    • StorageHealthMixin（需要 HasNvmeCli）
    • BmcHealthMixin（需要 HasIpmi + HasRedfish）
  3. 构建实现所有成分 traits 的 FullPlatformController
  4. 构建仅实现 HasNvmeCli 的 StorageOnlyController

pub trait HasGpio { fn gpio_read_alert(&self, pin: u8) -> bool; }

关键要点

  1. 练习强化核心模式 —— 每个练习应用来自 ch02-ch10 的模式。
  2. 类型化命令用于协议交互 —— 命令类型决定响应类型。
  3. 类型状态用于状态机 —— 每个状态是不同类型。
  4. 量纲类型用于物理量 —— 防止单位混淆。
  5. 幻象类型用于资源跟踪 —— 编码宽度、方向、权限。
  6. 能力混入用于代码重用 —— blanket impl 提供行为。
