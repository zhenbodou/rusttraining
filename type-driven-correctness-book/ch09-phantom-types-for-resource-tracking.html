Type-Driven Correctness in Rust

幻象类型用于资源跟踪 🟡

你将学到： PhantomData 标记如何在类型级别编码寄存器宽度、DMA 方向和文件描述符状态 —— 在零运行时成本下防止整类资源不匹配 bug。

交叉引用： ch05（类型状态）、ch06（量纲类型）、ch08（混入）、ch10（集成）

问题：混淆资源

硬件资源在代码中看起来相似但不可互换：

  • 32 位寄存器和 16 位寄存器都是"寄存器"
  • 用于读取的 DMA 缓冲区和用于写入的 DMA 缓冲区看起来都像 *mut u8
  • 打开的文件描述符和关闭的文件描述符都是 i32

在 C 中：

// C —— 所有寄存器看起来都一样
uint32_t read_reg32(volatile void *base, uint32_t offset);
uint16_t read_reg16(volatile void *base, uint32_t offset);

// Bug：用 32 位函数读取 16 位寄存器
uint32_t status = read_reg32(pcie_bar, LINK_STATUS_REG);  // 应该是 reg16！

幻象类型参数

幻象类型是出现在结构体定义中但不在任何字段中的类型参数。它纯粹用于携带类型级信息：

use std::marker::PhantomData;

// 寄存器宽度标记 —— 零大小
pub struct Width8;
pub struct Width16;
pub struct Width32;
pub struct Width64;

/// 参数化为宽度的寄存器句柄。
/// PhantomData<W> 成本零字节 —— 它只是编译时标记。
pub struct Register<W> {
    base: usize,
    offset: usize,
    _width: PhantomData<W>,
}

impl Register<Width8> {
    pub fn read(&self) -> u8 { 0 }
    pub fn write(&self, _value: u8) {}
}

impl Register<Width16> {
    pub fn read(&self) -> u16 { 0 }
    pub fn write(&self, _value: u16) {}
}

impl Register<Width32> {
    pub fn read(&self) -> u32 { 0 }
    pub fn write(&self, _value: u32) {}
}

/// PCIe 配置空间寄存器定义。
pub struct PcieConfig { base: usize }

impl PcieConfig {
    pub fn vendor_id(&self) -> Register<Width16> {
        Register { base: self.base, offset: 0x00, _width: PhantomData }
    }
    pub fn bar0(&self) -> Register<Width32> {
        Register { base: self.base, offset: 0x10, _width: PhantomData }
    }
}

fn pcie_example() {
    let cfg = PcieConfig { base: 0xFE00_0000 };
    let vid: u16 = cfg.vendor_id().read();    // 返回 u16 ✅
    let bar: u32 = cfg.bar0().read();         // 返回 u32 ✅
    // 不能混淆：
    // let bad: u32 = cfg.vendor_id().read(); // ❌ 错误：期望 u16
}

DMA 缓冲区访问控制

DMA 缓冲区有方向：有些用于设备到主机（读取），其他用于主机到设备（写入）。使用错误的方向会损坏数据或导致总线错误：

use std::marker::PhantomData;

// 方向标记
pub struct ToDevice;     // 主机写入，设备读取
pub struct FromDevice;   // 设备写入，主机读取

/// 带有方向强制的 DMA 缓冲区。
pub struct DmaBuffer<Dir> {
    ptr: *mut u8,
    len: usize,
    dma_addr: u64,
    _dir: PhantomData<Dir>,
}

impl DmaBuffer<ToDevice> {
    pub fn write_data(&mut self, data: &[u8]) {
        unsafe { std::ptr::copy_nonoverlapping(data.as_ptr(), self.ptr, data.len()) }
    }
    pub fn device_addr(&self) -> u64 { self.dma_addr }
}

impl DmaBuffer<FromDevice> {
    pub fn read_data(&self) -> &[u8] {
        unsafe { std::slice::from_raw_parts(self.ptr, self.len) }
    }
    pub fn device_addr(&self) -> u64 { self.dma_addr }
}

// 不能写入 FromDevice 缓冲区：
// fn oops(buf: &mut DmaBuffer<FromDevice>) { buf.write_data(&[1, 2, 3]); }  // ❌

// 不能从 ToDevice 缓冲区读取：
// fn oops2(buf: &DmaBuffer<ToDevice>) { let data = buf.read_data(); }  // ❌

文件描述符所有权

常见 bug：在文件描述符关闭后使用它。幻象类型可以跟踪打开/关闭状态：

use std::marker::PhantomData;

pub struct Open;
pub struct Closed;

/// 带有状态跟踪的文件描述符。
pub struct Fd<State> {
    raw: i32,
    _state: PhantomData<State>,
}

impl Fd<Open> {
    pub fn open(path: &str) -> Result<Self, String> {
        Ok(Fd { raw: 3, _state: PhantomData })
    }
    pub fn read(&self, buf: &mut [u8]) -> Result<usize, String> { Ok(0) }
    pub fn write(&self, data: &[u8]) -> Result<usize, String> { Ok(data.len()) }
    pub fn close(self) -> Fd<Closed> {
        Fd { raw: self.raw, _state: PhantomData }
    }
}

impl Fd<Closed> {
    // 没有 read() 或 write() 方法 —— 它们在 Fd<Closed> 上不存在。
    pub fn raw_fd(&self) -> i32 { self.raw }
}

fn fd_example() -> Result<(), String> {
    let fd = Fd::open("/dev/ipmi0")?;
    let mut buf = [0u8; 256];
    fd.read(&mut buf)?;
    let closed = fd.close();
    // closed.read(&mut buf)?;  // ❌ Fd<Closed> 上没有方法 `read`
    Ok(())
}

将幻象类型与早期模式结合

幻象类型与我们所见的一切组合：

use std::marker::PhantomData;
pub struct Width16;
pub struct Register<W> { _w: PhantomData<W> }
impl Register<Width16> { pub fn read(&self) -> u16 { 0 } }
#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
pub struct Celsius(pub f64);

/// 结合幻象类型（寄存器宽度）与量纲类型（Celsius）。
fn read_temp_sensor(reg: &Register<Width16>) -> Celsius {
    let raw = reg.read();  // 幻象类型保证 u16
    Celsius(raw as f64 * 0.0625)  // 返回类型保证 Celsius
}

何时使用幻象类型

场景使用幻象参数？
寄存器宽度编码✅ 总是 —— 防止宽度不匹配
DMA 缓冲区方向✅ 总是 —— 防止数据损坏
文件描述符状态✅ 总是 —— 防止关闭后使用
内存区域权限（R/W/X）✅ 总是 —— 强制执行访问控制
通用容器（Vec、HashMap）❌ 不 —— 使用具体类型参数
运行时可变属性❌ 不 —— 幻象类型是编译时专用

练习：内存区域权限

为具有读取、写入和执行权限的内存区域设计幻象类型：

  • MemRegion<ReadOnly> 有 fn read(&self, offset: usize) -> u8
  • MemRegion<ReadWrite> 有 read 和 write
  • MemRegion<Executable> 有 read 和 fn execute(&self)
  • 写入 ReadOnly 或执行 ReadWrite 不应编译。
解决方案 
use std::marker::PhantomData;

pub struct ReadOnly;
pub struct ReadWrite;
pub struct Executable;

pub struct MemRegion<Perm> {
    base: *mut u8,
    len: usize,
    _perm: PhantomData<Perm>,
}

// 所有权限类型都可读取
impl<P> MemRegion<P> {
    pub fn read(&self, offset: usize) -> u8 {
        assert!(offset < self.len);
        unsafe { *self.base.add(offset) }
    }
}

impl MemRegion<ReadWrite> {
    pub fn write(&mut self, offset: usize, val: u8) {
        assert!(offset < self.len);
        unsafe { *self.base.add(offset) = val; }
    }
}

impl MemRegion<Executable> {
    pub fn execute(&self) {}
}

// ❌ region_ro.write(0, 0xFF);  // 编译错误：没有方法 `write`
// ❌ region_rw.execute();       // 编译错误：没有方法 `execute`

关键要点

  1. PhantomData 以零大小携带类型级信息 —— 标记仅存在于编译器中。
  2. 寄存器宽度不匹配成为编译错误 —— Register<Width16> 返回 u16，不是 u32。
  3. DMA 方向在结构上强制执行 —— DmaBuffer<Read> 没有 write() 方法。
  4. 与量纲类型结合（ch06） —— Register<Width16> 可以通过解析步骤返回 Celsius。
  5. 幻象类型仅编译时 —— 它们不适用于运行时可变属性；对那些使用枚举。
