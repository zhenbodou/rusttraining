镜像问题

第 17 章问："我如何正确消费 Redfish？" 本章问镜像问题："我如何正确生产 Redfish？"

一个处理程序中的四个 bug。在客户端，每个 bug 影响一个客户端。在服务器端，每个 bug 影响每个查询此 BMC 的客户端。

第 1 节 —— 响应构建器类型状态："构造，不要序列化"（ch07 反向）

第 7 章教导"解析，不要验证" —— 在边界验证入站数据一次，在类型中携带证明。服务器端镜像是**"构造，不要序列化"** —— 通过构建器构建出站响应，在全部必需字段存在时门控 .build()。

use std::marker::PhantomData;

pub struct HasField;
pub struct MissingField;

pub struct ComputerSystemBuilder<Name, Uuid, PowerState, Status> {
    // 必需字段 —— 在类型级别跟踪
    name: Option<String>,
    uuid: Option<String>,
    power_state: Option<PowerStateValue>,
    status: Option<ResourceStatus>,
    // 可选字段 —— 不跟踪（总是可设置）
    manufacturer: Option<String>,
    model: Option<String>,
    serial_number: Option<String>,
    bios_version: Option<String>,
    _markers: PhantomData<(Name, Uuid, PowerState, Status)>,
}

#[derive(Debug, Clone, serde::Serialize)]
pub enum PowerStateValue { On, Off, PoweringOn, PoweringOff }

#[derive(Debug, Clone, serde::Serialize)]
pub struct ResourceStatus {
    #[serde(rename = "State")] pub state: StatusState,
    #[serde(rename = "Health")] pub health: HealthValue,
}

#[derive(Debug, Clone, Copy, serde::Serialize)]
pub enum StatusState { Enabled, Disabled, Absent }

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, serde::Serialize)]
pub enum HealthValue { OK, Warning, Critical }

impl ComputerSystemBuilder<MissingField, MissingField, MissingField, MissingField> {
    pub fn new() -> Self {
        ComputerSystemBuilder {
            name: None, uuid: None, power_state: None, status: None,
            manufacturer: None, model: None, serial_number: None,
            bios_version: None, _markers: PhantomData,
        }
    }
}

impl<U, P, S> ComputerSystemBuilder<MissingField, U, P, S> {
    pub fn name(self, name: String) -> ComputerSystemBuilder<HasField, U, P, S> {
        ComputerSystemBuilder { name: Some(name), ..self }
    }
}

impl<N, P, S> ComputerSystemBuilder<N, MissingField, P, S> {
    pub fn uuid(self, uuid: String) -> ComputerSystemBuilder<N, HasField, P, S> {
        ComputerSystemBuilder { uuid: Some(uuid), ..self }
    }
}

impl<N, U, S> ComputerSystemBuilder<N, U, MissingField, S> {
    pub fn power_state(self, ps: PowerStateValue) -> ComputerSystemBuilder<N, U, HasField, S> {
        ComputerSystemBuilder { power_state: Some(ps), ..self }
    }
}

impl<N, U, P> ComputerSystemBuilder<N, U, P, MissingField> {
    pub fn status(self, status: ResourceStatus) -> ComputerSystemBuilder<N, U, P, HasField> {
        ComputerSystemBuilder { status: Some(status), ..self }
    }
}

impl<N, U, P, S> ComputerSystemBuilder<N, U, P, S> {
    pub fn manufacturer(mut self, m: String) -> Self { self.manufacturer = Some(m); self }
    pub fn model(mut self, m: String) -> Self { self.model = Some(m); self }
    pub fn serial_number(mut self, s: String) -> Self { self.serial_number = Some(s); self }
    pub fn bios_version(mut self, v: String) -> Self { self.bios_version = Some(v); self }
}

impl ComputerSystemBuilder<HasField, HasField, HasField, HasField> {
    pub fn build(self, id: &str) -> serde_json::Value {
        serde_json::json!({
            "@odata.id": format!("/redfish/v1/Systems/{id}"),
            "@odata.type": "#ComputerSystem.v1_13_0.ComputerSystem",
            "Id": id,
            "Name": self.name.unwrap(),
            "UUID": self.uuid.unwrap(),
            "PowerState": self.power_state.unwrap(),
            "Status": self.status.unwrap(),
        })
    }
}

// ✅ 所有必需字段设置 —— .build() 可用：
// ComputerSystemBuilder::new()
//     .name("PowerEdge R750".into())
//     .uuid("4c4c4544-...".into())
//     .power_state(PowerStateValue::On)
//     .status(ResourceStatus { .. })
//     .build("1")

// ❌ 缺少 "Name" —— 编译错误：
// ComputerSystemBuilder::new()
//     .uuid("4c4c4544-...".into())
//     .power_state(PowerStateValue::On)
//     .status(ResourceStatus { .. })
//     .build("1")
//   ERROR: method `build` not found for
//   `ComputerSystemBuilder<MissingField, HasField, HasField, HasField>`

消除的 bug 类： 模式非合规响应。处理程序物理上不能序列化 ComputerSystem 而不提供每个必需字段。编译器错误消息甚至告诉你哪个字段缺失 —— 它就在类型参数中：Name 位置中的 MissingField。

第 2 节 —— 源可用性令牌（能力令牌，ch04 —— 新转折）

在 ch04 和 ch17 中，能力令牌证明授权 —— "调用者被允许做这个。"在服务器端，相同模式证明可用性 —— "此数据源成功初始化。"

BMC 查询的每个子系统可以独立失败。SMBIOS 表可能损坏。传感器子系统可能仍在初始化。PCIe 总线扫描可能超时。将每个编码为证明令牌：

/// 证明 SMBIOS 表成功解析。
/// 仅由 SMBIOS init 函数生成。
pub struct SmbiosReady { _private: () }

/// 证明 IPMI 传感器子系统响应。
pub struct SensorsReady { _private: () }

/// 证明 PCIe 总线扫描完成。
pub struct PcieReady { _private: () }

pub struct SmbiosTables {
    pub product_name: String, pub manufacturer: String,
    pub serial_number: String, pub uuid: String,
}

pub struct SensorCache {
    pub cpu_temp: Celsius, pub inlet_temp: Celsius,
    pub fan_readings: Vec<(String, Rpm)>,
    pub psu_power: Vec<(String, Watts)>,
}

pub fn init_smbios() -> Option<(SmbiosReady, SmbiosTables)> {
    // 读取 SMBIOS 入口点，解析表...
    // 如果表不存在或损坏则返回 None
    Some((SmbiosReady { _private: () }, SmbiosTables { /* ... */ }))
}

pub fn init_sensors() -> Option<(SensorsReady, SensorCache)> {
    // 初始化 SDR 存储库，读取所有传感器...
    // 如果 IPMI 子系统不响应则返回 None
    Some((SensorsReady { _private: () }, SensorCache { /* ... */ }))
}

现在，从数据源填充构建器字段的函数需要相应的证明令牌：

/// 从 SMBIOS 填充字段。需要证明 SMBIOS 可用。
fn populate_from_smbios<P, S>(
    builder: ComputerSystemBuilder<MissingField, MissingField, P, S>,
    _proof: &SmbiosReady,
    tables: &SmbiosTables,
) -> ComputerSystemBuilder<HasField, HasField, P, S> {
    builder.name(tables.product_name.clone()).uuid(tables.uuid.clone())
}

/// 当 SMBIOS 不可用时的回退 —— 用安全默认值提供必需字段。
fn populate_smbios_fallback<P, S>(
    builder: ComputerSystemBuilder<MissingField, MissingField, P, S>,
) -> ComputerSystemBuilder<HasField, HasField, P, S> {
    builder.name("Unknown System".into()).uuid("00000000-0000-0000-0000-000000000000".into())
}

处理程序根据哪些令牌可用来选择路径：

fn build_computer_system(
    smbios: &Option<(SmbiosReady, SmbiosTables)>,
    power_state: PowerStateValue,
    health: ResourceStatus,
) -> serde_json::Value {
    let builder = ComputerSystemBuilder::new().power_state(power_state).status(health);
    let builder = match smbios {
        Some((proof, tables)) => populate_from_smbios(builder, proof, tables),
        None => populate_smbios_fallback(builder),
    };
    // 两条路径都为 Name 和 UUID 生成 HasField。
    // .build() 在任一种方式中都可用。
    builder.build("1")
}

消除的 bug 类： 调用初始化失败的子系统。如果 SMBIOS 没有解析，你没有 SmbiosReady 令牌 —— 编译器强制你通过回退路径。没有运行时 if (smbios != NULL) 可以忘记。

第 3 节 —— 序列化边界的量纲类型（ch06）

在客户端（ch17 §4），量纲类型防止读取 °C 作为 RPM。在服务器端，它们防止写入 RPM 到 Celsius JSON 字段。这可能更危险 —— 服务器上的错误值传播到每个客户端。

use serde::Serialize;

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd, Serialize)]
pub struct Celsius(pub f64);
#[derive(Debug, Clone, Copy, PartialEq, PartialOrd, Serialize)]
pub struct Rpm(pub u32);
#[derive(Debug, Clone, Copy, PartialEq, PartialOrd, Serialize)]
pub struct Watts(pub f64);

#[derive(Serialize)]
#[serde(rename_all = "PascalCase")]
pub struct TemperatureMember {
    pub member_id: String,
    pub name: String,
    pub reading_celsius: Celsius,           // ← 必须是 Celsius
    #[serde(skip_serializing_if = "Option::is_none")]
    pub upper_threshold_critical: Option<Celsius>,
    pub status: ResourceStatus,
}

#[derive(Serialize)]
#[serde(rename_all = "PascalCase")]
pub struct FanMember {
    pub member_id: String,
    pub name: String,
    pub reading: Rpm,                       // ← 必须是 Rpm
    pub reading_units: &'static str,        // 总是 "RPM"
    pub status: ResourceStatus,
}

fn build_thermal_response(_proof: &SensorsReady, cache: &SensorCache) -> serde_json::Value {
    let temps = vec![
        TemperatureMember {
            member_id: "0".into(), name: "CPU Temp".into(),
            reading_celsius: cache.cpu_temp,     // Celsius → Celsius ✅
            upper_threshold_critical: Some(Celsius(95.0)),
            status: ResourceStatus { state: StatusState::Enabled, health: HealthValue::OK },
        },
        // ❌ 编译错误 —— 不能将 Rpm 放入 Celsius 字段：
        // TemperatureMember { reading_celsius: cache.fan_readings[0].1, .. }
    ];
    let fans: Vec<FanMember> = cache.fan_readings.iter().enumerate().map(|(i, (name, rpm))| {
        FanMember {
            member_id: i.to_string(), name: name.clone(),
            reading: *rpm,                       // Rpm → Rpm ✅
            reading_units: "RPM",
            status: ResourceStatus { state: StatusState::Enabled, health: HealthValue::OK },
        }
    }).collect();
    serde_json::json!({ "@odata.id": "/redfish/v1/Chassis/1/Thermal", "Temperatures": temps, "Fans": fans })
}

消除的 bug 类：

  • 单位混淆： Celsius ≠ Rpm —— 编译器拒绝将风扇读数放入温度字段。
  • 错误序列化： reading_celsius 字段需要 Celsius，不是 f64 或 Rpm。
  • 类型安全 JSON 生成： serde 使用 Serialize impl 正确序列化量纲类型。

第 4 节 —— 健康汇总（来自 ch07 SEL 的消费者管道）

来自 ch07 的 TypedSelSummary 馈送到 Redfish 健康汇总：

/// 从每个子系统健康计算整体系统健康。
/// 使用 Ord 实例保证正确性：Critical > Warning > OK。
fn compute_system_health(
    sensor_cache: Option<&(SensorsReady, SensorCache)>,
    sel_summary: Option<&(SelReady, TypedSelSummary)>,
) -> ResourceStatus {
    let mut health = HealthValue::OK;

    // 来自传感器的健康
    if let Some((_, cache)) = sensor_cache {
        if cache.cpu_temp > Celsius(95.0) { health = HealthValue::Critical; }
        if cache.inlet_temp > Celsius(42.0) { health = health.max(HealthValue::Warning); }
    }

    // 来自 SEL 的健康（来自 ch07 的消费者管道）
    if let Some((_, summary)) = sel_summary {
        health = health.max(summary.processor_health);
        health = health.max(summary.memory_health);
        health = health.max(summary.power_health);
        health = health.max(summary.thermal_health);
        health = health.max(summary.fan_health);
    }

    ResourceStatus {
        state: StatusState::Enabled,
        health,
        health_rollup: Some(health),
    }
}

消除的 bug 类：

  • 忘记子系统 —— 每个子系统显式汇总。
  • 错误严重性 —— HealthValue::max() 保证正确汇总（Critical > Warning > OK）。
  • 来自 ch07 的类型化 SEL —— TypedSelSummary 携带每个子系统健康，不是损失性的 bool。

第 5 节 —— 模式版本化（幻象类型，ch09）

Redfish 模式随时间演进。幻象类型跟踪响应对哪个模式版本合规：

pub struct SchemaV1_5;
pub struct SchemaV1_13;

pub struct VersionedResponse<Schema> {
    data: serde_json::Value,
    _schema: PhantomData<Schema>,
}

impl VersionedResponse<SchemaV1_13> {
    pub fn with_last_reset_time(data: serde_json::Value, timestamp: &str) -> Self {
        // LastResetTime 在 v1.13 中添加
        let mut obj = data;
        obj["LastResetTime"] = serde_json::json!(timestamp);
        VersionedResponse { data: obj, _schema: PhantomData }
    }
}

// ❌ 编译错误：VersionedResponse<SchemaV1_5> 没有 with_last_reset_time()
// VersionedResponse::<SchemaV1_5>::with_last_reset_time(data, "2024-01-01");

消除的 bug 类：

  • 访问不存在的字段 —— LastResetTime 方法仅在 SchemaV1_13 上可用。
  • 模式版本不匹配 —— 类型强制执行正确的模式版本用于字段访问。

完整示例：ComputerSystem GET 处理程序

fn handle_get_computer_system(
    smbios: &Option<(SmbiosReady, SmbiosTables)>,
    sensors: &Option<(SensorsReady, SensorCache)>,
    sel: &Option<(SelReady, TypedSelSummary)>,
    power_state: PowerStateValue,
) -> serde_json::Value {
    // 1. 计算健康汇总
    let health = compute_system_health(
        sensors.as_ref(),
        sel.as_ref(),
    );

    // 2. 构建响应，使用类型状态保证完整性
    let builder = ComputerSystemBuilder::new()
        .power_state(power_state)
        .status(health);

    let builder = match smbios {
        Some((proof, tables)) => populate_from_smbios(builder, proof, tables),
        None => populate_smbios_fallback(builder),
    };

    // 3. 可选字段
    let builder = if let Some((_, cache)) = sensors {
        builder.bios_version("2.1.0".into())
    } else {
        builder
    };

    // 4. 构建并返回（.build() 可用，因为必需字段已设置）
    builder.build("1")
}

关键要点

  1. 构建器类型状态保证响应完整性 —— .build() 门控在所有必需字段上。
  2. 源可用性令牌证明数据源初始化 —— 没有令牌，你不能从该源填充。
  3. 量纲类型防止序列化单位混淆 —— Celsius ≠ Rpm 在 JSON 生成中。
  4. 健康汇总使用 Ord 用于正确严重性 —— Critical > Warning > OK。
  5. 模式版本化使用幻象类型 —— VersionedResponse<SchemaV1_13> 有 v1.13 专用字段。
  6. 镜像客户端模式（ch17） —— 客户端验证入站数据，服务器构建立即合规的出站数据。
  7. 七个模式组合成零开销服务器 —— 每个模式违规是编译错误，不是运行时 bug。
