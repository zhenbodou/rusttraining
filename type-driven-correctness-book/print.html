Type-Driven Correctness in Rust

Rust 中的类型驱动正确性

一本实用指南，介绍如何使用 Rust 的类型系统让整类 bug 无法编译。虽然配套的 Rust 模式 书籍涵盖了机制（trait、关联类型、type-state），但本指南展示了如何将这些机制应用到现实世界的领域 — 硬件诊断、加密、协议验证和嵌入式系统。

这里的每个模式都遵循一个原则：将不变量从运行时检查推入类型系统，让编译器强制执行。

运行时检查的代价

考虑诊断代码库中典型的运行时守卫：

fn read_sensor(sensor_type: &str, raw: &[u8]) -> f64 {
    match sensor_type {
        "temperature" => raw[0] as i8 as f64,          // 有符号字节
        "fan_speed"   => u16::from_le_bytes([raw[0], raw[1]]) as f64,
        "voltage"     => u16::from_le_bytes([raw[0], raw[1]]) as f64 / 1000.0,
        _             => panic!("unknown sensor type: {sensor_type}"),
    }
}

这个函数有四种失败模式是编译器无法捕获的：

  1. 拼写错误："temperture" → 运行时 panic
  2. raw 长度错误：fan_speed 只有 1 字节 → 运行时 panic
  3. 调用者将返回的 f64 当作 RPM 使用，而实际上是 °C → 逻辑 bug，静默
  4. 添加了新传感器类型但未更新此 match → 运行时 panic

每种失败模式都是部署后才发现的。测试有帮助，但它们只覆盖有人想到要编写的情况。类型系统覆盖所有情况，包括没人想象到的情况。

正确性的三个级别

级别 1 — 值正确性

使无效值无法表示。

// ❌ 任何 u16 都可以是"端口" — 0 无效但可以编译
fn connect(port: u16) { /* ... */ }

// ✅ 只有经过验证的端口才能存在
pub struct Port(u16);  // 私有字段

impl TryFrom<u16> for Port {
    type Error = &'static str;
    fn try_from(v: u16) -> Result<Self, Self::Error> {
        if v > 0 { Ok(Port(v)) } else { Err("port must be > 0") }
    }
}

fn connect(port: Port) { /* ... */ }
// Port(0) 永远无法构造 — 不变量在任何地方都成立

硬件示例： SensorId(u8) — 包装原始传感器编号，并验证它在 SDR 范围内。

级别 2 — 状态正确性

使无效转换无法表示。

use std::marker::PhantomData;

struct Disconnected;
struct Connected;

struct Socket<State> {
    fd: i32,
    _state: PhantomData<State>,
}

impl Socket<Disconnected> {
    fn connect(self, addr: &str) -> Socket<Connected> {
        // ... 连接逻辑 ...
        Socket { fd: self.fd, _state: PhantomData }
    }
}

impl Socket<Connected> {
    fn send(&mut self, data: &[u8]) { /* ... */ }
    fn disconnect(self) -> Socket<Disconnected> {
        Socket { fd: self.fd, _state: PhantomData }
    }
}

// Socket<Disconnected> 没有 send() 方法 — 尝试调用会编译错误

硬件示例： GPIO 引脚模式 — Pin<Input> 有 read() 但没有 write()。

级别 3 — 协议正确性

使无效交互无法表示。

use std::io;

trait IpmiCmd {
    type Response;
    fn parse_response(&self, raw: &[u8]) -> io::Result<Self::Response>;
}

// 简化用于说明 — 见第 2 章完整的 trait，包含
// net_fn()、cmd_byte()、payload() 和 parse_response()。

struct ReadTemp { sensor_id: u8 }
impl IpmiCmd for ReadTemp {
    type Response = Celsius;
    fn parse_response(&self, raw: &[u8]) -> io::Result<Celsius> {
        Ok(Celsius(raw[0] as i8 as f64))
    }
}

#[derive(Debug)] struct Celsius(f64);

fn execute<C: IpmiCmd>(cmd: &C, raw: &[u8]) -> io::Result<C::Response> {
    cmd.parse_response(raw)
}
// ReadTemp 总是返回 Celsius — 不会意外得到 Rpm

硬件示例： IPMI、Redfish、NVMe Admin 命令 — 请求类型决定响应类型。

类型作为编译器检查的保证

当你编写：

fn execute<C: IpmiCmd>(cmd: &C) -> io::Result<C::Response>

你不仅仅是在编写一个函数 — 你是在陈述一个保证："对于任何实现 IpmiCmd 的命令类型 C，执行它会产生精确的 C::Response。"编译器在每次构建代码时验证这个保证。如果类型不匹配，程序将无法编译。

这就是 Rust 类型系统如此强大的原因 — 它不仅仅是捕获错误，它是在编译时强制执行正确性。

何时不使用这些模式

构造即正确并不总是正确的选择：

情况建议
安全关键边界（电源时序、加密）✅ 始终使用 — 这里的 bug 会融化硬件或泄露机密
跨模块公共 API✅ 通常使用 — 误用应该是编译错误
有 3+ 个状态的状态机✅ 通常使用 — type-state 防止错误转换
一个 50 行函数内的内部辅助❌ 过度 — 简单的 assert! 就足够了
原型设计 / 探索未知硬件❌ 先用原始类型 — 理解行为后再细化
面向用户的 CLI 解析⚠️ 边界使用 clap + TryFrom，内部使用原始类型即可

关键问题是："如果这个 bug 在生产环境中发生，有多严重？"

  • 风扇停止 → GPU 融化 → 使用类型
  • 错误的 DER 记录 → 客户得到错误数据 → 使用类型
  • 调试日志消息略有错误 → 使用 assert!

关键要点

  1. 三个正确性级别 — 值（newtype）、状态（type-state）、协议（关联类型） — 每个都消除了更广泛的 bug 类别。
  2. 类型作为保证 — 每个泛型函数签名都是编译器在每次构建时检查的契约。
  3. 代价问题 — "如果这个 bug 发布，有多严重？"决定类型还是测试是正确的工具。
  4. 类型补充测试 — 它们消除整个类别；测试覆盖特定值和边界情况。
  5. 知道何时停止 — 内部辅助和临时原型很少需要类型级强制执行。

类型化命令模式

步骤 1 —— 领域新类型

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
pub struct Celsius(pub f64);

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
pub struct Rpm(pub u32);  // u32：原始 IPMI 传感器值（整数 RPM）

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
pub struct Volts(pub f64);

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
pub struct Watts(pub f64);

关于 Rpm(u32) 与 Rpm(f64) 的说明： 在本章中内部类型是 u32， 因为 IPMI 传感器读数是整数值。在 ch06（量纲分析）中，Rpm 使用 f64 以支持算术运算（平均、缩放）。两者都是有效的 —— 无论内部类型如何， 新类型都能防止跨单位混淆。

步骤 2 —— 命令 trait（类型索引分发）

关联类型 Response 是关键 —— 它将每个命令结构体绑定到其返回类型。 每个实现结构体将 Response 固定到特定的领域类型，因此 execute() 总是 返回完全正确的类型：

pub trait IpmiCmd {
    /// "类型索引" —— 决定 execute() 返回什么。
    type Response;

    fn net_fn(&self) -> u8;
    fn cmd_byte(&self) -> u8;
    fn payload(&self) -> Vec<u8>;

    /// 解析封装在这里 —— 每个命令知道自己的字节布局。
    fn parse_response(&self, raw: &[u8]) -> io::Result<Self::Response>;
}

步骤 3 —— 每个命令一个结构体

pub struct ReadTemp { pub sensor_id: u8 }
impl IpmiCmd for ReadTemp {
    type Response = Celsius;
    fn net_fn(&self) -> u8 { 0x04 }
    fn cmd_byte(&self) -> u8 { 0x2D }
    fn payload(&self) -> Vec<u8> { vec![self.sensor_id] }
    fn parse_response(&self, raw: &[u8]) -> io::Result<Celsius> {
        if raw.is_empty() {
            return Err(io::Error::new(io::ErrorKind::InvalidData, "empty response"));
        }
        // 注意：ch01 的无类型示例使用 `raw[0] as i8 as f64`（有符号）
        // 因为该函数演示的是没有 SDR 元数据的通用解析。这里我们使用无符号（`as f64`），
        // 因为 IPMI 规范 §35.5 中的 SDR 线性化公式将无符号原始读数转换为校准值。
        // 在生产环境中，应用完整的 SDR 公式：result = (M × raw + B) × 10^(R_exp)。
        Ok(Celsius(raw[0] as f64))  // 无符号原始字节，根据 SDR 公式转换
    }
}

pub struct ReadFanSpeed { pub fan_id: u8 }
impl IpmiCmd for ReadFanSpeed {
    type Response = Rpm;
    fn net_fn(&self) -> u8 { 0x04 }
    fn cmd_byte(&self) -> u8 { 0x2D }
    fn payload(&self) -> Vec<u8> { vec![self.fan_id] }
    fn parse_response(&self, raw: &[u8]) -> io::Result<Rpm> {
        if raw.len() < 2 {
            return Err(io::Error::new(io::ErrorKind::InvalidData,
                format!("fan speed needs 2 bytes, got {}", raw.len())));
        }
        Ok(Rpm(u16::from_le_bytes([raw[0], raw[1]]) as u32))
    }
}

pub struct ReadVoltage { pub rail: u8 }
impl IpmiCmd for ReadVoltage {
    type Response = Volts;
    fn net_fn(&self) -> u8 { 0x04 }
    fn cmd_byte(&self) -> u8 { 0x2D }
    fn payload(&self) -> Vec<u8> { vec![self.rail] }
    fn parse_response(&self, raw: &[u8]) -> io::Result<Volts> {
        if raw.len() < 2 {
            return Err(io::Error::new(io::ErrorKind::InvalidData,
                format!("voltage needs 2 bytes, got {}", raw.len())));
        }
        Ok(Volts(u16::from_le_bytes([raw[0], raw[1]]) as f64 / 1000.0))
    }
}

fn diagnose_thermal_typed(bmc: &BmcConnection) -> io::Result<()> {

IPMI：不会混淆的传感器读数

添加新传感器只需要一个结构体 + 一个 impl —— 无需分散的解析：

pub struct ReadPowerDraw { pub domain: u8 }
impl IpmiCmd for ReadPowerDraw {
    type Response = Watts;
    fn net_fn(&self) -> u8 { 0x04 }
    fn cmd_byte(&self) -> u8 { 0x2D }
    fn payload(&self) -> Vec<u8> { vec![self.domain] }
    fn parse_response(&self, raw: &[u8]) -> io::Result<Watts> {
        if raw.len() < 2 {
            return Err(io::Error::new(io::ErrorKind::InvalidData,
                format!("power draw needs 2 bytes, got {}", raw.len())));
        }
        Ok(Watts(u16::from_le_bytes([raw[0], raw[1]]) as f64))
    }
}

练习：PLDM 类型化命令

设计一个 PldmCmd trait（与 IpmiCmd 形状相同）用于两个 PLDM 命令：

  • GetFwParams → FwParamsResponse { active_version: String, pending_version: Option<String> }
  • QueryDeviceIds → DeviceIdResponse { descriptors: Vec<Descriptor> }

要求：静态分发，parse_response 返回 io::Result<Self::Response>。

解决方案 
use std::io;

pub trait PldmCmd {
    type Response;
    fn pldm_type(&self) -> u8;
    fn command_code(&self) -> u8;
    fn parse_response(&self, raw: &[u8]) -> io::Result<Self::Response>;
}

#[derive(Debug, Clone)]
pub struct FwParamsResponse {
    pub active_version: String,
    pub pending_version: Option<String>,
}

pub struct GetFwParams;
impl PldmCmd for GetFwParams {
    type Response = FwParamsResponse;
    fn pldm_type(&self) -> u8 { 0x05 } // 固件更新
    fn command_code(&self) -> u8 { 0x02 }
    fn parse_response(&self, raw: &[u8]) -> io::Result<FwParamsResponse> {
        // 简化 —— 真实实现解码 PLDM 固件更新规范字段
        if raw.len() < 4 {
            return Err(io::Error::new(io::ErrorKind::InvalidData, "too short"));
        }
        Ok(FwParamsResponse {
            active_version: String::from_utf8_lossy(&raw[..4]).to_string(),
            pending_version: None,
        })
    }
}

#[derive(Debug, Clone)]
pub struct Descriptor { pub descriptor_type: u16, pub data: Vec<u8> }

#[derive(Debug, Clone)]
pub struct DeviceIdResponse { pub descriptors: Vec<Descriptor> }

pub struct QueryDeviceIds;
impl PldmCmd for QueryDeviceIds {
    type Response = DeviceIdResponse;
    fn pldm_type(&self) -> u8 { 0x05 }
    fn command_code(&self) -> u8 { 0x04 }
    fn parse_response(&self, raw: &[u8]) -> io::Result<DeviceIdResponse> {
        Ok(DeviceIdResponse { descriptors: vec![] }) // 存根
    }
}

关键要点

  1. 关联类型 = 编译时契约 —— 命令 trait 上的 type Response 将每个请求锁定到恰好一个响应类型。
  2. 解析被封装 —— 字节布局知识存在于 parse_response 中，而不是分散在调用者中。
  3. 零成本分发 —— 泛型 execute<C: IpmiCmd> 单态化为直接调用，无虚表。
  4. 一个模式，多个协议 —— IPMI、Redfish、NVMe、PLDM、MCTP 都适合同样的 trait Cmd { type Response; } 形状。
  5. 枚举分发桥接静态和动态 —— 将类型化命令包装在枚举中，用于运行时驱动的脚本，同时不失去每个分支内的类型安全性。
  6. 渐进复杂度增强直觉 —— IPMI（传感器 ID → 读数）、Redfish（端点 → JSON 模式）和 NVMe（操作码 → 4 KB 结构覆盖）都使用相同的 trait 形状，但每个节拍增加一层解析复杂度。

一次性类型 —— 通过所有权实现加密保证 🟡

你将学到： Rust 的移动语义如何作为线性类型系统，使 nonce 重用、双重密钥协商和意外熔丝重新编程在编译时成为不可能。

交叉引用： ch01（哲学）、ch04（能力令牌）、ch05（类型状态）、ch14（编译失败测试）

Nonce 重用灾难

在认证加密（AES-GCM、ChaCha20-Poly1305）中，对同一密钥重用 nonce 是灾难性的 —— 它会泄露两个明文的 XOR，通常还会泄露认证密钥本身。这不是理论上的担忧：

  • 2016：TLS 中 AES-GCM 的 Forbidden Attack —— nonce 重用允许明文恢复
  • 2020：多个 IoT 固件更新系统因 RNG 不佳被发现重用 nonce

在 C/C++ 中，nonce 只是 uint8_t[12]。没有什么能阻止你使用它两次。

// C —— 没有什么能阻止 nonce 重用
uint8_t nonce[12];
generate_nonce(nonce);
encrypt(key, nonce, msg1, out1);   // ✅ 第一次使用
encrypt(key, nonce, msg2, out2);   // 🐛 灾难性：相同的 nonce

作为线性类型的移动语义

Rust 的所有权系统实际上是线性类型系统 —— 值只能使用恰好一次（移动），除非它实现 Copy。ring crate 利用了这一点：

// ring::aead::Nonce 是：
// - 不是 Clone
// - 不是 Copy
// - 按值使用时被消耗
pub struct Nonce(/* private */);

impl Nonce {
    pub fn try_assume_unique_for_key(value: &[u8]) -> Result<Self, Unspecified> {
        // ...
    }
    // 没有 Clone，没有 Copy —— 只能使用一次
}

当你将 Nonce 传递给 seal_in_place() 时，它被移动了：

// 伪代码，反映 ring 的 API 形状
fn seal_in_place(
    key: &SealingKey,
    nonce: Nonce,       // ← 被移动，不是借用
    data: &mut Vec<u8>,
) -> Result<(), Error> {
    // ... 原地加密数据 ...
    // nonce 被消耗 —— 不能再使用
    Ok(())
}

尝试重用它：

fn bad_encrypt(key: &SealingKey, data1: &mut Vec<u8>, data2: &mut Vec<u8>) {
    // .unwrap() 是安全的 —— 12 字节数组总是有效的 nonce。
    let nonce = Nonce::try_assume_unique_for_key(&[0u8; 12]).unwrap();

}

案例研究：ring 的 Nonce

ring crate 更进一步，使用 NonceSequence —— 一个生成 nonce 的 trait，也是不可克隆的：

/// 唯一 nonce 序列。
/// 不是 Clone —— 一旦绑定到密钥，就不能复制。
pub trait NonceSequence {
    fn advance(&mut self) -> Result<Nonce, Unspecified>;
}

/// SealingKey 包装 NonceSequence —— 每次 seal() 自动推进。
pub struct SealingKey<N: NonceSequence> {
    key: UnboundKey,   // 在构造期间被消耗
    nonce_seq: N,
}

impl<N: NonceSequence> SealingKey<N> {
    pub fn new(key: UnboundKey, nonce_seq: N) -> Self {
        // UnboundKey 被移动 —— 不能同时用于密封和解封
        SealingKey { key, nonce_seq }
    }

    pub fn seal_in_place_append_tag(
        &mut self,       // &mut —— 独占访问
        aad: Aad<&[u8]>,
        in_out: &mut Vec<u8>,
    ) -> Result<(), Unspecified> {
        let nonce = self.nonce_seq.advance()?; // 自动生成唯一 nonce
        // ... 用 nonce 加密 ...
        Ok(())
    }
}
pub struct UnboundKey;
pub struct Aad<T>(T);
pub struct Unspecified;

所有权链防止：

  1. Nonce 重用 —— Nonce 不是 Clone，每次调用时被消耗
  2. 密钥复制 —— UnboundKey 被移动到 SealingKey，不能同时创建 OpeningKey
  3. 序列复制 —— NonceSequence 不是 Clone，所以没有两个密钥共享计数器

这些都不需要运行时检查。 编译器强制执行所有三个。

服务器平台有OTP（一次性可编程）熔丝用于安全密钥、 板序列号和功能位。写入熔丝是不可逆的 —— 用不同数据写入两次会使板子报废。这非常适合移动语义：

/// 带验证的构建器 —— 创建 FusePayload 的唯一方法。
pub struct FusePayloadBuilder {
    address: Option<u32>,
    data: Option<Vec<u8>>,
}

impl FusePayloadBuilder {
    pub fn new() -> Self {
        FusePayloadBuilder { address: None, data: None }
    }

    pub fn address(mut self, addr: u32) -> Self {
        self.address = Some(addr);
        self
    }

    pub fn data(mut self, data: Vec<u8>) -> Self {
        self.data = Some(data);
        self
    }

    pub fn build(self) -> Result<FusePayload, &'static str> {
        let address = self.address.ok_or("address required")?;
        let data = self.data.ok_or("data required")?;
        if data.len() > 32 { return Err("fuse data too long"); }
        Ok(FusePayload { address, data })
    }
}

}

某些传感器需要校准步骤，每个电源周期必须恰好发生一次。校准令牌强制执行这一点：

pub struct CalibrationToken {
    _private: (),
}

pub struct SensorController {
    calibrated: bool,
}

impl SensorController {
    /// 在开机时调用 —— 返回校准令牌。
    pub fn power_on() -> (Self, CalibrationToken) {
        (
            SensorController { calibrated: false },
            CalibrationToken { _private: () },
        )
    }

    /// 校准传感器 —— 消耗令牌。
    pub fn calibrate(&mut self, _token: CalibrationToken) -> io::Result<()> {
        // ... 运行校准序列 ...
        self.calibrated = true;
        Ok(())

    /// 读取传感器 —— 仅在校准后有意义。
    ///
    /// **限制：** 移动语义保证是*部分的*。调用者可以
    /// `drop(cal_token)` 而不调用 `calibrate()` —— 令牌将被
    /// 销毁但校准不会运行。`#[must_use]` 注解（见下文）
    /// 生成警告但不是硬错误。
    ///
    /// 这里的运行时 `self.calibrated` 检查是该间隙的**安全网**。

    pub fn read(&self) -> io::Result<f64> {
        if !self.calibrated {
            return Err(io::Error::new(io::ErrorKind::Other, "not calibrated"));
        }
        Ok(25.0) // 存根
    }
}

    let temp = ctrl.read()?;

    // 不能再校准 —— 令牌被消耗：
    // ctrl.calibrate(cal_token);  // ❌ 使用已移动的值

    Ok(())

    key_id: String,
    // 不是 Clone，不是 Copy
}

pub struct SignedImage {
    pub signature: Vec<u8>,
    pub key_id: String,
}

    pub fn issue(key_id: &str) -> Self {
        SigningToken { key_id: key_id.to_string() }
    }
}

关键要点

  1. 移动 = 线性使用 —— 非 Clone、非 Copy 类型只能被消耗恰好一次；编译器强制执行这一点。
  2. Nonce 重用是灾难性的 —— Rust 的所有权系统在结构上防止它，而不是靠纪律。
  3. 模式适用于加密之外 —— OTP 熔丝、校准令牌、审计条目 —— 任何最多发生一次的事情。
  4. 临时密钥免费获得前向保密性 —— 密钥协商值被移动到派生的共享密钥中并消失。
  5. 如有疑问，移除 Clone —— 你以后总是可以添加它；从已发布的 API 中移除它是破坏性变更。

零大小类型作为证明令牌

能力令牌是零大小类型（ZST），证明调用者有权执行操作。它在运行时零字节成本 —— 它只存在于类型系统中：

use std::marker::PhantomData;

/// 证明调用者有管理员权限。
/// 零大小 —— 完全编译掉。
/// 不是 Clone，不是 Copy —— 必须显式传递。
pub struct AdminToken {
    _private: (),   // 防止在此模块外构造
}

/// 证明 PCIe 链路已训练并准备好。
pub struct LinkTrainedToken {
    _private: (),
}

pub struct BmcController { /* ... */ }

impl BmcController {
    /// 认证为管理员 —— 返回能力令牌。
    /// 这是创建 AdminToken 的唯一方法。
    pub fn authenticate_admin(
        &mut self,
        credentials: &[u8],
    ) -> Result<AdminToken, &'static str> {
        // ... 验证凭据 ...
        let valid = true;
        if valid {
            Ok(AdminToken { _private: () })
        } else {
            Err("authentication failed")
        }
    }

    /// 训练 PCIe 链路 —— 返回已训练的证明。
    pub fn train_link(&mut self) -> Result<LinkTrainedToken, &'static str> {
        // ... 执行链路训练 ...
        Ok(LinkTrainedToken { _private: () })
    }

    /// 重置 PCIe 链路 —— 需要管理员 + 链路训练证明。
    /// 不需要运行时检查 —— 令牌就是证明。
    pub fn reset_pcie_link(
        &mut self,
        _admin: &AdminToken,         // 零成本权限证明
        _trained: &LinkTrainedToken,  // 零成本状态证明
        slot: u32,

        println!("Resetting PCIe link on slot {slot}");
        Ok(())
    }
}

使用 —— 类型系统强制执行工作流：

fn maintenance_workflow(bmc: &mut BmcController) -> Result<(), &'static str> {

问题：协议违规

硬件协议有严格的状态机。IPMI 会话有状态： 未认证 → 已认证 → 活跃 → 关闭。PCIe 链路训练经历 Detect → Polling → Configuration → L0。在错误状态下发送命令 会损坏会话或挂起总线。

IPMI 会话状态机：

stateDiagram-v2
    [*] --> Idle
    Idle --> Authenticated : authenticate(user, pass)
    Authenticated --> Active : activate_session()
    Active --> Active : send_command(cmd)
    Active --> Closed : close()
    Closed --> [*]

    note right of Active : send_command() 仅在此存在
    note right of Idle : send_command() → 编译错误

PCIe 链路训练状态机（LTSSM）：

stateDiagram-v2
    [*] --> Detect
    Detect --> Polling : receiver detected
    Polling --> Configuration : bit lock + symbol lock
    Configuration --> L0 : link number + lane assigned
    L0 --> L0 : send_tlp() / receive_tlp()
    L0 --> Recovery : error threshold
    Recovery --> L0 : retrained
    Recovery --> Detect : retraining failed

    note right of L0 : 仅在 L0 中发送 TLP

类型状态模式

使用类型状态，每个协议状态是不同的类型。转换是消耗一个状态并返回另一个状态的方法。编译器防止在错误状态下调用方法，因为那些方法在该类型上不存在。

use std::marker::PhantomData;

// 状态 —— 零大小标记类型
pub struct Idle;
# 案例研究：IPMI 会话生命周期

pub struct Authenticated;
pub struct Active;
pub struct Closed;

/// IPMI 会话参数化为其当前状态。
/// 状态仅存在于类型系统中（PhantomData 是零大小的）。
pub struct IpmiSession<State> {
    transport: String,     // 例如 "192.168.1.100"
    session_id: Option<u32>,
    _state: PhantomData<State>,
}

// 转换：Idle → Authenticated
impl IpmiSession<Idle> {
    pub fn new(host: &str) -> Self {
        IpmiSession {
            transport: host.to_string(),
            session_id: None,
            _state: PhantomData,
        }
    }

    pub fn authenticate(
        self,              // ← 消耗 Idle 会话
        user: &str,

    let link = link.detect_receiver()?;
    let link = link.poll_compliance()?;
    let mut link = link.negotiate(16, 5)?; // x16 Gen5

    // ✅ 现在我们可以发送 TLP：
    let _resp = link.send_tlp(&[0x00, 0x01, 0x02]);
    println!("Link: {}", link.link_info());

    // 恢复和重新训练：
    let recovery = link.enter_recovery();
    let mut link = recovery.retrain(4)?;  // 降级到 Gen4
    let _resp = link.send_tlp(&[0x03]);

    Ok(())
}

节拍 3：固件更新 —— 多阶段 FSM 与组合

固件更新生命周期比会话有更多状态，并与能力令牌和一次性类型（ch03）组合。这是本书中最复杂的类型状态示例 —— 如果你对它感到舒适，你就掌握了这个模式。

stateDiagram-v2
    [*] --> Idle
    Idle --> Uploading : begin_upload(admin, image)
    Uploading --> Verifying : finish_upload()
    Uploading --> Idle : abort()
    Verifying --> Verified : verify_ok()
    Verifying --> Idle : verify_fail()
    Verified --> Applying : apply(single-use VerifiedImage token)

use std::marker::PhantomData;

// ── 状态 ──
pub struct Idle;
pub struct Uploading;
pub struct Verifying;
pub struct Verified;
pub struct Applying;
pub struct WaitingReboot;

// ── 一次性证明镜像通过验证（ch03）──
pub struct VerifiedImage {
    _private: (),
    pub digest: [u8; 32],
}

// ── 能力令牌：只有管理员可以启动（ch04）──
pub struct FirmwareAdminToken { _private: () }

pub struct FwUpdate<S> {
    version: String,
    _state: PhantomData<S>,
}

impl FwUpdate<Idle> {
    pub fn new() -> Self {
        FwUpdate { version: String::new(), _state: PhantomData }
    }

    /// 开始上传 —— 需要管理员权限。
    pub fn begin_upload(
        self,
        _admin: &FirmwareAdminToken,
        version: &str,
    ) -> FwUpdate<Uploading> {
        println!("Uploading firmware v{version}...");
        FwUpdate { version: version.to_string(), _state: PhantomData }
    }
}

impl FwUpdate<Uploading> {
    pub fn finish_upload(self) -> FwUpdate<Verifying> {
        println!("Upload complete, verifying v{}...", self.version);
        FwUpdate { version: self.version, _state: PhantomData }
    }

    /// 中止返回 Idle —— 在上传过程中的任何点都安全。
    pub fn abort(self) -> FwUpdate<Idle> {
        println!("Upload aborted.");
        FwUpdate { version: String::new(), _state: PhantomData }
    }
}

impl FwUpdate<Verifying> {
    /// 成功时，生成一次性 VerifiedImage 令牌。
    pub fn verify_ok(self, digest: [u8; 32]) -> (FwUpdate<Verified>, VerifiedImage) {
        println!("Verification passed for v{}", self.version);
        (
            FwUpdate { version: self.version, _state: PhantomData },
            VerifiedImage { _private: (), digest },
        )
    }

    pub fn verify_fail(self) -> FwUpdate<Idle> {
        println!("Verification failed — returning to idle.");
        FwUpdate { version: String::new(), _state: PhantomData }
    }
}

impl FwUpdate<Verified> {
    /// 应用消耗 VerifiedImage 令牌 —— 不能应用两次。
    pub fn apply(self, proof: VerifiedImage) -> FwUpdate<Applying> {
        println!("Applying v{} (digest: {:02x?})", self.version, &proof.digest[..4]);
        // proof 被移动 —— 不能重用
        FwUpdate { version: self.version, _state: PhantomData }
    }
}

impl FwUpdate<Applying> {
    pub fn apply_complete(self) -> FwUpdate<WaitingReboot> {
        println!("Apply complete — waiting for reboot.");
        FwUpdate { version: self.version, _state: PhantomData }
    }
}

impl FwUpdate<WaitingReboot> {
    pub fn reboot(self) {
        println!("Rebooting into v{}...", self.version);
    }
}

// ── 使用 ──

fn firmware_workflow() {
    let fw = FwUpdate::new();

    // fw.finish_upload();  // ❌ FwUpdate<Idle> 上没有方法 `finish_upload`

    let admin = FirmwareAdminToken { _private: () }; // 来自认证系统
    let fw = fw.begin_upload(&admin, "2.10.1");
    let fw = fw.finish_upload();

    let digest = [0xAB; 32]; // 验证期间计算
    let (fw, token) = fw.verify_ok(digest);

    let fw = fw.apply(token);
    // fw.apply(token);  // ❌ 使用已移动的值：`token`

    let fw = fw.apply_complete();
    fw.reboot();
}

何时使用类型状态

协议类型状态值得？
IPMI 会话生命周期✅ 是 —— 认证 → 激活 → 命令 → 关闭

练习：USB 设备枚举类型状态

建模必须经历的 USB 设备：Attached → Powered → Default → Addressed → Configured。每个转换应消耗前一个状态并生成下一个状态。send_data() 应仅在 Configured 中可用。

关键要点

  1. 类型状态使错误顺序调用不可能 —— 方法仅存在于它们有效的状态上。
  2. 每个转换消耗 self —— 转换后不能保留旧状态。
  3. 与能力令牌结合 —— firmware_update() 需要两者 Session<Active> 和 AdminToken。
  4. 三个节拍，递增复杂度 —— IPMI（纯 FSM）、PCIe LTSSM（恢复分支）和固件更新（FSM + 令牌 + 一次性证明）显示模式从简单扩展到丰富组合。
  5. 不要过度应用 —— 两状态请求/响应协议没有类型状态更简单。
  6. 模式扩展到完整 Redfish 工作流 —— ch17 将类型状态应用于 Redfish 会话生命周期，ch18 使用构建器类型状态进行响应构造。

硬件量纲的宏

手写新类型变得重复。宏消除了样板代码：

/// 为物理量生成新类型。
macro_rules! quantity {
    ($Name:ident, $unit:expr) => {
        #[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
        pub struct $Name(pub f64);

        impl $Name {
            pub fn new(value: f64) -> Self { $Name(value) }
            pub fn value(self) -> f64 { self.0 }
        }

        impl std::fmt::Display for $Name {
            fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
                write!(f, "{:.2} {}", self.0, $unit)
            }
        }

        impl std::ops::Add for $Name {
            type Output = Self;
            fn add(self, rhs: Self) -> Self { $Name(self.0 + rhs.0) }
        }

        impl std::ops::Sub for $Name {
            type Output = Self;
            fn sub(self, rhs: Self) -> Self { $Name(self.0 - rhs.0) }
        }
    };
}

// 使用：
quantity!(Celsius, "°C");
quantity!(Fahrenheit, "°F");
quantity!(Volts, "V");
quantity!(Millivolts, "mV");
quantity!(Rpm, "RPM");
quantity!(Watts, "W");
quantity!(Amperes, "A");
quantity!(Pascals, "Pa");
quantity!(Hertz, "Hz");
quantity!(Bytes, "B");

每行生成一个完整的类型，带有 Display、Add、Sub 和比较运算符。 全部零运行时成本。

物理注意事项： 宏为所有量生成 Add，包括 Celsius。添加绝对温度（25°C + 30°C = 55°C）在物理上没有意义 —— 你需要单独的 TemperatureDelta 类型用于差值。uom crate（稍后展示）正确处理这个。对于只需要比较和显示的简单传感器诊断，你可以从温度类型中省略 Add/Sub，并将它们保留给加法有意义的量（Watts、Volts、Bytes）。如果你需要增量算术，定义一个 CelsiusDelta(f64) 新类型，带有 impl Add<CelsiusDelta> for Celsius。

#[derive(Debug, PartialEq)]
pub enum ThresholdResult {
    Normal,
    Warning,
    Critical,
}

impl<T: PartialOrd> Threshold<T> {
    pub fn check(&self, value: &T) -> ThresholdResult {
        if *value >= self.critical {
            ThresholdResult::Critical
        } else if *value >= self.warning {
            ThresholdResult::Warning
        } else {
            ThresholdResult::Normal

    type Output = Watts;

pub struct PowerBudget {
    total: Watts,
    limit: Watts,
}

impl PowerBudget {
    pub fn new(limit: Watts) -> Self {
        PowerBudget { total: Watts(0.0), limit }
    }
    pub fn add(&mut self, w: Watts) -> Result<(), String> {
        let new_total = Watts(self.total.0 + w.0);
        if new_total > self.limit {
            return Err(format!("budget exceeded: {:?} > {:?}", new_total, self.limit));
        }
        self.total = new_total;
        Ok(())
    }
}

关键要点

  1. 新类型以零成本防止单位混淆 —— Celsius 和 Rpm 内部都是 f64，但编译器将它们视为不同类型。

问题：散弹枪验证

在典型代码中，验证分散在各处。每个接收数据的函数都重新检查它"以防万一"：

// C —— 验证分散在代码库中
int process_fru_data(uint8_t *data, int len) {
    if (data == NULL) return -1;          // 检查：非空
    if (len < 8) return -1;              // 检查：最小长度
    if (data[0] != 0x01) return -1;      // 检查：格式版本
    if (checksum(data, len) != 0) return -1; // 检查：校验和

    // ... 10 多个重复相同检查的函数 ...
}

这种模式（"散弹枪验证"）有两个问题：

  1. 冗余 —— 相同的检查出现在数十个地方
  2. 不完整 —— 在一个地方忘记一个检查，你就有了 bug

解析，不要验证

正确构造方法：在边界验证一次，然后在类型中携带有效性证明。

案例研究：IPMI FRU 数据

#[derive(Debug)]
pub struct RawFruData(Vec<u8>);

/// 验证的 IPMI FRU 数据。只能通过 TryFrom 创建，
/// 它强制执行所有不变量。一旦你有了 ValidFru，
/// 所有数据都保证正确。
#[derive(Debug)]
pub struct ValidFru {
    format_version: u8,
    internal_area_offset: u8,
    chassis_area_offset: u8,
    board_area_offset: u8,
    product_area_offset: u8,
    data: Vec<u8>,
}

#[derive(Debug)]
pub enum FruError {
    TooShort { actual: usize, minimum: usize },
    BadFormatVersion(u8),
    ChecksumMismatch { expected: u8, actual: u8 },
    InvalidAreaOffset { area: &'static str, offset: u8 },
}

impl TryFrom<RawFruData> for ValidFru {
    type Error = FruError;

    fn try_from(raw: RawFruData) -> Result<Self, FruError> {
        let data = raw.0;

        if data.len() < 8 {
            return Err(FruError::TooShort { actual: data.len(), minimum: 8 });
        }
        if data[0] != 0x01 {
            return Err(FruError::BadFormatVersion(data[0]));
        }

        let checksum: u8 = data[..8].iter().fold(0u8, |acc, &b| acc.wrapping_add(b));
        if checksum != 0 {
            return Err(FruError::ChecksumMismatch { expected: 0, actual: checksum });
        }

        for (name, idx) in [("internal", 1), ("chassis", 2), ("board", 3), ("product", 4)] {
            let offset = data[idx];
            if offset != 0 && (offset as usize * 8) >= data.len() {
                return Err(FruError::InvalidAreaOffset { area: name, offset });
            }
        }

        Ok(ValidFru {
            format_version: data[0],
            internal_area_offset: data[1],
            chassis_area_offset: data[2],
            board_area_offset: data[3],
            product_area_offset: data[4],
            data,
        })
    }
}

impl ValidFru {
    pub fn board_area(&self) -> Option<&[u8]> {
        if self.board_area_offset == 0 { return None; }
        let start = self.board_area_offset as usize * 8;
        Some(&self.data[start..])
    }
}

任何接受 &ValidFru 的函数都知道数据格式良好。无需重新检查。

问题：诊断代码重复

服务器平台在子系统间共享诊断模式。风扇诊断、温度监控和电源排序都遵循类似的工作流，但操作不同的硬件总线。没有抽象，你就会得到复制粘贴：

// C —— 在子系统间复制逻辑
int run_fan_diag(spi_bus_t *spi, i2c_bus_t *i2c) {
    // ... 50 行 SPI 传感器读取 ...
    // ... 30 行 I2C 寄存器检查 ...
    // ... 20 行阈值比较（与 CPU 诊断相同）...
}

int run_cpu_temp_diag(i2c_bus_t *i2c, gpio_t *gpio) {
    // ... 30 行 I2C 寄存器检查（与风扇诊断相同）...
    // ... 15 行 GPIO 警报检查 ...
    // ... 20 行阈值比较（与风扇诊断相同）...
}

阈值比较逻辑相同，但你无法提取它，因为总线类型不同。使用能力混入，每个硬件总线是成分 trait，诊断行为在具备正确成分时自动提供。

成分 Trait（硬件能力）

每个总线或外设是 trait 上的关联类型。诊断控制器声明它有哪些总线：

pub trait HasSpi {
    type Spi: SpiBus;

}

/// I2C 总线能力。

    type I2c: I2cBus;
    fn i2c(&self) -> &Self::I2c;
}

/// GPIO 引脚访问能力。

    type Gpio: GpioController;

/// IPMI 访问能力。

混入 Trait（诊断行为）

混入为任何具有所需能力的类型自动提供行为：

/// 风扇诊断混入 —— 为任何具有 SPI + I2C 的类型自动实现。
pub trait FanDiagMixin: HasSpi + HasI2c {
    fn read_fan_speed(&self, fan_id: u8) -> u32 {
        // 通过 SPI 读取转速计
        let cmd = [0x80 | fan_id, 0x00];
        let response = self.spi().transfer(&cmd);
        u32::from_be_bytes([0, 0, response[0], response[1]])
    }

    fn set_fan_pwm(&self, fan_id: u8, duty_percent: u8) {
        // 通过 I2C 控制器设置 PWM
        self.i2c().write_register(0x2E, fan_id, duty_percent);
    }

    fn run_fan_diagnostic(&self) -> bool {
        // 完整诊断：读取所有风扇，检查阈值
        for fan_id in 0..6 {
            let speed = self.read_fan_speed(fan_id);
            if speed < 1000 || speed > 20000 {
                println!("Fan {fan_id}: FAIL ({speed} RPM)");
                return false;
            }
        }
        true
    }
}

// Blanket 实现 —— 任何具有 SPI + I2C 的类型免费获得 FanDiagMixin
impl<T: HasSpi + HasI2c> FanDiagMixin for T {}

/// 温度监控混入 —— 需要 I2C + GPIO。
pub trait TempMonitorMixin: HasI2c + HasGpio {
    fn read_temperature(&self, sensor_addr: u8) -> f64 {
        let raw = self.i2c().read_register(sensor_addr, 0x00);
        raw as f64 * 0.5  // 每 LSB 0.5°C
    }

    fn check_thermal_alert(&self, alert_pin: u32) -> bool {
        self.gpio().read_pin(alert_pin)
    }

    fn run_thermal_diagnostic(&self) -> bool {
        for addr in [0x48, 0x49, 0x4A] {
            let temp = self.read_temperature(addr);
            if temp > 95.0 {
                println!("Sensor 0x{addr:02X}: CRITICAL ({temp}°C)");
                return false;
            }
            if self.check_thermal_alert(addr as u32) {
                println!("Sensor 0x{addr:02X}: ALERT pin asserted");
                return false;
            }
        }
        true
    }
}

impl<T: HasI2c + HasGpio> TempMonitorMixin for T {}

正确构造方面

混入模式是正确构造的，因为：

  1. 没有 SPI 就不能调用 read_fan_speed() —— 方法仅存在于实现 HasSpi + HasI2c 的类型上
  2. 你不能忘记总线 —— 如果从 BaseBoardController 中移除 HasSpi，FanDiagMixin 方法在编译时消失
  3. 模拟测试是自动的 —— 用 MockSpi 替换 LinuxSpi，所有混入逻辑与模拟一起工作
  4. 新平台只需声明能力 —— 只有 I2C 的 GPU 子卡获得 TempMonitorMixin（如果它也有 GPIO），但没有 FanDiagMixin（没有 SPI）

练习：网络诊断混入

为网络诊断设计混入系统：

  • 成分 traits：HasEthernet、HasIpmi
  • 混入：LinkHealthMixin（需要 HasEthernet）带有 check_link_status(&self)
  • 混入：RemoteDiagMixin（需要 HasEthernet + HasIpmi）带有 remote_health_check(&self)
  • 具体类型：实现两个成分的 NicController。
解决方案 
pub trait HasEthernet {
    fn eth_link_up(&self) -> bool;
}

pub trait HasIpmi {
    fn ipmi_ping(&self) -> bool;
}

pub trait LinkHealthMixin: HasEthernet {
    fn check_link_status(&self) -> &'static str {
        if self.eth_link_up() { "link: UP" } else { "link: DOWN" }
    }
}
impl<T: HasEthernet> LinkHealthMixin for T {}

pub trait RemoteDiagMixin: HasEthernet + HasIpmi {
    fn remote_health_check(&self) -> &'static str {
        if self.eth_link_up() && self.ipmi_ping() {
            "remote: HEALTHY"
        } else {
            "remote: DEGRADED"
        }
    }
}
impl<T: HasEthernet + HasIpmi> RemoteDiagMixin for T {}

pub struct NicController;
impl HasEthernet for NicController {
    fn eth_link_up(&self) -> bool { true }
}
impl HasIpmi for NicController {
    fn ipmi_ping(&self) -> bool { true }
}
// NicController 自动获得两个混入方法

关键要点

  1. 成分 trait 声明硬件能力 —— HasSpi、HasI2c、HasGpio 是关联类型 trait。
  2. 混入 trait 通过 blanket impl 提供行为 —— impl<T: HasSpi + HasI2c> FanDiagMixin for T {}。
  3. 添加新平台 = 列出其能力 —— 编译器提供所有匹配的混入方法。
  4. 移除总线 = 使用它的地方出现编译错误 —— 你不能忘记更新下游代码。
  5. 模拟测试是免费的 —— 用 MockSpi 交换 LinuxSpi；所有混入逻辑不变地工作。

问题：混淆资源

文件描述符所有权

    pub fn open(path: &str) -> Result<Self, String> {
        Ok(Fd { raw: 3, _state: PhantomData })

    pub fn write(&self, data: &[u8]) -> Result<usize, String> { Ok(data.len()) }
    pub fn close(self) -> Fd<Closed> {

    pub fn raw_fd(&self) -> i32 { self.raw }

练习：内存区域权限

为具有读取、写入和执行权限的内存区域设计幻象类型：

  • MemRegion<ReadOnly> 有 fn read(&self, offset: usize) -> u8
  • MemRegion<ReadWrite> 有 read 和 write
  • MemRegion<Executable> 有 read 和 fn execute(&self)
  • 写入 ReadOnly 或执行 ReadWrite 不应编译。
解决方案 
use std::marker::PhantomData;

pub struct ReadOnly;
pub struct ReadWrite;
pub struct Executable;

pub struct MemRegion<Perm> {
    base: *mut u8,
    len: usize,
    _perm: PhantomData<Perm>,
}

// 所有权限类型都可读取
impl<P> MemRegion<P> {
    pub fn read(&self, offset: usize) -> u8 {
        assert!(offset < self.len);
        unsafe { *self.base.add(offset) }
    }
}

impl MemRegion<ReadWrite> {
    pub fn write(&mut self, offset: usize, val: u8) {
        assert!(offset < self.len);
        unsafe { *self.base.add(offset) = val; }
    }
}

impl MemRegion<Executable> {
    pub fn execute(&self) {}
}

// ❌ region_ro.write(0, 0xFF);  // 编译错误：没有方法 `write`
// ❌ region_rw.execute();       // 编译错误：没有方法 `execute`

关键要点

  1. PhantomData 以零大小携带类型级信息 —— 标记仅存在于编译器中。
  2. 寄存器宽度不匹配成为编译错误 —— Register<Width16> 返回 u16，不是 u32。
  3. DMA 方向在结构上强制执行 —— DmaBuffer<Read> 没有 write() 方法。
  4. 与量纲类型结合（ch06） —— Register<Width16> 可以通过解析步骤返回 Celsius。
  5. 幻象类型仅编译时 —— 它们不适用于运行时可变属性；对那些使用枚举。

问题：撒谎的内存映射

在嵌入式和系统编程中，内存映射是一切的基础 —— 它们定义引导加载程序、固件、数据段和栈的位置。边界错误，两个子系统静默相互损坏。在 C 中，这些映射通常是 #define 常量，没有结构关系：

/* STM32F4 SRAM 布局 —— 256 KB 在 0x20000000 */
#define SRAM_BASE       0x20000000
#define SRAM_SIZE       (256 * 1024)
#define BOOT_BASE       0x20000000
#define BOOT_SIZE       (16 * 1024)
#define FW_BASE         0x20004000
#define FW_SIZE         (128 * 1024)
#define DATA_BASE       0x20024000
#define DATA_SIZE       (80 * 1024)     /* 某人将它从 64K 增加到 80K */
#define STACK_BASE      0x20038000
#define STACK_SIZE      (48 * 1024)     /* 0x20038000 + 48K = 0x20044000 —— 超出 SRAM 结尾！ */

Bug：16 + 128 + 80 + 48 = 272 KB，但 SRAM 只有 256 KB。栈超出物理内存结尾 16 KB。没有编译器警告，没有链接器错误，没有运行时检查 —— 只是当栈增长到未映射空间时的静默损坏。

Const Fn：将编译器变成证明引擎

Rust 的 const fn 函数可以在编译时运行。当 const fn 在编译时 panic 时，panic 变成编译错误。与 assert! 结合，这将编译器变成你的不变量的定理证明器：

#![allow(unused)]
fn main() {
pub const fn checked_add(a: u32, b: u32) -> u32 {
    let sum = a as u64 + b as u64;
    assert!(sum <= u32::MAX as u64, "overflow");
    sum as u32
}

// ✅ 编译 —— 100 + 200 适合 u32
const X: u32 = checked_add(100, 200);

// ❌ 编译错误："overflow"
// const Y: u32 = checked_add(u32::MAX, 1);
}

关键见解： const fn + assert! = 证明义务。每个断言是编译器必须验证的定理。如果证明失败，程序不编译。不需要测试套件，不需要代码审查捕获 —— 编译器本身是审计员。

构建验证的 SRAM 内存映射

区域类型

Region 表示连续内存块。其构造函数是 const fn，强制执行基本有效性：

#![allow(unused)]
fn main() {
#[derive(Debug, Clone, Copy)]
pub struct Region { pub base: u32, pub size: u32 }

impl Region {
    pub const fn new(base: u32, size: u32) -> Self {
        assert!(size > 0, "region size must be non-zero");
        assert!(base as u64 + size as u64 <= u32::MAX as u64, "overflow");
        Self { base, size }
    }
    pub const fn end(&self) -> u32 { self.base + self.size }
    pub const fn contains(&self, inner: &Region) -> bool {
        inner.base >= self.base && inner.end() <= self.end()
    }
    pub const fn overlaps(&self, other: &Region) -> bool {
        self.base < other.end() && other.base < self.end()
    }
}

const R: Region = Region::new(0x2000_0000, 1024);
}

        assert!(shift as u32 + width as u32 <= 32, "bit field exceeds 32-bit register");
        let mask = ((1u64 << width as u64) - 1) as u32;
        Self { mask: mask << shift as u32, shift }
    }
}

const fn fields_disjoint(a: &BitField, b: &BitField) -> bool {
    a.mask & b.mask == 0
}

const SPI_EN:     BitField = BitField::new(0, 1);
const SPI_MODE:   BitField = BitField::new(1, 2);
const SPI_CLKDIV: BitField = BitField::new(4, 4);

const _: () = {
    assert!(fields_disjoint(&SPI_EN, &SPI_MODE));
    assert!(fields_disjoint(&SPI_EN, &SPI_CLKDIV));
    assert!(fields_disjoint(&SPI_MODE, &SPI_CLKDIV));
};
}

Send & Sync —— 编译时并发证明 🟠

你将学到： Rust 的 Send 和 Sync auto-traits 如何将编译器变成并发审计员 —— 在编译时证明哪些类型可以跨线程边界以及哪些可以共享，零运行时成本。

交叉引用： ch04（能力令牌）、ch09（幻象类型）、ch15（const fn 证明）

问题：没有安全网的并发访问

在系统编程中，外设、共享缓冲区和全局状态从多个上下文访问 —— 主循环、中断处理程序、DMA 回调和工作线程。在 C 中，编译器不提供任何强制执行：

/* 共享传感器缓冲区 —— 从主循环和 ISR 访问 */
volatile uint32_t sensor_buf[64];
volatile uint32_t buf_index = 0;

void SENSOR_IRQHandler(void) {
    sensor_buf[buf_index++] = read_sensor();  /* 竞争：buf_index 读 + 写 */
}

void process_sensors(void) {
    for (uint32_t i = 0; i < buf_index; i++) {  /* buf_index 在循环中间改变 */
        process(sensor_buf[i]);                   /* 数据在读取中间被覆盖 */
    }
    buf_index = 0;                                /* ISR 在这几行之间触发 */
}

volatile 关键字防止编译器优化掉读取，但对数据竞争毫无作用。两个上下文可以同时读写 buf_index，产生撕裂值、丢失更新或缓冲区溢出。pthread_mutex_t 也出现同样的问题 —— 编译器会愉快地让你忘记锁定：

pthread_mutex_t lock;
int shared_counter;

void increment(void) {
    shared_counter++;  /* 哎呀 —— 忘记 pthread_mutex_lock(&lock) */
}

每个并发 bug 都在运行时发现 —— 通常在负载下、在生产中、间歇性地。

Send 和 Sync 证明什么

Rust 定义两个编译器自动派生的 marker traits：

Trait证明非正式含义
SendT 类型的值可以安全地移动到另一个线程"这可以跨线程边界"
Sync共享引用 &T 可以安全地被多个线程使用"这可以跨线程读取"

这些是auto-traits —— 编译器通过检查每个字段派生它们。如果所有字段都是 Send，结构体就是 Send。如果任何字段选择退出，整个结构体选择退出。不需要注解，没有运行时开销 —— 证明是结构的。

编译器是审计员。 在 C 中，线程安全注解存在于注释和头文件文档中 —— 建议性的，从不强制执行。在 Rust 中，Send 和 Sync 从类型本身的结构派生。添加单个 Cell<f32> 字段自动使包含的结构体 !Sync。不需要程序员操作，无法忘记。

两个 trait 通过关键恒等式链接：

T 是 Sync 当且仅当 &T 是 Send。

这直观上有意义：如果共享引用可以安全地发送到另一个线程，那么底层类型对并发读取是安全的。

选择退出的类型

某些类型故意是 !Send 或 !Sync：

类型SendSync为什么
u32、String、Vec<T>✅✅无内部可变性，无原始指针
Cell<T>、RefCell<T>✅❌无同步的内部可变性
Rc<T>❌❌引用计数不是原子的
*const T、*mut T❌❌原始指针没有安全保证
Arc<T>（其中 T: Send + Sync）✅✅原子引用计数
Mutex<T>（其中 T: Send）✅✅锁定序列化所有访问

此表中的每个 ❌ 是编译时不变量。你不能意外发送 Rc 到另一个线程 —— 编译器拒绝它。

!Send 外设句柄

在嵌入式系统中，外设寄存器块位于固定内存地址，应仅从单个执行上下文访问。原始指针天然是 !Send 和 !Sync，所以包装一个自动使包含类型退出两个 trait：

/// 内存映射 UART 外设的句柄。
/// 原始指针使这个自动 !Send 和 !Sync。
pub struct Uart { regs: *const u32 }

impl Uart {
    pub fn new(base: usize) -> Self { Self { regs: base as *const u32 } }
    pub fn write_byte(&self, byte: u8) { /* ... */ }
}

fn main() {
    let uart = Uart::new(0x4000_1000);
    uart.write_byte(b'A');  // ✅ 在创建线程上使用
    // ❌ 不会编译：Uart 是 !Send
    // std::thread::spawn(move || { uart.write_byte(b'B'); });
}

注释掉的 thread::spawn 将产生：

error[E0277]: `*const u32` cannot be sent between threads safely

没有原始指针？使用 PhantomData。 有时类型没有原始指针但仍应限制在一个线程 —— 例如文件描述符索引或从 C 库获得的句柄：

#![allow(unused)]
fn main() {
use std::marker::PhantomData;

/// 来自 C 库的不透明句柄。PhantomData<*const ()> 使它 !Send + !Sync
/// 即使内部 fd 只是普通整数。
pub struct LibHandle {
    fd: i32,
    _not_send: PhantomData<*const ()>,
}
}

Mutex 将 !Sync 转换为 Sync

Cell<T> 和 RefCell<T> 提供内部可变性而没有任何同步 —— 所以它们是 !Sync。但有时你确实需要跨线程共享可变状态。Mutex<T> 添加缺少的同步，编译器识别这个：

如果 T: Send，那么 Mutex<T>: Send + Sync。

锁定序列化所有访问，所以 !Sync 内部类型变得安全共享。编译器在结构上证明这一点 —— 没有"程序员忘记锁定"的运行时检查：

use std::sync::{Arc, Mutex};
use std::cell::Cell;

struct SensorCache {
    last_reading: Cell<u32>,
    reading_count: Cell<u32>,
}

fn main() {
    // Mutex 使 SensorCache 安全共享 —— 编译器证明它
    let cache = Arc::new(Mutex::new(SensorCache {
        last_reading: Cell::new(0),
        reading_count: Cell::new(0),
    }));

    let handles: Vec<_> = (0..4).map(|i| {
        let c = Arc::clone(&cache);
        std::thread::spawn(move || {
            let guard = c.lock().unwrap();  // 访问前必须锁定
            guard.last_reading.set(i * 10);
            guard.reading_count.set(guard.reading_count.get() + 1);
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
}

与 C 版本比较：pthread_mutex_lock 是程序员可能忘记的运行时调用。这里，类型系统使不通过锁定访问 SensorCache 不可能。证明是结构的 —— 唯一运行时成本是锁定本身。

Mutex 不仅同步 —— 它证明同步。 Mutex::lock() 返回 MutexGuard，它 Deref 到 &T。没有办法获得内部数据的引用而不通过锁定。API 使"忘记锁定"在结构上不可表示。

作为定理的函数边界

std::thread::spawn 有这个签名：

pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,

Send + 'static 边界不仅是实现细节 —— 它是定理：

"任何传递给 spawn 的闭包和返回值在编译时被证明安全地在另一个线程上运行，没有悬垂引用。"

你可以将相同模式应用到你自己的 API：

use std::sync::mpsc;

fn run_on_background<F, T>(task: F) -> T
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,
{
    let (tx, rx) = mpsc::channel();
    std::thread::spawn(move || { let _ = tx.send(task()); });
    rx.recv().expect("background task panicked")
}

fn main() {
    // ✅ u32 是 Send，闭包不捕获非 Send 内容
    let result = run_on_background(|| 6 * 7);
    // ✅ String 是 Send
    let greeting = run_on_background(|| String::from("hello from background"));
    // ❌ 不会编译：Rc 是 !Send
    // use std::rc::Rc;
    // let data = Rc::new(42);
    // run_on_background(move || *data);
}

取消注释 Rc 示例产生精确诊断：

error[E0277]: `Rc<i32>` cannot be sent between threads safely

编译器跟踪违反回到确切边界 —— 并告诉程序员为什么。与 C 的 pthread_create 比较：

int pthread_create(pthread_t *thread, const pthread_attr_t *attr,
                   void *(*start_routine)(void *), void *arg);

void *arg 接受任何内容 —— 线程安全或不是。C 编译器不能区分非原子引用计数和普通整数。Rust 的 trait 边界在类型级别进行区分。

练习：DMA 传输保护

设计 DmaTransfer<T>，在 DMA 传输进行中时持有缓冲区。要求：

  1. DmaTransfer 必须是 !Send —— DMA 控制器使用绑定到此核心内存总线的物理地址
  2. DmaTransfer 必须是 !Sync —— DMA 写入时并发读取会看到撕裂数据
  3. 提供 wait() 方法消耗保护并返回缓冲区 —— 所有权证明传输完成
  4. 缓冲区类型 T 必须实现 DmaSafe marker trait
解决方案 
use std::marker::PhantomData;

trait DmaSafe {}
impl DmaSafe for [u8; 64] {}

pub struct DmaTransfer<T: DmaSafe> {
    buffer: T,
    channel: u8,
    _no_send_sync: PhantomData<*const ()>,
}

impl<T: DmaSafe> DmaTransfer<T> {
    pub fn start(buffer: T, channel: u8) -> Self {
        println!("DMA channel {} started", channel);
        Self { buffer, channel, _no_send_sync: PhantomData }
    }
    pub fn wait(self) -> T {
        println!("DMA channel {} complete", self.channel);
        self.buffer
    }
}

fn main() {
    let buf = [0u8; 64];
    let transfer = DmaTransfer::start(buf, 2);
    // ❌ buf 不再可访问 —— 所有权防止 DMA 期间使用
    // ❌ 不会编译：DmaTransfer 是 !Send
    // std::thread::spawn(move || { transfer.wait(); });
    let buf = transfer.wait();  // ✅ 在原始线程上等待，取回缓冲区
}

关键要点

  1. Send 和 Sync 是关于并发安全的编译时证明 —— 编译器通过检查每个字段在结构上派生它们。
  2. 原始指针自动选择退出 —— 任何包含 *const T 或 *mut T 的类型变成 !Send + !Sync。
  3. PhantomData<*const ()> 是显式选择退出 —— 当类型没有原始指针但仍应是线程限制时使用。
  4. Mutex<T> 恢复 Sync 并证明 —— 编译器在结构上证明所有访问通过锁定。
  5. 函数边界是定理 —— 签名中的 F: Send + 'static 是编译时证明义务。
  6. 模式补充所有其他正确性技术 —— 类型状态证明协议排序，幻象类型证明权限，const fn 证明值不变量，Send/Sync 证明并发安全。

pub trait IpmiCmd {
    type Response;
    fn net_fn(&self) -> u8;
    fn cmd_byte(&self) -> u8;
    fn payload(&self) -> Vec<u8>;
    fn parse_response(&self, raw: &[u8]) -> io::Result<Self::Response>;
}

pub struct ReadTemp { pub sensor_id: u8 }
impl IpmiCmd for ReadTemp {
    type Response = Celsius;
    fn net_fn(&self) -> u8 { 0x04 }
    fn cmd_byte(&self) -> u8 { 0x2D }
    fn payload(&self) -> Vec<u8> { vec![self.sensor_id] }
    fn parse_response(&self, raw: &[u8]) -> io::Result<Celsius> {
        if raw.is_empty() { return Err(io::Error::new(io::ErrorKind::InvalidData, "empty")); }
        Ok(Celsius(raw[0] as f64))
    }
}

pub struct ReadFanSpeed { pub fan_id: u8 }
impl IpmiCmd for ReadFanSpeed {
    type Response = Rpm;
    fn net_fn(&self) -> u8 { 0x04 }
    fn cmd_byte(&self) -> u8 { 0x2D }
    fn payload(&self) -> Vec<u8> { vec![self.fan_id] }
    fn parse_response(&self, raw: &[u8]) -> io::Result<Rpm> {
        if raw.len() < 2 { return Err(io::Error::new(io::ErrorKind::InvalidData, "need 2 bytes")); }
        Ok(Rpm(u16::from_le_bytes([raw[0], raw[1]]) as f64))
    }
}

    let fan: Rpm = session.execute(&ReadFanSpeed { fan_id: 1 }).map_err(|e| e.to_string())?;

    // 4. 读取幻象类型 PCIe 寄存器
    let pcie = PcieDev::new();
    let vid: u16 = pcie.vendor_id.read();  // 保证 u16

    // 5. 在边界验证 FRU 数据
    let raw_fru = vec![0x01, 0x00, 0x00, 0x01, 0x01, 0x00, 0x00, 0xFD];
    let fru = ValidFru::parse(&raw_fru).map_err(|e| e.to_string())?;

    // 6. 发行一次性审计令牌
    let audit = AuditToken::issue(1001);

    // 7. 生成报告（所有数据都是类型化和验证的）
    let report = format!(
        "Server: {} (SN: {}), VID: 0x{:04X}, CPU: {:?}, Fan: {:?}",
        fru.product_name, fru.board_serial, vid, temp, fan,
    );

    // 8. 消耗审计令牌 —— 不能记录两次
    audit.log(&report);
    // audit.log("oops");  // ❌ 使用已移动的值

    // 9. 关闭会话（类型状态：Active → dropped）
    session.close();

    Ok(())
}

第 10 章围绕 IPMI 组合核心模式 —— 字节级协议。但大多数 BMC 平台现在 alongside（或代替）IPMI 暴露 Redfish REST API，Redfish 引入它自己的正确性危险类别：

第 1 节 —— 会话生命周期（类型状态，ch05）

Redfish 会话有严格的生命周期：connect → authenticate → use → close。将每个状态编码为不同型号。

    Disconnected --> Connected : connect(host)

    Authenticated --> Authenticated : get() / patch() / post_action()
    Authenticated --> Closed : logout()

use std::marker::PhantomData;

pub struct ComputerSystemBuilder<Name, Uuid, PowerState, Status> {

    uuid: Option<String>,
    power_state: Option<PowerStateValue>,
    status: Option<ResourceStatus>,
    // 可选字段 —— 不跟踪（总是可设置）

    model: Option<String>,
    serial_number: Option<String>,
    bios_version: Option<String>,
    _markers: PhantomData<(Name, Uuid, PowerState, Status)>,
}

#[derive(Debug, Clone, serde::Serialize)]
pub enum PowerStateValue { On, Off, PoweringOn, PoweringOff }

#[derive(Debug, Clone, serde::Serialize)]
pub struct ResourceStatus {
    #[serde(rename = "State")] pub state: StatusState,
    #[serde(rename = "Health")] pub health: HealthValue,
}

#[derive(Debug, Clone, Copy, serde::Serialize)]
pub enum StatusState { Enabled, Disabled, Absent }

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, serde::Serialize)]
pub enum HealthValue { OK, Warning, Critical }

impl ComputerSystemBuilder<MissingField, MissingField, MissingField, MissingField> {
    pub fn new() -> Self {
        ComputerSystemBuilder {
            name: None, uuid: None, power_state: None, status: None,

消除的 bug 类： 模式非合规响应。处理程序物理上不能序列化 ComputerSystem 而不提供每个必需字段。编译器错误消息甚至告诉你哪个字段缺失 —— 它就在类型参数中：Name 位置中的 MissingField。

第 2 节 —— 源可用性令牌（能力令牌，ch04 —— 新转折）

在 ch04 和 ch17 中，能力令牌证明授权 —— "调用者被允许做这个。"在服务器端，相同模式证明可用性 —— "此数据源成功初始化。"

BMC 查询的每个子系统可以独立失败。SMBIOS 表可能损坏。传感器子系统可能仍在初始化。PCIe 总线扫描可能超时。将每个编码为证明令牌：

/// 证明 IPMI 传感器子系统响应。

/// 证明 PCIe 总线扫描完成。
pub struct PcieReady { _private: () }

pub struct SmbiosTables {
    pub product_name: String, pub manufacturer: String,
    pub serial_number: String, pub uuid: String,

pub struct SensorCache {
    pub cpu_temp: Celsius, pub inlet_temp: Celsius,
    pub fan_readings: Vec<(String, Rpm)>,

}

    // 读取 SMBIOS 入口点，解析表...
    // 如果表不存在或损坏则返回 None

}

pub fn init_sensors() -> Option<(SensorsReady, SensorCache)> {
    // 初始化 SDR 存储库，读取所有传感器...

现在，从数据源填充构建器字段的函数需要相应的证明令牌：

/// 从 SMBIOS 填充字段。需要证明 SMBIOS 可用。
fn populate_from_smbios<P, S>(
    builder: ComputerSystemBuilder<MissingField, MissingField, P, S>,
    _proof: &SmbiosReady,
    tables: &SmbiosTables,

    builder.name(tables.product_name.clone()).uuid(tables.uuid.clone())
}

/// 当 SMBIOS 不可用时的回退 —— 用安全默认值提供必需字段。
fn populate_smbios_fallback<P, S>(
    builder: ComputerSystemBuilder<MissingField, MissingField, P, S>,

    builder.name("Unknown System".into()).uuid("00000000-0000-0000-0000-000000000000".into())
}

处理程序根据哪些令牌可用来选择路径：

fn build_computer_system(

    health: ResourceStatus,

第 3 节 —— 序列化边界的量纲类型（ch06）

在客户端（ch17 §4），量纲类型防止读取 °C 作为 RPM。在服务器端，它们防止写入 RPM 到 Celsius JSON 字段。这可能更危险 —— 服务器上的错误值传播到每个客户端。

use serde::Serialize;

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd, Serialize)]
pub struct Watts(pub f64);

#[derive(Serialize)]

    pub member_id: String,

    pub reading_celsius: Celsius,           // ← 必须是 Celsius

    pub status: ResourceStatus,
}

pub struct FanMember {
    pub member_id: String,

fn build_thermal_response(_proof: &SensorsReady, cache: &SensorCache) -> serde_json::Value {
    let temps = vec![
        TemperatureMember {
            member_id: "0".into(), name: "CPU Temp".into(),

            upper_threshold_critical: Some(Celsius(95.0)),
            status: ResourceStatus { state: StatusState::Enabled, health: HealthValue::OK },
        },
        // ❌ 编译错误 —— 不能将 Rpm 放入 Celsius 字段：

第 4 节 —— 健康汇总（来自 ch07 SEL 的消费者管道）

来自 ch07 的 TypedSelSummary 馈送到 Redfish 健康汇总：

/// 从每个子系统健康计算整体系统健康。
/// 使用 Ord 实例保证正确性：Critical > Warning > OK。
fn compute_system_health(
    sensor_cache: Option<&(SensorsReady, SensorCache)>,
    sel_summary: Option<&(SelReady, TypedSelSummary)>,
) -> ResourceStatus {
    let mut health = HealthValue::OK;

    // 来自传感器的健康
    if let Some((_, cache)) = sensor_cache {
        if cache.cpu_temp > Celsius(95.0) { health = HealthValue::Critical; }
        if cache.inlet_temp > Celsius(42.0) { health = health.max(HealthValue::Warning); }
    }

    // 来自 SEL 的健康（来自 ch07 的消费者管道）
    if let Some((_, summary)) = sel_summary {
        health = health.max(summary.processor_health);
        health = health.max(summary.memory_health);
        health = health.max(summary.power_health);
        health = health.max(summary.thermal_health);
        health = health.max(summary.fan_health);
    }

    ResourceStatus {
        state: StatusState::Enabled,
        health,
        health_rollup: Some(health),
    }
}

消除的 bug 类：

  • 忘记子系统 —— 每个子系统显式汇总。
  • 错误严重性 —— HealthValue::max() 保证正确汇总（Critical > Warning > OK）。
  • 来自 ch07 的类型化 SEL —— TypedSelSummary 携带每个子系统健康，不是损失性的 bool。

技巧 1 —— 边界处的哨兵 → Option

硬件协议充满哨兵值：IPMI 使用 0xFF 表示"传感器不存在"，PCI 使用 0xFFFF 表示"无设备"，SMBIOS 使用 0x00 表示"未知"。如果你将这些哨兵作为普通整数携带在代码中，每个消费者必须记住检查魔法值。如果即使一个比较忘记，你就会得到幻影 255°C 读数或虚假的供应商 ID 匹配。

规则： 在第一个解析边界将哨兵转换为 Option，并仅在序列化边界转换回哨兵。

反模式（来自 pcie_tree/src/lspci.rs）

// 哨兵在内部携带 —— 每个比较必须记住
let mut current_vendor_id: u16 = 0xFFFF;
let mut current_device_id: u16 = 0xFFFF;

正确模式（来自 nic_sel/src/events.rs）

pub struct ThermalEvent {
    pub record_id: u16,
    pub temperature: Option<u8>,  // 如果传感器报告 0xFF 则为 None
}

impl ThermalEvent {
    pub fn from_raw(record_id: u16, raw_temp: u8) -> Self {
        ThermalEvent {
            record_id,
            temperature: if raw_temp != 0xFF { Some(raw_temp) } else { None },
        }
    }
}

现在每个消费者必须处理 None 情况 —— 编译器强制它：

fn is_overtemp(temp: Option<u8>, threshold: u8) -> bool {
    temp.map_or(false, |t| t > threshold)
}

技巧 2 —— 密封 Trait

第 2 章介绍了带有将每个命令绑定到其响应的关联类型的 IpmiCmd。但有一个漏洞：如果任何代码可以实现 IpmiCmd，某人可以编写 MaliciousCmd，其 parse_response 返回错误类型或 panic。密封 trait 通过使 trait 需要只有你的 crate 可以实现的私有 supertrait 来关闭此漏洞。

mod private { pub trait Sealed {} }

pub trait IpmiCmd: private::Sealed {
    type Response;
    fn net_fn(&self) -> u8;
    fn cmd_byte(&self) -> u8;
    fn payload(&self) -> Vec<u8>;
    fn parse_response(&self, raw: &[u8]) -> io::Result<Self::Response>;
}

pub struct ReadTemp { pub sensor_id: u8 }
impl private::Sealed for ReadTemp {}
impl IpmiCmd for ReadTemp {
    type Response = Celsius;
    // ... 实现 ...
}

验证边界的基于属性测试

验证边界（ch07）解析数据一次并拒绝无效输入。但你如何知道验证捕获所有无效输入？使用 proptest 的基于属性测试生成数千个随机输入来压力测试边界：

use proptest::prelude::*;

proptest! {
    /// 任何通过验证的字节序列必须可以使用而不 panic。
    #[test]
    fn valid_fru_never_panics(data in proptest::collection::vec(any::<u8>(), 0..1024)) {
        if let Ok(fru) = ValidFru::try_from(RawFruData(data)) {
            // 这些在验证的 FRU 上绝不能 panic
            let _ = fru.format_version();
            let _ = fru.board_area();
            let _ = fru.product_area();
        }
    }

    /// 往返：格式版本通过重新解析保留。
    #[test]
    fn fru_round_trip(data in valid_fru_strategy()) {
        let raw = RawFruData(data.clone());
        let fru = ValidFru::try_from(raw).unwrap();
        let version = fru.format_version();
        let reparsed = ValidFru::try_from(RawFruData(data)).unwrap();
        prop_assert_eq!(version, reparsed.format_version());
    }
}

/// 自定义策略：生成满足 FRU 规范头部的字节向量。
fn valid_fru_strategy() -> impl Strategy<Value = Vec<u8>> {
    let header = vec![0x01, 0x00, 0x01, 0x02, 0x00, 0x00, 0x00];
    proptest::collection::vec(any::<u8>(), 64..256)
        .prop_map(move |body| {
            let mut fru = header.clone();
            let sum: u8 = fru.iter().fold(0u8, |a, &b| a.wrapping_add(b));
            fru.push(0u8.wrapping_sub(sum));
            fru.extend_from_slice(&body);
            fru
        })
}

RAII 验证

RAII（技巧 12）保证清理。要测试这个，验证 Drop impl 实际上触发：

use std::sync::atomic::{AtomicBool, Ordering};

static DROPPED: AtomicBool = AtomicBool::new(false);

struct TestSession;
impl Drop for TestSession {
    fn drop(&mut self) { DROPPED.store(true, Ordering::SeqCst); }
}

#[test]
fn session_drops_on_early_return() {
    DROPPED.store(false, Ordering::SeqCst);
    let result: Result<(), &str> = (|| {
        let _session = TestSession;
        Err("simulated failure")?;
        Ok(())
    })();
    assert!(result.is_err());
    assert!(DROPPED.load(Ordering::SeqCst), "Drop must fire on early return");
}

零成本抽象：通过汇编证明

常见担忧："新类型和幻象类型添加运行时开销吗？"答案是不 —— 它们编译为与原始基元相同的汇编。

设置：

cargo install cargo-show-asm

示例：新类型与原始 u32：

#[derive(Clone, Copy)]
pub struct Rpm(pub u32);

#[inline(never)]
pub fn add_rpm(a: Rpm, b: Rpm) -> Rpm { Rpm(a.0 + b.0) }

#[inline(never)]
pub fn add_raw(a: u32, b: u32) -> u32 { a + b }

运行：

cargo asm my_crate::add_rpm
cargo asm my_crate::add_raw

结果 —— 相同汇编：

; add_rpm (新类型)           ; add_raw (原始 u32)
my_crate::add_rpm:            my_crate::add_raw:
  lea eax, [rdi + rsi]         lea eax, [rdi + rsi]
  ret                          ret

Rpm 包装器在编译时完全擦除。PhantomData<S>（零字节）、ZST 令牌（零字节）和整个本书中使用的所有其他类型级标记也是如此。

关键要点

  1. trybuild 测试无效代码不会编译 —— 对于在重构期间维护类型级不变量至关重要。
  2. proptest 模糊验证边界 —— 生成数千个随机输入来压力测试 TryFrom 实现。
  3. RAII 验证测试 Drop 运行 —— Arc 计数器或模拟标志证明清理发生。
  4. cargo-show-asm 证明零成本 —— 幻象类型、ZST 和新类型产生与原始 C 相同的汇编。
  5. 为每个"不可能"状态添加编译失败测试 —— 如果有人意外在一次性类型上派生 Clone，测试会捕获它。
