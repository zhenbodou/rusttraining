Type-Driven Correctness in Rust

Send & Sync —— 编译时并发证明 🟠

你将学到： Rust 的 Send 和 Sync auto-traits 如何将编译器变成并发审计员 —— 在编译时证明哪些类型可以跨线程边界以及哪些可以共享，零运行时成本。

交叉引用： ch04（能力令牌）、ch09（幻象类型）、ch15（const fn 证明）

问题：没有安全网的并发访问

在系统编程中，外设、共享缓冲区和全局状态从多个上下文访问 —— 主循环、中断处理程序、DMA 回调和工作线程。在 C 中，编译器不提供任何强制执行：

/* 共享传感器缓冲区 —— 从主循环和 ISR 访问 */
volatile uint32_t sensor_buf[64];
volatile uint32_t buf_index = 0;

void SENSOR_IRQHandler(void) {
    sensor_buf[buf_index++] = read_sensor();  /* 竞争：buf_index 读 + 写 */
}

void process_sensors(void) {
    for (uint32_t i = 0; i < buf_index; i++) {  /* buf_index 在循环中间改变 */
        process(sensor_buf[i]);                   /* 数据在读取中间被覆盖 */
    }
    buf_index = 0;                                /* ISR 在这几行之间触发 */
}

volatile 关键字防止编译器优化掉读取，但对数据竞争毫无作用。两个上下文可以同时读写 buf_index，产生撕裂值、丢失更新或缓冲区溢出。pthread_mutex_t 也出现同样的问题 —— 编译器会愉快地让你忘记锁定：

pthread_mutex_t lock;
int shared_counter;

void increment(void) {
    shared_counter++;  /* 哎呀 —— 忘记 pthread_mutex_lock(&lock) */
}

每个并发 bug 都在运行时发现 —— 通常在负载下、在生产中、间歇性地。

Send 和 Sync 证明什么

Rust 定义两个编译器自动派生的 marker traits：

Trait证明非正式含义
SendT 类型的值可以安全地移动到另一个线程"这可以跨线程边界"
Sync共享引用 &T 可以安全地被多个线程使用"这可以跨线程读取"

这些是auto-traits —— 编译器通过检查每个字段派生它们。如果所有字段都是 Send，结构体就是 Send。如果任何字段选择退出，整个结构体选择退出。不需要注解，没有运行时开销 —— 证明是结构的。

编译器是审计员。 在 C 中，线程安全注解存在于注释和头文件文档中 —— 建议性的，从不强制执行。在 Rust 中，Send 和 Sync 从类型本身的结构派生。添加单个 Cell<f32> 字段自动使包含的结构体 !Sync。不需要程序员操作，无法忘记。

两个 trait 通过关键恒等式链接：

T 是 Sync 当且仅当 &T 是 Send。

这直观上有意义：如果共享引用可以安全地发送到另一个线程，那么底层类型对并发读取是安全的。

选择退出的类型

某些类型故意是 !Send 或 !Sync：

类型SendSync为什么
u32、String、Vec<T>✅✅无内部可变性，无原始指针
Cell<T>、RefCell<T>✅❌无同步的内部可变性
Rc<T>❌❌引用计数不是原子的
*const T、*mut T❌❌原始指针没有安全保证
Arc<T>（其中 T: Send + Sync）✅✅原子引用计数
Mutex<T>（其中 T: Send）✅✅锁定序列化所有访问

此表中的每个 ❌ 是编译时不变量。你不能意外发送 Rc 到另一个线程 —— 编译器拒绝它。

!Send 外设句柄

在嵌入式系统中，外设寄存器块位于固定内存地址，应仅从单个执行上下文访问。原始指针天然是 !Send 和 !Sync，所以包装一个自动使包含类型退出两个 trait：

/// 内存映射 UART 外设的句柄。
/// 原始指针使这个自动 !Send 和 !Sync。
pub struct Uart { regs: *const u32 }

impl Uart {
    pub fn new(base: usize) -> Self { Self { regs: base as *const u32 } }
    pub fn write_byte(&self, byte: u8) { /* ... */ }
}

fn main() {
    let uart = Uart::new(0x4000_1000);
    uart.write_byte(b'A');  // ✅ 在创建线程上使用
    // ❌ 不会编译：Uart 是 !Send
    // std::thread::spawn(move || { uart.write_byte(b'B'); });
}

注释掉的 thread::spawn 将产生：

error[E0277]: `*const u32` cannot be sent between threads safely

没有原始指针？使用 PhantomData。 有时类型没有原始指针但仍应限制在一个线程 —— 例如文件描述符索引或从 C 库获得的句柄：

#![allow(unused)]
fn main() {
use std::marker::PhantomData;

/// 来自 C 库的不透明句柄。PhantomData<*const ()> 使它 !Send + !Sync
/// 即使内部 fd 只是普通整数。
pub struct LibHandle {
    fd: i32,
    _not_send: PhantomData<*const ()>,
}
}

Mutex 将 !Sync 转换为 Sync

Cell<T> 和 RefCell<T> 提供内部可变性而没有任何同步 —— 所以它们是 !Sync。但有时你确实需要跨线程共享可变状态。Mutex<T> 添加缺少的同步，编译器识别这个：

如果 T: Send，那么 Mutex<T>: Send + Sync。

锁定序列化所有访问，所以 !Sync 内部类型变得安全共享。编译器在结构上证明这一点 —— 没有"程序员忘记锁定"的运行时检查：

use std::sync::{Arc, Mutex};
use std::cell::Cell;

struct SensorCache {
    last_reading: Cell<u32>,
    reading_count: Cell<u32>,
}

fn main() {
    // Mutex 使 SensorCache 安全共享 —— 编译器证明它
    let cache = Arc::new(Mutex::new(SensorCache {
        last_reading: Cell::new(0),
        reading_count: Cell::new(0),
    }));

    let handles: Vec<_> = (0..4).map(|i| {
        let c = Arc::clone(&cache);
        std::thread::spawn(move || {
            let guard = c.lock().unwrap();  // 访问前必须锁定
            guard.last_reading.set(i * 10);
            guard.reading_count.set(guard.reading_count.get() + 1);
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
}

与 C 版本比较：pthread_mutex_lock 是程序员可能忘记的运行时调用。这里，类型系统使不通过锁定访问 SensorCache 不可能。证明是结构的 —— 唯一运行时成本是锁定本身。

Mutex 不仅同步 —— 它证明同步。 Mutex::lock() 返回 MutexGuard，它 Deref 到 &T。没有办法获得内部数据的引用而不通过锁定。API 使"忘记锁定"在结构上不可表示。

作为定理的函数边界

std::thread::spawn 有这个签名：

pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,

Send + 'static 边界不仅是实现细节 —— 它是定理：

"任何传递给 spawn 的闭包和返回值在编译时被证明安全地在另一个线程上运行，没有悬垂引用。"

你可以将相同模式应用到你自己的 API：

use std::sync::mpsc;

fn run_on_background<F, T>(task: F) -> T
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,
{
    let (tx, rx) = mpsc::channel();
    std::thread::spawn(move || { let _ = tx.send(task()); });
    rx.recv().expect("background task panicked")
}

fn main() {
    // ✅ u32 是 Send，闭包不捕获非 Send 内容
    let result = run_on_background(|| 6 * 7);
    // ✅ String 是 Send
    let greeting = run_on_background(|| String::from("hello from background"));
    // ❌ 不会编译：Rc 是 !Send
    // use std::rc::Rc;
    // let data = Rc::new(42);
    // run_on_background(move || *data);
}

取消注释 Rc 示例产生精确诊断：

error[E0277]: `Rc<i32>` cannot be sent between threads safely

编译器跟踪违反回到确切边界 —— 并告诉程序员为什么。与 C 的 pthread_create 比较：

int pthread_create(pthread_t *thread, const pthread_attr_t *attr,
                   void *(*start_routine)(void *), void *arg);

void *arg 接受任何内容 —— 线程安全或不是。C 编译器不能区分非原子引用计数和普通整数。Rust 的 trait 边界在类型级别进行区分。

何时使用 Send/Sync 证明

场景方法
包装原始指针的外设句柄自动 !Send + !Sync —— 无需操作
来自 C 库的句柄（整数 fd/句柄）添加 PhantomData<*const ()> 用于 !Send + !Sync
锁定后的共享配置Arc<Mutex<T>> —— 编译器证明访问安全
跨线程消息传递mpsc::channel —— Send 边界自动强制执行
任务派生器或线程池 API在签名中要求 F: Send + 'static
单线程资源（例如 GPU 上下文）PhantomData<*const ()> 防止共享
类型应该是 Send 但包含原始指针unsafe impl Send 带有文档化的安全理由

成本摘要

内容运行时成本
Send / Sync 自动派生仅编译时 —— 0 字节
PhantomData<*const ()> 字段零大小 —— 优化掉
!Send / !Sync 强制执行仅编译时 —— 无运行时检查
F: Send + 'static 函数边界单态化 —— 静态分发，无装箱
Mutex<T> 锁定运行时锁定（共享可变性不可避免）
Arc<T> 引用计数原子递增/递减（共享所有权不可避免）

前四行是零成本 —— 它们只存在于类型系统中，编译后消失。Mutex 和 Arc 携带不可避免的运行时成本，但这些成本是任何正确并发程序必须支付的最低成本 —— Rust 只是确保你支付它们。

练习：DMA 传输保护

设计 DmaTransfer<T>，在 DMA 传输进行中时持有缓冲区。要求：

  1. DmaTransfer 必须是 !Send —— DMA 控制器使用绑定到此核心内存总线的物理地址
  2. DmaTransfer 必须是 !Sync —— DMA 写入时并发读取会看到撕裂数据
  3. 提供 wait() 方法消耗保护并返回缓冲区 —— 所有权证明传输完成
  4. 缓冲区类型 T 必须实现 DmaSafe marker trait
解决方案 
use std::marker::PhantomData;

trait DmaSafe {}
impl DmaSafe for [u8; 64] {}

pub struct DmaTransfer<T: DmaSafe> {
    buffer: T,
    channel: u8,
    _no_send_sync: PhantomData<*const ()>,
}

impl<T: DmaSafe> DmaTransfer<T> {
    pub fn start(buffer: T, channel: u8) -> Self {
        println!("DMA channel {} started", channel);
        Self { buffer, channel, _no_send_sync: PhantomData }
    }
    pub fn wait(self) -> T {
        println!("DMA channel {} complete", self.channel);
        self.buffer
    }
}

fn main() {
    let buf = [0u8; 64];
    let transfer = DmaTransfer::start(buf, 2);
    // ❌ buf 不再可访问 —— 所有权防止 DMA 期间使用
    // ❌ 不会编译：DmaTransfer 是 !Send
    // std::thread::spawn(move || { transfer.wait(); });
    let buf = transfer.wait();  // ✅ 在原始线程上等待，取回缓冲区
}

关键要点

  1. Send 和 Sync 是关于并发安全的编译时证明 —— 编译器通过检查每个字段在结构上派生它们。
  2. 原始指针自动选择退出 —— 任何包含 *const T 或 *mut T 的类型变成 !Send + !Sync。
  3. PhantomData<*const ()> 是显式选择退出 —— 当类型没有原始指针但仍应是线程限制时使用。
  4. Mutex<T> 恢复 Sync 并证明 —— 编译器在结构上证明所有访问通过锁定。
  5. 函数边界是定理 —— 签名中的 F: Send + 'static 是编译时证明义务。
  6. 模式补充所有其他正确性技术 —— 类型状态证明协议排序，幻象类型证明权限，const fn 证明值不变量，Send/Sync 证明并发安全。
