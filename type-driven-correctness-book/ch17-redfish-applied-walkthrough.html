Type-Driven Correctness in Rust

应用演练 —— 类型安全 Redfish 客户端 🟡

你将学到： 如何将类型状态会话、能力令牌、幻象类型资源导航、量纲分析、验证边界、构建器类型状态和一次性类型组合成完整、零开销的 Redfish 客户端 —— 每个协议违规都是编译错误。

交叉引用： ch02（类型化命令）、ch03（一次性类型）、ch04（能力令牌）、ch05（类型状态）、ch06（量纲类型）、ch07（验证边界）、ch09（幻象类型）、ch10（IPMI 集成）、ch11（技巧 4 —— 构建器类型状态）

为什么 Redfish 值得拥有自己的章节

第 10 章围绕 IPMI 组合核心模式 —— 字节级协议。但大多数 BMC 平台现在 alongside（或代替）IPMI 暴露 Redfish REST API，Redfish 引入它自己的正确性危险类别：

危险示例后果
格式错误的 URIGET /redfish/v1/Chassis/1/Processors（错误父级）404 或静默返回错误数据
错误电源状态上的操作在已关闭系统上 Reset(ForceOff)BMC 返回错误，或更糟，与另一个操作竞争
缺少权限操作员级代码调用 Manager.ResetToDefaults生产中的 403，安全审计发现
不完整的 PATCH从 PATCH 体中省略必需的 BIOS 属性静默无操作或部分配置损坏
未验证的固件应用镜像完整性检查之前调用 SimpleUpdateBMC 损坏
模式版本不匹配在 v1.5 BMC 上访问 LastResetTime（v1.13 添加）null 字段 → 运行时 panic
遥测中的单位混淆比较入口温度（°C）到功率绘制（W）无意义的阈值决策

在 C、Python 或无类型 Rust 中，每个这些都仅通过纪律和测试防止。本章使它们成为编译错误。

无类型 Redfish 客户端

典型的 Redfish 客户端看起来像这样：

struct RedfishClient { base_url: String, token: Option<String> }

impl RedfishClient {
    fn get(&self, path: &str) -> Result<serde_json::Value, String> { /* ... */ }
    fn patch(&self, path: &str, body: &serde_json::Value) -> Result<(), String> { /* ... */ }
    fn post_action(&self, path: &str, body: &serde_json::Value) -> Result<(), String> { /* ... */ }
}

fn check_thermal(client: &RedfishClient) -> Result<(), String> {
    let resp = client.get("/redfish/v1/Chassis/1/Thermal")?;
    // 🐛 此字段总是存在吗？如果 BMC 返回 null 怎么办？
    let cpu_temp = resp["Temperatures"][0]["ReadingCelsius"].as_f64().unwrap();
    let fan_rpm = resp["Fans"][0]["Reading"].as_f64().unwrap();
    // 🐛 比较 °C 到 RPM —— 都是 f64
    if cpu_temp > fan_rpm { println!("thermal issue"); }
    // 🐛 这是正确的路径吗？没有编译时检查。
    client.post_action("/redfish/v1/Systems/1/Actions/ComputerSystem.Reset",
        &serde_json::json!({"ResetType": "ForceOff"}))?;
    Ok(())
}

这"工作" —— 直到它不。每个 unwrap() 是潜在 panic，每个字符串路径是未检查的假设，单位混淆是看不见的。

第 1 节 —— 会话生命周期（类型状态，ch05）

Redfish 会话有严格的生命周期：connect → authenticate → use → close。将每个状态编码为不同型号。

stateDiagram-v2
    [*] --> Disconnected
    Disconnected --> Connected : connect(host)
    Connected --> Authenticated : login(user, pass)
    Authenticated --> Authenticated : get() / patch() / post_action()
    Authenticated --> Closed : logout()
    Closed --> [*]

use std::marker::PhantomData;

pub struct Disconnected;
pub struct Connected;
pub struct Authenticated;

pub struct RedfishSession<S> {
    base_url: String,
    auth_token: Option<String>,
    _state: PhantomData<S>,
}

impl RedfishSession<Disconnected> {
    pub fn new(host: &str) -> Self {
        RedfishSession { base_url: format!("https://{}", host), auth_token: None, _state: PhantomData }
    }
    pub fn connect(self) -> Result<RedfishSession<Connected>, RedfishError> {
        println!("Connecting to {}/redfish/v1", self.base_url);
        Ok(RedfishSession { base_url: self.base_url, auth_token: None, _state: PhantomData })
    }
}

impl RedfishSession<Connected> {
    pub fn login(self, user: &str, _pass: &str) -> Result<(RedfishSession<Authenticated>, LoginToken), RedfishError> {
        println!("Authenticated as {}", user);
        Ok((
            RedfishSession { base_url: self.base_url, auth_token: Some("X-Auth-Token".to_string()), _state: PhantomData },
            LoginToken { _private: () },
        ))
    }
}

impl RedfishSession<Authenticated> {
    fn http_get(&self, path: &str) -> Result<serde_json::Value, RedfishError> { /* ... */ }
    fn http_patch(&self, path: &str, body: &serde_json::Value) -> Result<serde_json::Value, RedfishError> { /* ... */ }
    fn http_post(&self, path: &str, body: &serde_json::Value) -> Result<serde_json::Value, RedfishError> { /* ... */ }
    pub fn logout(self) { println!("Session closed"); }
}

消除的 bug 类： 在断开连接或未认证会话上发送请求。方法根本不存在 —— 没有运行时检查可以忘记。

第 2 节 —— 权限令牌（能力令牌，ch04）

Redfish 定义四个权限级别：Login、ConfigureComponents、ConfigureManager、ConfigureSelf。与其在运行时检查权限，不如将它们编码为零大小证明令牌。

pub struct LoginToken { _private: () }
pub struct ConfigureComponentsToken { _private: () }
pub struct ConfigureManagerToken { _private: () }

/// 任何有 Login 的人都可以读取热数据。
fn get_thermal(session: &RedfishSession<Authenticated>, _proof: &LoginToken) -> Result<serde_json::Value, RedfishError> { /* ... */ }

/// 更改启动顺序需要 ConfigureComponents。
fn set_boot_order(session: &RedfishSession<Authenticated>, _proof: &ConfigureComponentsToken, order: &[&str]) -> Result<(), RedfishError> { /* ... */ }

/// 恢复出厂设置需要 ConfigureManager。
fn reset_to_defaults(session: &RedfishSession<Authenticated>, _proof: &ConfigureManagerToken) -> Result<(), RedfishError> { /* ... */ }

消除的 bug 类： 权限提升。操作员级登录物理上不能生成 ConfigureManagerToken —— 编译器不会让代码引用它。零运行时成本：对于编译的二进制文件，这些令牌不存在。

第 3 节 —— 类型化资源导航（幻象类型，ch09）

Redfish 资源形成树。将层次结构编码为类型防止构造非法 URI：

use std::marker::PhantomData;

pub struct ServiceRoot;
pub struct SystemsCollection;
pub struct ComputerSystem;
pub struct ChassisCollection;
pub struct ChassisInstance;
pub struct ThermalResource;

pub struct RedfishPath<R> { uri: String, _resource: PhantomData<R> }

impl RedfishPath<ServiceRoot> {
    pub fn root() -> Self { RedfishPath { uri: "/redfish/v1".to_string(), _resource: PhantomData } }
    pub fn chassis(&self) -> RedfishPath<ChassisCollection> {
        RedfishPath { uri: format!("{}/Chassis", self.uri), _resource: PhantomData }
    }
}

impl RedfishPath<ChassisCollection> {
    pub fn instance(&self, id: &str) -> RedfishPath<ChassisInstance> {
        RedfishPath { uri: format!("{}/{}", self.uri, id), _resource: PhantomData }
    }
}

impl RedfishPath<ChassisInstance> {
    pub fn thermal(&self) -> RedfishPath<ThermalResource> {
        RedfishPath { uri: format!("{}/Thermal", self.uri), _resource: PhantomData }
    }
}

fn build_paths() {
    let thermal = RedfishPath::root().chassis().instance("1").thermal();
    assert_eq!(thermal.uri(), "/redfish/v1/Chassis/1/Thermal");
    // ❌ 编译错误：ServiceRoot 没有 .thermal() 方法
    // RedfishPath::root().thermal();
}

消除的 bug 类： 格式错误的 URI，导航到给定父级下不存在的子资源。层次结构在结构上强制执行 —— 你只能通过 Chassis → Instance → Thermal 到达 Thermal。

第 4 节 —— 类型化遥测读取（类型化命令 + 量纲分析，ch02 + ch06）

将类型化资源路径与量纲返回类型结合，所以编译器知道每个读数携带什么单位：

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
pub struct Celsius(pub f64);
#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
pub struct Rpm(pub u32);

pub trait RedfishResource {
    type Response;
    fn parse(json: &serde_json::Value) -> Result<Self::Response, RedfishError>;
}

#[derive(Debug)]
pub struct ValidThermalResponse {
    pub temperatures: Vec<TemperatureReading>,
    pub fans: Vec<FanReading>,
}

#[derive(Debug)]
pub struct TemperatureReading {
    pub name: String,
    pub reading: Celsius,           // ← 量纲类型，不是 f64
    pub upper_critical: Celsius,
    pub status: HealthStatus,
}

#[derive(Debug)]
pub struct FanReading {
    pub name: String,
    pub reading: Rpm,               // ← 量纲类型，不是 u32
    pub status: HealthStatus,
}

impl RedfishResource for ThermalResource {
    type Response = ValidThermalResponse;
    fn parse(json: &serde_json::Value) -> Result<ValidThermalResponse, RedfishError> { /* ... */ }
}

impl RedfishSession<Authenticated> {
    pub fn get_resource<R: RedfishResource>(&self, path: &RedfishPath<R>) -> Result<R::Response, RedfishError> {
        let json = self.http_get(path.uri())?;
        R::parse(&json)
    }
}

fn read_thermal(session: &RedfishSession<Authenticated>, _proof: &LoginToken) -> Result<(), RedfishError> {
    let path = RedfishPath::root().chassis().instance("1").thermal();
    let thermal = session.get_resource(&path)?;
    for t in &thermal.temperatures {
        // t.reading 是 Celsius —— 只能与 Celsius 比较
        if t.reading > t.upper_critical { println!("CRITICAL: {} at {:?}", t.name, t.reading); }
        // ❌ 编译错误：不能比较 Celsius 与 Rpm
        // if t.reading > thermal.fans[0].reading { }
    }
    Ok(())
}

消除的 bug 类：

  • 单位混淆： Celsius ≠ Rpm ≠ Watts —— 编译器拒绝比较。
  • 缺失字段 panic： parse() 在边界验证；ValidThermalResponse 保证所有字段存在。
  • 错误响应类型： get_resource(&thermal_path) 返回 ValidThermalResponse，不是原始 JSON。

第 5 节 —— 带有构建器类型状态的 PATCH（ch11，技巧 4）

Redfish PATCH 负载必须包含特定字段。在必需字段设置后门控 .apply() 的构建器防止不完整或空补丁：

pub struct FieldUnset;
pub struct FieldSet;

pub struct BiosPatchBuilder<BootOrder, TpmState> {
    boot_order: Option<Vec<String>>,
    tpm_enabled: Option<bool>,
    _markers: PhantomData<(BootOrder, TpmState)>,
}

impl BiosPatchBuilder<FieldSet, FieldSet> {
    pub fn apply(self, session: &RedfishSession<Authenticated>, _proof: &ConfigureComponentsToken, system: &RedfishPath<ComputerSystem>) -> Result<(), RedfishError> {
        let body = serde_json::json!({
            "Boot": { "BootOrder": self.boot_order.unwrap() },
            "Oem": { "TpmEnabled": self.tpm_enabled.unwrap() }
        });
        session.http_patch(&format!("{}/Bios/Settings", system.uri()), &body)?;
        Ok(())
    }
}

// ✅ 编译 —— 两个必需字段设置
BiosPatchBuilder::new()
    .boot_order(vec!["Pxe".into(), "Hdd".into()])
    .tpm_enabled(true)
    .apply(session, configure, &system)?;

// ❌ 编译错误：.apply() 在 BiosPatchBuilder<FieldSet, FieldUnset> 上找不到
// BiosPatchBuilder::new().boot_order(...).apply(...);

消除的 bug 类： 不完整的 PATCH 请求。构建器在类型级别跟踪哪些字段已设置，.apply() 仅在两者都设置后可用。

关键要点

  1. 类型状态用于会话生命周期 —— Disconnected → Connected → Authenticated → Closed。
  2. 能力令牌用于权限 —— LoginToken、ConfigureComponentsToken、ConfigureManagerToken。
  3. 幻象类型用于资源导航 —— RedfishPath<R> 强制执行有效的 URI 层次结构。
  4. 量纲类型用于遥测 —— Celsius、Rpm、Watts 防止单位混淆。
  5. 验证边界用于解析 —— RedfishResource::parse() 在边界验证 JSON。
  6. 构建器类型状态用于 PATCH —— BiosPatchBuilder 门控 .apply() 在必需字段上。
  7. 七个模式组合成零开销客户端 —— 每个协议违规是编译错误，不是运行时 bug。
