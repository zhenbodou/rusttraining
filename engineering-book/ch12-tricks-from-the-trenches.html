Rust Engineering Practices — Beyond cargo build

实战技巧 🟡

你将学到：

  • 不适合整齐地放入一章的经过实战检验的模式
  • 常见陷阱及其修复 — 从 CI 不稳定到二进制膨胀
  • 你今天就可以应用到任何 Rust 项目的快速获胜技术

交叉引用：本书的每一章 — 这些技巧跨越所有主题

本章收集在生产 Rust 代码库中反复出现的工程模式。 每个技巧都是自包含的 — 按任意顺序阅读它们。

1. deny(warnings) 陷阱

问题：源代码中的 #![deny(warnings)] 在 Clippy 添加新 lints 时会破坏构建 — 昨天编译的代码今天失败。

修复：在 CI 中使用 CARGO_ENCODED_RUSTFLAGS 而不是源代码级属性：

# CI：将警告视为错误而不触碰源代码
env:
  CARGO_ENCODED_RUSTFLAGS: "-Dwarnings"

或使用 [workspace.lints] 进行更细粒度的控制：

# Cargo.toml
[workspace.lints.rust]
unsafe_code = "deny"

[workspace.lints.clippy]
all = { level = "deny", priority = -1 }
pedantic = { level = "warn", priority = -1 }

参见 编译时工具，工作空间 Lints 了解完整模式。

2. 编译一次，随处测试

问题：cargo test 在 --lib、--doc 和 --test 之间切换时重新编译， 因为它们使用不同的 profile。

修复：对单元/集成测试使用 cargo nextest，单独运行 doc 测试：

cargo nextest run --workspace        # 快速：并行、缓存
cargo test --workspace --doc         # Doc 测试（nextest 无法运行这些）

参见 编译时工具 了解 cargo-nextest 设置。

3. 特性标志卫生

问题：库 crate 有 default = ["std"] 但没有人测试 --no-default-features。 一天，嵌入式用户报告它无法编译。

修复：在 CI 中添加 cargo-hack：

- name: 特性矩阵
  run: |
    cargo hack check --each-feature --no-dev-deps
    cargo check --no-default-features
    cargo check --all-features

参见 no_std 和特性验证 了解完整模式。

4. Lock 文件辩论 — 提交还是忽略？

经验法则：

Crate 类型提交 Cargo.lock？为什么
二进制 / 应用是可重现构建
库否（.gitignore）让下游选择版本
两者都有工作空间是二进制优先

添加 CI 检查以确保 lock 文件保持最新：

- name: 检查 lock 文件
  run: cargo update --locked  # 如果 Cargo.lock 过时则失败

5. 带有优化依赖的 Debug 构建

问题：Debug 构建非常慢，因为依赖（尤其是 serde、regex）未优化。

修复：在 dev profile 中优化依赖，同时保持你的代码未优化以快速重新编译：

# Cargo.toml
[profile.dev.package."*"]
opt-level = 2  # 在 dev 模式下优化所有依赖

这会稍微减慢第一次构建，但使开发期间的运行时显著更快。 对数据库支持的服务和解析器影响尤其大。

参见 Release Profile 了解每个 crate 的 profile 覆盖。

6. CI 缓存抖动

问题：Swatinem/rust-cache@v2 在每个 PR 上保存新缓存，膨胀存储并减慢恢复时间。

修复：仅从 main 保存缓存，从任何地方恢复：

- uses: Swatinem/rust-cache@v2
  with:
    save-if: ${{ github.ref == 'refs/heads/main' }}

对于具有多个二进制文件的工作空间，添加 shared-key：

- uses: Swatinem/rust-cache@v2
  with:
    shared-key: "ci-${{ matrix.target }}"
    save-if: ${{ github.ref == 'refs/heads/main' }}

参见 CI/CD 流水线 了解完整工作流。

7. RUSTFLAGS vs CARGO_ENCODED_RUSTFLAGS

问题：RUSTFLAGS="-Dwarnings" 应用于一切 — 包括构建脚本和 proc-macros。 serde_derive 的 build.rs 中的警告会使你的 CI 失败。

修复：使用 CARGO_ENCODED_RUSTFLAGS，它仅应用于顶级 crate：

# 错误 — 在第三方构建脚本警告时破坏
RUSTFLAGS="-Dwarnings" cargo build

# 好 — 仅影响你的 crate
CARGO_ENCODED_RUSTFLAGS="-Dwarnings" cargo build

# 也好 — 工作空间 lints（Cargo.toml）
[workspace.lints.rust]
warnings = "deny"

8. 使用 SOURCE_DATE_EPOCH 的可重现构建

问题：在 build.rs 中嵌入 chrono::Utc::now() 使构建不可重现 — 每次构建产生不同的二进制哈希。

修复：尊重 SOURCE_DATE_EPOCH：

#![allow(unused)]
fn main() {
// build.rs
let timestamp = std::env::var("SOURCE_DATE_EPOCH")
    .ok()
    .and_then(|s| s.parse::<i64>().ok())
    .unwrap_or_else(|| chrono::Utc::now().timestamp());
println!("cargo:rustc-env=BUILD_TIMESTAMP={timestamp}");
}

参见 构建脚本 了解完整的 build.rs 模式。

9. cargo tree 去重工作流

问题：cargo tree --duplicates 显示 5 个版本的 syn 和 3 个版本的 tokio-util。 编译时间很痛苦。

修复：系统去重：

# 步骤 1：查找重复项
cargo tree --duplicates

# 步骤 2：查找谁拉入旧版本
cargo tree --invert --package syn@1.0.109

# 步骤 3：更新罪魁祸首
cargo update -p serde_derive  # 可能拉入 syn 2.x

# 步骤 4：如果没有可用更新，在 [patch] 中固定
# [patch.crates-io]
# old-crate = { git = "...", branch = "syn2-migration" }

# 步骤 5：验证
cargo tree --duplicates  # 应该更短

参见 依赖管理 了解 cargo-deny 和供应链安全。

10. 推送前冒烟测试

问题：你推送，CI 花费 10 分钟，在格式化问题上失败。

修复：在推送前在本地运行快速检查：

# Makefile.toml（cargo-make）
[tasks.pre-push]
description = "推送前本地冒烟测试"
script = '''
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace --lib
'''

cargo make pre-push  # < 30 秒
git push

或使用 git pre-push 钩子：

#!/bin/sh
# .git/hooks/pre-push
cargo fmt --all -- --check && cargo clippy --workspace -- -D warnings

参见 CI/CD 流水线 了解 Makefile.toml 模式。

🏋️ 练习

🟢 练习 1：应用三个技巧

从本章中选择三个技巧并应用到现有的 Rust 项目。哪个影响最大？

解决方案

典型的高影响组合：

  1. [profile.dev.package."*"] opt-level = 2 — 立即改进 dev 模式运行时（解析密集型代码快 2-10 倍）

  2. CARGO_ENCODED_RUSTFLAGS — 消除来自第三方警告的假 CI 失败

  3. cargo-hack --each-feature — 通常在具有 3+ 特性的任何项目中至少发现一个破坏的特性组合

# 应用技巧 5：
echo '[profile.dev.package."*"]' >> Cargo.toml
echo 'opt-level = 2' >> Cargo.toml

# 在 CI 中应用技巧 7：
# 用 CARGO_ENCODED_RUSTFLAGS 替换 RUSTFLAGS

# 应用技巧 3：
cargo install cargo-hack
cargo hack check --each-feature --no-dev-deps

🟡 练习 2：去重你的依赖树

在真实项目上运行 cargo tree --duplicates。消除至少一个重复项。 测量前后的编译时间。

解决方案 
# 之前
time cargo build --release 2>&1 | tail -1
cargo tree --duplicates | wc -l  # 计算重复行数

# 查找并修复一个重复项
cargo tree --duplicates
cargo tree --invert --package <duplicate-crate>@<old-version>
cargo update -p <parent-crate>

# 之后
time cargo build --release 2>&1 | tail -1
cargo tree --duplicates | wc -l  # 应该更少

# 典型结果：每个消除的重复项减少 5-15% 编译时间
# （尤其是对于 syn、tokio 等重型 crate）

关键要点

  • 使用 CARGO_ENCODED_RUSTFLAGS 而不是 RUSTFLAGS 以避免破坏第三方构建脚本
  • [profile.dev.package."*"] opt-level = 2 是单个最高影响力的开发体验技巧
  • 缓存调整（仅在 main 上 save-if）防止活跃仓库上的 CI 缓存膨胀
  • cargo tree --duplicates + cargo update 是免费的编译时间获胜 — 每月做一次
  • 使用 cargo make pre-push 在本地运行快速检查以避免 CI 往返浪费
