默认的 cargo build --release 已经很好。但对于生产部署 — 尤其是部署到数千台服务器的单二进制工具 — "好"和"优化"之间存在显著差距。 本章涵盖 profile 调整和测量二进制大小的工具。

Release Profile 详解

Cargo profile 控制 rustc 如何编译你的代码。默认值是保守的 — 设计用于广泛兼容，而非最大性能：

# Cargo.toml — Cargo 的内置默认值（如果你不指定任何内容会得到什么）

[profile.release]
opt-level = 3        # 优化级别（0=无，1=基本，2=好，3=激进）
lto = false          # 链接时优化关闭
codegen-units = 16   # 并行编译单元（更快的编译，更少的优化）
panic = "unwind"     # panic 时栈展开（更大的二进制文件，catch_unwind 有效）
strip = "none"       # 保留所有符号和 debug 信息
overflow-checks = false  # release 中无整数溢出检查
debug = false        # release 中无 debug 信息

生产优化 profile（项目已经使用的）：

[profile.release]
lto = true           # 完整的跨 crate 优化
codegen-units = 1    # 单个代码生成单元 — 最大优化机会
panic = "abort"      # 无展开开销 — 更小、更快
strip = true         # 移除所有符号 — 更小的二进制文件

每个设置的影响：

设置默认 → 优化二进制大小运行时速度编译时间
lto = false → true—-10 到 -20%+5 到 +20%2-5 倍慢
codegen-units = 16 → 1—-5 到 -10%+5 到 +10%1.5-2 倍慢
panic = "unwind" → "abort"—-5 到 -10%可忽略可忽略
strip = "none" → true—-50 到 -70%无无
opt-level = 3 → "s"—-10 到 -30%-5 到 -10%相似
opt-level = 3 → "z"—-15 到 -40%-10 到 -20%相似

额外的 profile 调整：

[profile.release]
# 以上所有内容，加上：
overflow-checks = true    # 即使在 release 中也保留溢出检查（安全 > 速度）
debug = "line-tables-only" # 最小 debug 信息用于回溯，无完整 DWARF
rpath = false             # 不嵌入运行时库路径
incremental = false       # 禁用增量编译（更干净的构建）

# 对于大小优化的构建（嵌入式、WASM）：
# opt-level = "z"         # 激进地为大小优化
# strip = "symbols"       # 剥离符号但保留 debug 部分

每个 crate 的 profile 覆盖 — 优化热 crate，其他不变：

# Dev 构建：优化依赖但不优化你的代码（快速重新编译）
[profile.dev.package."*"]
opt-level = 2          # 在 dev 模式下优化所有依赖

# Release 构建：覆盖特定 crate 优化
[profile.release.package.serde_json]
opt-level = 3          # JSON 解析的最大优化
codegen-units = 1

# Test profile：匹配 release 行为以获得准确的集成测试
[profile.test]
opt-level = 1          # 一些优化以避免慢测试超时

LTO 深入 — Thin vs Fat vs Cross-Language

链接时优化允许 LLVM 跨 crate 边界优化 — 将 serde_json 中的函数内联到你的解析代码中， 从 regex 中移除死代码等。没有 LTO，每个 crate 都是单独的优化孤岛。

[profile.release]
# 选项 1：Fat LTO（lto = true 时的默认值）
lto = true
# 所有代码合并到一个 LLVM 模块 → 最大优化
# 最慢的编译，最小/最快的二进制文件

# 选项 2：Thin LTO
lto = "thin"
# 每个 crate 保持独立但 LLVM 进行跨模块优化
# 比 fat LTO 更快的编译，几乎一样好的优化
# 大多数项目的最佳权衡

# 选项 3：无 LTO
lto = false
# 仅 crate 内优化
# 最快的编译，更大的二进制文件

# 选项 4：Off（显式）
lto = "off"
# 与 false 相同

Fat LTO vs Thin LTO：

方面Fat LTO (true)Thin LTO ("thin")
优化质量最佳~95% 的 fat
编译时间慢（所有代码在一个模块中）中等（并行模块）
内存使用高（所有 LLVM IR 在内存中）较低（流式）
并行性无（单个模块）好（每个模块）
推荐用于最终 release 构建CI 构建、开发

跨语言 LTO — 跨 Rust 和 C 边界优化：

[profile.release]
lto = true

# Cargo.toml — 用于使用 cc crate 的 crate
[build-dependencies]
cc = "1.0"

// build.rs — 启用跨语言（链接器插件）LTO
fn main() {
    // cc crate 尊重来自环境的 CFLAGS。
    // 对于跨语言 LTO，使用以下命令编译 C 代码：
    //   -flto=thin -O2
    cc::Build::new()
        .file("csrc/fast_parser.c")
        .flag("-flto=thin")
        .opt_level(2)
        .compile("fast_parser");
}
# 启用链接器插件 LTO（需要兼容的 LLD 或 gold 链接器）
RUSTFLAGS="-Clinker-plugin-lto -Clinker=clang -Clink-arg=-fuse-ld=lld" \
    cargo build --release

跨语言 LTO 允许 LLVM 将 C 函数内联到 Rust 调用者中，反之亦然。 这对于频繁调用小型 C 函数的 FFI 繁重代码影响最大（例如，IPMI ioctl 包装器）。

使用 cargo-bloat 进行二进制大小分析

cargo-bloat 回答： "哪些函数和 crate 在我的二进制文件中占用最多空间？"

# 安装
cargo install cargo-bloat

# 显示最大的函数
cargo bloat --release -n 20
# 输出：
#  File  .text     Size          Crate    Name
#  2.8%   5.1%  78.5KiB  serde_json       serde_json::de::Deserializer::parse_...
#  2.1%   3.8%  58.2KiB  regex_syntax     regex_syntax::ast::parse::ParserI::p...
#  1.5%   2.7%  42.1KiB  accel_diag         accel_diag::vendor::parse_smi_output
#  ...

# 按 crate 显示（哪些依赖最大）
cargo bloat --release --crates
# 输出：
#  File  .text     Size Crate
# 12.3%  22.1%  340KiB serde_json
#  8.7%  15.6%  240KiB regex
#  6.2%  11.1%  170KiB std
#  5.1%   9.2%  141KiB accel_diag
#  ...

# 比较两个构建（优化前后）
cargo bloat --release --crates > before.txt
# ... 进行更改 ...
cargo bloat --release --crates > after.txt
diff before.txt after.txt

常见的膨胀源和修复：

膨胀源典型大小修复
regex（完整引擎）200-400 KB如果不需要 Unicode 则使用 regex-lite
serde_json（完整）200-350 KB如果性能重要则考虑 simd-json 或 sonic-rs
泛型单态化变化在 API 边界使用 dyn Trait
格式化机制（Display、Debug）50-150 KB大型 enum 上的 #[derive(Debug)] 会累积
Panic 消息字符串20-80 KBpanic = "abort" 移除展开，strip 移除字符串
未使用的特性变化禁用默认特性：serde = { version = "1", default-features = false }

使用 cargo-udeps 修剪依赖

cargo-udeps 查找 在 Cargo.toml 中声明但你的代码实际未使用的依赖：

# 安装（需要 nightly）
cargo install cargo-udeps

# 查找未使用的依赖
cargo +nightly udeps --workspace
# 输出：
# unused dependencies:
# `diag_tool v0.1.0`
# └── "tempfile" (dev-dependency)
#
# `accel_diag v0.1.0`
# └── "once_cell"    ← LazyLock 之前需要，现在死亡

每个未使用的依赖：

  • 增加编译时间
  • 增加二进制大小
  • 增加供应链风险
  • 增加潜在的许可证复杂性

替代方案：cargo-machete — 更快、基于启发式的方法：

cargo install cargo-machete
cargo machete
# 更快但可能有假阳性（启发式，而非基于编译）

替代方案：cargo-shear — cargo-udeps 和 cargo-machete 之间的最佳点：

cargo install cargo-shear
cargo shear --fix
# 比 cargo-machete 慢但比 cargo-udeps 快得多
# 比 cargo-machete 少得多的假阳性

大小优化决策树

flowchart TD
    START["二进制文件太大？"] --> STRIP{"strip = true？"}
    STRIP -->|"否"| DO_STRIP["添加 strip = true\n-50 到 -70% 大小"]
    STRIP -->|"是"| LTO{"启用 LTO？"}
    LTO -->|"否"| DO_LTO["添加 lto = true\ncodegen-units = 1"]
    LTO -->|"是"| BLOAT["运行 cargo-bloat\n--crates"]
    BLOAT --> BIG_DEP{"大依赖？"}
    BIG_DEP -->|"是"| REPLACE["替换为更轻的\n替代方案或禁用\n默认特性"]
    BIG_DEP -->|"否"| UDEPS["cargo-udeps\n移除未使用的依赖"]
    UDEPS --> OPT_LEVEL{"需要更小？"}
    OPT_LEVEL -->|"是"| SIZE_OPT["opt-level = 's' 或 'z'"]

🏋️ 练习

🟢 练习 1：测量 LTO 影响

使用默认 release 设置构建项目，然后使用 lto = true + codegen-units = 1 + strip = true。 比较二进制大小和编译时间。

解决方案 
# 默认 release
cargo build --release
ls -lh target/release/my-binary
time cargo build --release  # 注意时间

# 优化的 release — 添加到 Cargo.toml：
# [profile.release]
# lto = true
# codegen-units = 1
# strip = true
# panic = "abort"

cargo clean
cargo build --release
ls -lh target/release/my-binary  # 通常小 30-50%
time cargo build --release       # 通常慢 2-3 倍编译

🟡 练习 2：找到你最大的 Crate

在项目上运行 cargo bloat --release --crates。识别最大的依赖。 你能通过禁用默认特性或切换到更轻的替代方案来减少它吗？

解决方案 
cargo install cargo-bloat
cargo bloat --release --crates
# 输出：
#  File  .text     Size Crate
# 12.3%  22.1%  340KiB serde_json
#  8.7%  15.6%  240KiB regex

# 对于 regex — 如果不需要 Unicode 则尝试 regex-lite：
# regex-lite = "0.1"  # 比完整 regex 小约 10 倍

# 对于 serde — 如果不需要 std 则禁用默认特性：
# serde = { version = "1", default-features = false, features = ["derive"] }

cargo bloat --release --crates  # 更改后比较

关键要点

  • lto = true + codegen-units = 1 + strip = true + panic = "abort" 是生产 release profile
  • Thin LTO（lto = "thin"）以一小部分编译成本提供 Fat LTO 80% 的收益
  • cargo-bloat --crates 准确告诉你哪些依赖在占用二进制空间
  • cargo-udeps、cargo-machete 和 cargo-shear 查找浪费编译时间和二进制大小的死依赖
  • 每个 crate 的 profile 覆盖让你优化热 crate 而不拖慢整个构建
