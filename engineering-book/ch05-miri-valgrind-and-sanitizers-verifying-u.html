Rust Engineering Practices — Beyond cargo build

Miri、Valgrind 和 Sanitizers — 验证 unsafe 代码 🔴

你将学到：

  • Miri 作为 MIR 解释器 — 它能捕获什么（别名、UB、泄漏）以及不能捕获什么（FFI、系统调用）
  • Valgrind memcheck、Helgrind（数据竞争）、Callgrind（分析）和 Massif（堆）
  • LLVM sanitizers：ASan、MSan、TSan、LSan 与 nightly -Zbuild-std
  • cargo-fuzz 用于崩溃发现和 loom 用于并发模型检查
  • 选择正确验证工具的决策树

交叉引用： 代码覆盖率 — 覆盖率发现未测试的路径，Miri 验证已测试的路径 · no_std 与特性 — no_std 代码通常需要 Miri 可以验证的 unsafe · CI/CD 流水线 — 流水线中的 Miri 作业

安全 Rust 在编译时保证内存安全和数据竞争自由。但当你编写 unsafe 的那一刻 — 为了 FFI、手写数据结构或性能技巧 — 这些保证就变成了你的责任。 本章涵盖验证你的 unsafe 代码实际上遵守其声称的安全合约的工具。

Miri — Unsafe Rust 的解释器

Miri 是 Rust 中级中间表示（MIR）的解释器。 Miri 不是编译为机器码，而是逐步执行你的程序，在每次操作时进行详尽的未定义行为检查。

# 安装 Miri（仅 nightly 组件）
rustup +nightly component add miri

# 在 Miri 下运行你的测试套件
cargo +nightly miri test

# 在 Miri 下运行特定二进制文件
cargo +nightly miri run

# 运行特定测试
cargo +nightly miri test -- test_name

Miri 的工作原理：

源代码 → rustc → MIR → Miri 解释 MIR
                        │
                        ├─ 跟踪每个指针的出处
                        ├─ 验证每次内存访问
                        ├─ 在每次解引用时检查对齐
                        ├─ 检测释放后使用
                        ├─ 检测数据竞争（带线程）
                        └─ 强制 Stacked Borrows / Tree Borrows 规则

Miri 能捕获什么（以及不能捕获什么）

Miri 检测：

类别示例会在运行时崩溃吗？
越界访问ptr.add(100).read() 超出分配有时（取决于页面布局）
释放后使用通过原始指针读取已丢弃的 Box有时（取决于分配器）
双重释放调用 drop_in_place 两次通常
未对齐访问在奇数地址上 (ptr as *const u32).read()在某些架构上
无效值transmute::<u8, bool>(2)静默错误
悬垂引用&*ptr 其中 ptr 已释放否（静默损坏）
数据竞争两个线程，一个写入，无同步间歇性，难以重现
Stacked Borrows 违规别名 &mut 引用否（静默损坏）

Miri 不检测：

限制原因
逻辑 bugMiri 检查内存安全，而不是正确性
并发死锁Miri 检查数据竞争，而不是活锁
性能问题解释比原生慢 10-100 倍
OS/硬件交互Miri 无法模拟系统调用、设备 I/O
所有 FFI 调用无法解释 C 代码（仅 Rust MIR）
穷尽路径覆盖仅测试你的测试套件触及的路径

具体示例 — 捕获在实践中"有效"的不健全代码：

#![allow(unused)]
fn main() {
#[cfg(test)]
mod tests {
    #[test]
    fn test_miri_catches_ub() {
        // 这在 release 构建中"有效"但是未定义行为
        let mut v = vec![1, 2, 3];
        let ptr = v.as_ptr();

        // Push 可能重新分配，使 ptr 无效
        v.push(4);

        // ❌ UB：ptr 可能在重新分配后悬垂
        // Miri 会捕获这个，即使分配器碰巧没有移动缓冲区。
        // let _val = unsafe { *ptr };
        // 错误：Miri 会报告：
        //   "pointer to alloc1234 was dereferenced after this
        //    allocation got freed"

        // ✅ 正确：在变异后获取新指针
        let ptr = v.as_ptr();
        let val = unsafe { *ptr };
        assert_eq!(val, 1);
    }
}
}

在真实 Crate 上运行 Miri

带有 unsafe 的 crate 的实用 Miri 工作流：

# 步骤 1：在 Miri 下运行所有测试
cargo +nightly miri test 2>&1 | tee miri_output.txt

# 步骤 2：如果 Miri 报告错误，隔离它们
cargo +nightly miri test -- failing_test_name

# 步骤 3：使用 Miri 的回溯进行诊断
MIRIFLAGS="-Zmiri-backtrace=full" cargo +nightly miri test

# 步骤 4：选择借用模型
# Stacked Borrows（默认，更严格）：
cargo +nightly miri test

# Tree Borrows（实验性，更宽松）：
MIRIFLAGS="-Zmiri-tree-borrows" cargo +nightly miri test

常见场景的 Miri 标志：

# 禁用隔离（允许文件系统访问、环境变量）
MIRIFLAGS="-Zmiri-disable-isolation" cargo +nightly miri test

# 内存泄漏检测在 Miri 中默认开启。
# 要抑制泄漏错误（例如，用于故意泄漏）：
# MIRIFLAGS="-Zmiri-ignore-leaks" cargo +nightly miri test

# 为随机测试播种 RNG 以获得可重现的结果
MIRIFLAGS="-Zmiri-seed=42" cargo +nightly miri test

# 启用严格的出处检查
MIRIFLAGS="-Zmiri-strict-provenance" cargo +nightly miri test

# 多个标志
MIRIFLAGS="-Zmiri-disable-isolation -Zmiri-backtrace=full -Zmiri-strict-provenance" \
    cargo +nightly miri test

CI 中的 Miri：

# .github/workflows/miri.yml
name: Miri
on: [push, pull_request]

jobs:
  miri:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
        with:
          components: miri

      - name: 运行 Miri
        run: cargo miri test --workspace
        env:
          MIRIFLAGS: "-Zmiri-backtrace=full"
          # 泄漏检查在 Miri 中默认开启。
          # 跳过使用 Miri 无法处理的系统调用的测试
          # （文件 I/O、网络等）

性能注意：Miri 比原生执行慢 10-100 倍。原生运行 5 秒的测试套件在 Miri 下可能需要 5 分钟。 在 CI 中，仅在 focused 子集上运行 Miri：仅带有 unsafe 代码的 crate。

Valgrind 及其 Rust 集成

Valgrind 是经典的 C/C++ 内存检查器。它也适用于编译后的 Rust 二进制文件， 在机器码级别检查内存错误。

# 安装 Valgrind
sudo apt install valgrind  # Debian/Ubuntu
sudo dnf install valgrind  # Fedora

# 使用 debug 信息构建（Valgrind 需要符号）
cargo build --tests
# 或带有 debug 信息的 release：
# cargo build --release
# [profile.release]
# debug = true

# 在 Valgrind 下运行特定测试二进制文件
valgrind --tool=memcheck \
    --leak-check=full \
    --show-leak-kinds=all \
    --track-origins=yes \
    ./target/debug/deps/my_crate-abc123 --test-threads=1

# 运行主二进制文件
valgrind --tool=memcheck \
    --leak-check=full \
    --error-exitcode=1 \
    ./target/debug/diag_tool --run-diagnostics

memcheck 之外的 Valgrind 工具：

工具命令检测内容
Memcheck--tool=memcheck内存泄漏、释放后使用、缓冲区溢出
Helgrind--tool=helgrind数据竞争和锁顺序违规
DRD--tool=drd数据竞争（不同的检测算法）
Callgrind--tool=callgrindCPU 指令分析（路径级别）
Massif--tool=massif堆内存分析随时间变化
Cachegrind--tool=cachegrind缓存未命中分析

使用 Callgrind 进行指令级别分析：

# 记录指令计数（比挂钟时间更稳定）
valgrind --tool=callgrind \
    --callgrind-out-file=callgrind.out \
    ./target/release/diag_tool --run-diagnostics

# 使用 KCachegrind 可视化
kcachegrind callgrind.out
# 或基于文本的替代方案：
callgrind_annotate callgrind.out | head -100

两者都使用：

  • Miri 用于纯 Rust unsafe 代码（Stacked Borrows、出处）
  • Valgrind 用于 FFI 繁重的代码和整个程序的泄漏分析

AddressSanitizer、MemorySanitizer、ThreadSanitizer

LLVM sanitizers 是编译时插桩传递，插入运行时检查。 它们比 Valgrind 快（2-5 倍开销 vs 10-50 倍）并捕获不同类别的 bug。

# 必需：安装 Rust 源以使用 sanitizer 插桩重建 std
rustup component add rust-src --toolchain nightly
# AddressSanitizer（ASan） — 缓冲区溢出、释放后使用、栈溢出
RUSTFLAGS="-Zsanitizer=address" \
    cargo +nightly test -Zbuild-std --target x86_64-unknown-linux-gnu

# MemorySanitizer（MSan） — 未初始化内存读取
RUSTFLAGS="-Zsanitizer=memory" \
    cargo +nightly test -Zbuild-std --target x86_64-unknown-linux-gnu

# ThreadSanitizer（TSan） — 数据竞争
RUSTFLAGS="-Zsanitizer=thread" \
    cargo +nightly test -Zbuild-std --target x86_64-unknown-linux-gnu

# LeakSanitizer（LSan） — 内存泄漏（默认包含在 ASan 中）
RUSTFLAGS="-Zsanitizer=leak" \
    cargo +nightly test --target x86_64-unknown-linux-gnu

注意：ASan、MSan 和 TSan 需要 -Zbuild-std 来使用 sanitizer 插桩重建标准库。 LSan 不需要。

Sanitizer 比较：

Sanitizer开销捕获Nightly？-Zbuild-std？
ASan2 倍内存，2 倍 CPU缓冲区溢出、释放后使用、栈溢出是是
MSan3 倍内存，3 倍 CPU未初始化读取是是
TSan5-10 倍内存，5 倍 CPU数据竞争是是
LSan最小内存泄漏是否

实际示例 — 使用 TSan 捕获数据竞争：

#![allow(unused)]
fn main() {
use std::sync::Arc;
use std::thread;

fn racy_counter() -> u64 {
    // ❌ UB：无同步的共享可变状态
    let data = Arc::new(std::cell::UnsafeCell::new(0u64));
    let mut handles = vec![];

    for _ in 0..4 {
        let data = Arc::clone(&data);
        handles.push(thread::spawn(move || {
            for _ in 0..1000 {
                // SAFETY: 不健全 — 数据竞争！
                unsafe {
                    *data.get() += 1;
                }
            }
        }));
    }

    for h in handles {
        h.join().unwrap();
    }

    // 值应该是 4000 但由于竞争可能是任何值
    unsafe { *data.get() }
}

// Miri 和 TSan 都捕获这个：
// Miri:  "Data race detected between (1) write and (2) write"
// TSan:  "WARNING: ThreadSanitizer: data race"
//
// 修复：使用 AtomicU64 或 Mutex<u64>
}

相关工具：Fuzzing 和并发验证

cargo-fuzz — 覆盖率引导 Fuzzing（在解析器和解码器中发现崩溃）：

# 安装
cargo install cargo-fuzz

# 初始化模糊目标
cargo fuzz init
cargo fuzz add parse_gpu_csv

#![allow(unused)]
fn main() {
// fuzz/fuzz_targets/parse_gpu_csv.rs
#![no_main]
use libfuzzer_sys::fuzz_target;

fuzz_target!(|data: &[u8]| {
    if let Ok(s) = std::str::from_utf8(data) {
        // 模糊测试器生成数百万个输入以寻找 panic/崩溃。
        let _ = diag_tool::parse_gpu_csv(s);
    }
});
}
# 运行模糊测试器（运行直到被中断或发现崩溃）
cargo +nightly fuzz run parse_gpu_csv -- -max_total_time=300  # 5 分钟

# 最小化崩溃
cargo +nightly fuzz tmin parse_gpu_csv artifacts/parse_gpu_csv/crash-...

何时模糊测试：任何解析不受信任/半受信任输入的函数（传感器输出、配置文件、网络数据、JSON/CSV）。 Fuzzing 在每个主要 Rust 解析器 crate（serde、regex、image）中都发现了真正的 bug。

loom — 并发模型检查器（穷尽测试原子排序）：

[dev-dependencies]
loom = "0.7"

#![allow(unused)]
fn main() {
#[cfg(loom)]
mod tests {
    use loom::sync::atomic::{AtomicUsize, Ordering};
    use loom::thread;

    #[test]
    fn test_counter_is_atomic() {
        loom::model(|| {
            let counter = loom::sync::Arc::new(AtomicUsize::new(0));
            let c1 = counter.clone();
            let c2 = counter.clone();

            let t1 = thread::spawn(move || { c1.fetch_add(1, Ordering::SeqCst); });
            let t2 = thread::spawn(move || { c2.fetch_add(1, Ordering::SeqCst); });

            t1.join().unwrap();
            t2.join().unwrap();

            // loom 探索所有可能的线程交错
            assert_eq!(counter.load(Ordering::SeqCst), 2);
        });
    }
}
}

何时使用 loom：当你有无锁数据结构或自定义同步原语时。 Loom 穷尽地探索线程交错 — 它是模型检查器，而不是压力测试。 基于 Mutex/RwLock 的代码不需要。

何时使用哪个工具

unsafe 验证的决策树：

代码是纯 Rust（无 FFI）吗？
├─ 是 → 使用 Miri（捕获 Rust 特定 UB、Stacked Borrows）
│       也在 CI 中运行 ASan 进行纵深防御
└─ 否（通过 FFI 调用 C/C++ 代码）
   ├─ 内存安全问题？
   │  └─ 是 → 使用 Valgrind memcheck 和 ASan
   ├─ 并发问题？
   │  └─ 是 → 使用 TSan（更快）或 Helgrind（更彻底）
   └─ 内存泄漏问题？
      └─ 是 → 使用 Valgrind --leak-check=full

推荐的 CI 矩阵：

# 并行运行所有工具以获得快速反馈
jobs:
  miri:
    runs-on: ubuntu-latest
    steps:
      - uses: dtolnay/rust-toolchain@nightly
        with: { components: miri }
      - run: cargo miri test --workspace

  asan:
    runs-on: ubuntu-latest
    steps:
      - uses: dtolnay/rust-toolchain@nightly
      - run: |
          RUSTFLAGS="-Zsanitizer=address" \

  valgrind:
    runs-on: ubuntu-latest

      - run: sudo apt-get install -y valgrind

            valgrind --error-exitcode=1 --leak-check=full "$test_bin" --test-threads=1
          done

应用：零 Unsafe — 以及何时需要它

项目在 90K+ 行 Rust 中包含零 unsafe 块。 这对于系统级诊断工具来说是一项非凡的成就，并证明安全 Rust 足以用于：

  • IPMI 通信（通过 std::process::Command 到 ipmitool）
  • GPU 查询（通过 std::process::Command 到 accel-query）
  • PCIe 拓扑解析（纯 JSON/文本解析）
  • SEL 记录管理（纯数据结构）
  • DER 报告生成（JSON 序列化）

项目何时需要 unsafe？

可能引入 unsafe 的触发器：

场景为什么 unsafe推荐的验证
直接基于 ioctl 的 IPMIlibc::ioctl() 绕过 ipmitool 子进程Miri + Valgrind
直接 GPU 驱动查询accel-mgmt FFI 而不是 accel-query 解析Valgrind（C 库）
内存映射 PCIe 配置mmap 用于直接配置空间读取ASan + Valgrind
无锁 SEL 缓冲区AtomicPtr 用于并发事件收集Miri + TSan
嵌入式/no_std 变体裸机的原始指针操作Miri

准备：在引入 unsafe 之前，将验证工具添加到 CI：

# Cargo.toml — 添加特性标志用于 unsafe 优化
[features]
default = []
direct-ipmi = []     # 启用直接 ioctl IPMI 而不是 ipmitool 子进程
direct-accel-api = []     # 启用 accel-mgmt FFI 而不是 accel-query 解析

#![allow(unused)]
fn main() {
// src/ipmi.rs — 在特性标志后门控
#[cfg(feature = "direct-ipmi")]
mod direct {
    //! 通过 /dev/ipmi0 ioctl 直接 IPMI 设备访问。
    //!
    //! # Safety
    //! 此模块使用 `unsafe` 进行 ioctl 系统调用。
    //! 已验证：Miri（在可能的情况下）、Valgrind memcheck、ASan。

    use std::os::unix::io::RawFd;

    // ... unsafe ioctl 实现 ...
}

#[cfg(not(feature = "direct-ipmi"))]
mod subprocess {
    //! 通过 ipmitool 子进程的 IPMI（默认，完全安全）。
    // ... 当前实现 ...
}
}

关键见解：将 unsafe 保持在 特性标志 后面， 以便可以独立验证。在 CI 中运行 cargo +nightly miri test --features direct-ipmi 以持续验证 unsafe 路径， 而不影响安全的默认构建。

cargo-careful — 稳定版上的额外 UB 检查

cargo-careful 运行你的代码， 启用额外的标准库检查 — 捕获正常构建忽略的一些未定义行为， 无需 nightly 或 Miri 的 10-100 倍减速：

# 安装（需要 nightly，但以接近原生的速度运行你的代码）
cargo install cargo-careful

# 运行带有额外 UB 检查的测试（捕获未初始化内存、无效值）
cargo +nightly careful test

# 运行带有额外检查的二进制文件
cargo +nightly careful run -- --run-diagnostics

cargo-careful 捕获正常构建不捕获的内容：

  • MaybeUninit 和 zeroed() 中未初始化内存的读取
  • 通过 transmute 创建无效的 bool、char 或 enum 值
  • 未对齐的指针读取/写入
  • copy_nonoverlapping 与重叠范围

它在验证阶梯中的位置：

最小开销                                          最彻底
├─ cargo test ──► cargo careful test ──► Miri ──► ASan ──► Valgrind ─┤
│  (0 倍开销)   (~1.5 倍开销)     (10-100 倍)  (2 倍)     (10-50 倍)   │
│  仅安全 Rust   捕获一些 UB       纯 Rust     FFI+Rust  FFI+Rust     │

推荐：在 CI 中添加 cargo +nightly careful test 作为快速安全检查。 它以接近原生的速度运行（不像 Miri）并捕获安全 Rust 抽象掩盖的真正 bug。

Miri 和 Sanitizers 故障排除

症状原因修复
Miri does not support FFIMiri 是 Rust 解释器；无法执行 C 代码对 FFI 代码改用 Valgrind 或 ASan
error: unsupported operation: can't call foreign functionMiri 遇到 extern "C" 调用模拟 FFI 边界或在 #[cfg(miri)] 后门控
Stacked Borrows violation别名规则违规 — 即使代码"有效"Miri 是正确的；重构以避免 &mut 与 & 别名
Sanitizer 说 DEADLYSIGNALASan 检测到缓冲区溢出检查数组索引、切片操作和指针算术
LeakSanitizer: detected memory leaksBox::leak()、forget() 或缺少 drop()故意：使用 __lsan_disable() 抑制；无意：修复泄漏
Miri 非常慢Miri 解释，不编译 — 慢 10-100 倍仅在 --lib 测试上运行，或使用 #[cfg_attr(miri, ignore)] 标记慢测试
TSan: false positive 与原子TSan 不完全理解 Rust 的原子排序模型添加 TSAN_OPTIONS=suppressions=tsan.supp 与特定抑制

亲自尝试

  1. 触发 Miri UB 检测：编写一个 unsafe 函数，创建两个对同一 i32 的 &mut 引用（别名违规）。 运行 cargo +nightly miri test 并观察"Stacked Borrows"错误。使用 UnsafeCell 或单独的分配修复它。

  2. 在故意 bug 上运行 ASan：创建一个进行 unsafe 越界数组访问的测试。 使用 RUSTFLAGS="-Zsanitizer=address" 构建并观察 ASan 的报告。注意它如何精确指出行。

  3. 基准测试 Miri 开销：对同一测试套件计时 cargo test --lib vs cargo +nightly miri test --lib。 计算减速因子。基于此，决定在 CI 中哪些测试在 Miri 下运行，哪些使用 #[cfg_attr(miri, ignore)] 跳过。

🟡 练习 1：触发 Miri UB 检测

编写一个 unsafe 函数，创建两个对同一 i32 的 &mut 引用（别名违规）。 运行 cargo +nightly miri test 并观察 Stacked Borrows 错误。修复它。

解决方案 
#![allow(unused)]
fn main() {
#[cfg(test)]
mod tests {
    #[test]
    fn aliasing_ub() {
        let mut x: i32 = 42;
        let ptr = &mut x as *mut i32;
        unsafe {
            // BUG：两个 &mut 引用指向同一位置
            let _a = &mut *ptr;
            let _b = &mut *ptr; // Miri: Stacked Borrows 违规！
        }
    }
}
}

修复：使用单独的分配或 UnsafeCell：

#![allow(unused)]
fn main() {
use std::cell::UnsafeCell;

#[test]
fn no_aliasing_ub() {
    let x = UnsafeCell::new(42);
    unsafe {
        let a = &mut *x.get();
        *a = 100;
    }
}
}

🔴 练习 2：ASan 越界检测

创建一个带有 unsafe 越界数组访问的测试。 在 nightly 上使用 RUSTFLAGS="-Zsanitizer=address" 构建并观察 ASan 的报告。

解决方案 
#![allow(unused)]
fn main() {
#[test]
fn oob_access() {
    let arr = [1u8, 2, 3, 4, 5];
    let ptr = arr.as_ptr();
    unsafe {
        let _val = *ptr.add(10); // 越界！
    }
}
}
RUSTFLAGS="-Zsanitizer=address" cargo +nightly test -Zbuild-std \
  --target x86_64-unknown-linux-gnu -- oob_access
# ASan 报告：stack-buffer-overflow 在 <确切地址>

关键要点

  • Miri 是纯 Rust unsafe 的工具 — 它捕获编译并通过测试的别名违规、释放后使用和泄漏
  • Valgrind 是 FFI/C 互操作的工具 — 它在最终二进制文件上工作，无需重新编译
  • Sanitizers（ASan、TSan、MSan）需要 nightly 但以接近原生的速度运行 — 适合大型测试套件
  • loom 专为验证无锁并发数据结构而构建
  • 在每次推送时在 CI 中运行 Miri；在夜间计划运行 sanitizers 以避免拖慢主流水线
